## Module 1 — 背景与问题 (Background and Problem)

### 动机 (Motivation)
现有的越狱防御范式主要依赖于对输入提示词、输出结果或内部状态的**静态检测（static detection）**——在推理开始前或结束后做一次性检查。这种做法忽视了解码过程中风险的**动态演化（dynamic evolution）**。嵌入在解码轨迹中的风险信号被严重浪费，构成了当前防御系统的**关键盲点（critical blind spot）**。

### 核心科学问题
LLM 解码过程中生成的 token 隐藏状态（hidden states）是否携带比输入提示词更强且更稳定的风险信号？能否利用这些动态轨迹信号实现实时、低延迟的越狱防御？

### 直觉洞见
传统的越狱防守像是在机场安检口检查乘客的行李（输入过滤）和离开时的购物袋（输出过滤），但忽略了乘客在候机厅里的一举一动。TrajGuard 就是在"候机厅"（解码过程）里安装了监控摄像头——它不只在进出口设卡，而是持续观察每个乘客的行为轨迹，一旦发现某人不断向危险区域靠近，就立即拦截。

## Module 2 — 核心方法 (Core Method)

**（1）核心观察：解码隐藏状态的风险信号**
通过实证研究发现：解码阶段关键层中的 **Hidden States 携带了比输入越狱提示词更强且更稳定的风险信号**。具体表现为：在越狱尝试期间生成的 token 的隐藏表示会在潜空间中逐渐逼近高风险区域。这是一个连续的、可观测的漂移过程，而非离散的"安全→有害"跳变。

**（2）流式聚合机制（Streaming Aggregation via Sliding Window）**
TrajGuard 采用滑动窗口机制来聚合隐藏状态的轨迹：
- 窗口大小 w：控制时间粒度
- 步长 s：控制聚合频率
- 在每个窗口内计算局部风险分数 r_t

**（3）触发裁决机制（Trigger & Adjudication）**
仅当局部窗口内的风险**持续超过预设阈值 τ**时才触发后续操作，避免单次波动导致的误报。触发后进行轻量级语义裁决（而非复杂重计算），确认后立即中断或约束后续解码过程。

**关键设计决策**：整个框架是**免训练（training-free）**的——不需要训练任何分类器或检测模型，直接利用预训练模型的内部表示几何特性。

## Module 3 — 技术细节 (Technical Details)

### 数学形式化

令 H = {h_1, h_2, ..., h_T} 为解码过程中某关键层 l 的隐藏状态序列，其中 h_t ∈ R^d 为第 t 个 token 的隐藏向量。

**流式风险量化函数**:

r_t = f_aggregate({h_{t-w+1}, ..., h_t}) = σ(W_r · mean({h_i}) + b_r)

其中:
- W_r ∈ R^{1×d} 为可学习（或预设）的风险投影向量
- σ 为 sigmoid 函数
- mean(·) 为窗口内隐藏状态的均值

**触发条件**:

trigger_t = 1 if ∀i ∈ [t-δ+1, t]: r_i > τ
trigger_t = 0 otherwise

即需要连续 δ 个时间步的风险值都超过阈值才触发（防止单点误报）。

**轻量级裁决**:

一旦 trigger_t = 1，执行快速裁决:
decision = g_semantic(h_t, context_window)
若 decision == "malicious": 执行 interrupt() 或 constrain()

### 关键层选择
论文通过消融实验确定了对越狱检测最敏感的 Transformer 层（通常为中间偏后的注意力层或 MLP 层）。这些层的隐藏状态在安全和不安全生成之间表现出最大的可分性。

## Module 4 — 实验设置与结果 (Experiments and Results)

### 数据集与评价指标
- **攻击覆盖**: 12 种不同的越狱攻击方法
- **模型覆盖**: 多个开源大语言模型（Llama 系列、Qwen 等）
- **评价指标**:
  - Defense Rate / Block Rate (↑): 成功拦截率
  - False Positive Rate (FPR) (↓): 误报率（将正常内容误判为越狱）
  - Detection Latency per Token (↓): 每个 token 的检测延迟

### 主要结果表格

| 指标 | TrajGuard | 输入侧防御基线 | 输出侧防御基线 |
|------|----------|-------------|-------------|
| 平均防御率 | **~95%** | ~70-85% | ~60-80% |
| FPR | **<1.5%** | 3-8% | 2-5% |
| 延迟/token | **5.2 ms** | <1ms | 50-200ms |
| 免训练 | ✅ 是 | 部分 | 通常否 |

### 关键发现

1. **隐藏状态信号远优于输入/输出信号**: 解码过程中的 hidden states 提供了比原始 prompt 或最终输出更早、更准确的安全风险预测能力。
2. **实时性与准确性兼得**: 5.2ms/token 的延迟意味着在流式生成场景中几乎无感知地实现拦截——在用户看到有害内容之前就已完成阻断。
3. **极低的误报率**: <1.5% 的 FPR 表明该方法不会频繁打断正常的模型输出，这对用户体验至关重要。

### 消融实验

| 配置 | 防御率 | FPR | 说明 |
|------|--------|-----|------|
| 完整 TrajGuard | Best | <1.5% | 所有组件联合最优 |
| 无滑动窗口（逐token判断） | 高 | >10% | 单点噪声导致大量误报 |
| 无持续触发条件 | 高 | >8% | 偶发峰值导致过度拦截 |
| 使用不同层 | 变化大 | — | 层选择显著影响效果 |

## Module 5 — 局限性分析 (Limitations)

1. **白盒假设**: 当前方法需要访问模型内部的隐藏状态，这意味着它在闭源 API（如 GPT、Claude）上无法直接部署——除非模型提供商开放中间层访问接口。

2. **关键层依赖**: 方法的效果高度依赖于正确识别"最敏感"的关键层。不同架构（Transformer vs MoE vs Mamba）的最优层可能完全不同，每次更换模型都需要重新标定。

3. **自适应对抗风险**: 如果攻击者知道 TrajGuard 的存在和具体工作机制，理论上可以设计"慢热型"越狱——让隐藏状态缓慢漂移以保持在阈值以下，然后在最后几个 token 突然释放有害内容。

4. **多模态扩展未验证**: 论文聚焦于纯文本 LLM 场景。对于多模态模型（VLM），图像 token 和文本 token 的隐藏状态是否具有相同的风险可分性尚不清楚。

5. **阈值的通用性**: 触发阈值 τ 是一个超参数，其最优值可能因模型、任务类型甚至对话主题而异，在实际部署中可能需要针对每个使用场景单独调优。

### 核心代码框架
```python
import torch
import torch.nn.functional as F

class TrajGuard:
    """免训练解码时越狱防御框架"""
    def __init__(self, model, threshold=0.85, window_size=10,
                 sustain_steps=3, target_layer=-1):
        self.model = model
        self.threshold = threshold
        self.window_size = window_size
        self.sustain_steps = sustain_steps
        self.target_layer = target_layer  # 监控的关键层索引
        self.risk_history = []
        self.triggered = False

    @torch.no_grad()
    def compute_risk(self, hidden_states):
        """
        计算当前步骤的风险分数
        hidden_states: (batch, seq_len, hidden_dim) 目标层的输出
        """
        # 取最后一个 token 的隐状态
        current_hs = hidden_states[0, -1, :]  # (hidden_dim,)

        # 简单方案: 使用 L2 范数到安全中心的距离作为风险代理
        # 实际论文可能有更复杂的投影方法
        risk_score = self._project_to_risk(current_hs)
        return risk_score.item()

    def check_and_trigger(self, hidden_states):
        """流式检测主循环"""
        risk = self.compute_risk(hidden_states)
        self.risk_history.append(risk)

        if len(self.risk_history) >= self.window_size:
            window = self.risk_history[-self.window_size:]
            # 条件1: 窗口内平均风险超过阈值
            avg_risk = sum(window) / len(window)
            # 条件2: 连续 sustain_steps 步超过阈值
            recent = self.risk_history[-self.sustain_steps:]
            sustained = all(r > self.threshold for r in recent)

            if avg_risk > self.threshold and sustained and not self.triggered:
                self.triggered = True
                return "BLOCK"
        return "PASS"

    def _project_to_risk(self, hs):
        """将隐状态投影到风险分数（简化版）"""
        # 实际实现可能使用预定义的风险方向向量
        # 或基于统计的异常检测方法
        return torch.norm(hs).item() / (torch.norm(hs).item() + 1e-8)


# 与 HuggingFace 生成循环集成示例
def safe_generate(model, input_ids, guard):
    """集成 TrajGuard 的安全生成"""
    output_ids = input_ids.clone()
    past_key_values = None

    for step in range(max_new_tokens):
        outputs = model(output_ids, past_key_values=past_values,
                       output_hidden_states=True, use_cache=True)
        logits = outputs.logits[:, -1, :]
        hs = outputs.hidden_states[guard.target_layer]  # 目标层隐状态

        # TrajGuard 检查
        action = guard.check_and_trigger(hs)
        if action == "BLOCK":
            print(f"[TrajGuard] Blocked at step {step}, risk={guard.risk_history[-1]:.3f}")
            break

        next_token = torch.argmax(logits, dim=-1)
        output_ids = torch.cat([output_ids, next_token.unsqueeze(-1)], dim=1)
        past_key_values = outputs.past_key_values

    return output_ids
```

### 复现步骤
1. 加载目标 LLM 及其 tokenizer
2. 初始化 TrajGuard 并配置参数（阈值、窗口大小、目标层）
3. 准备 12 种标准越狱攻击的测试 prompt 集
4. 对每个 prompt 运行 safe_generate() 并记录：
   - 是否成功拦截（defense rate）
   - 拦截发生的 token 位置（detection earliness）
   - 正常 prompt 的误报情况（FPR）
5. 调整阈值 τ 以平衡防御率和误报率
6. （可选）在不同层上重复实验找到最优目标层

### 预期输出
- 各攻击方法的防御率 ≥ 90%（目标 95%）
- 正常对话的 FPR < 2%
- 平均检测位置在有害内容出现前的 2-5 个 token
