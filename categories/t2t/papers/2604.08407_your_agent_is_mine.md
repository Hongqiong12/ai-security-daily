## Module 1 — 背景与问题 (Background and Problem)

### 动机 (Motivation)
大语言模型（LLM）智能体越来越依赖第三方 API Router（API路由器）来将工具调用请求分发到多个上游提供商（如 OpenAI、Anthropic、Google 等）。这些 Router 作为应用层代理运行，能够完全明文访问每一个传输中的 JSON 载荷。然而，当前 LLM 生态中**没有任何提供商在客户端和上游模型之间强制执行加密完整性验证**——这意味着恶意中间人可以在完全不被察觉的情况下篡改所有通信内容。

### 核心科学问题
LLM API Router 这一供应链环节的攻击面究竟有多大？实际部署中的 Router 是否已被恶意行为者利用？如果被利用，攻击的影响范围和严重程度如何？能否设计有效的客户端防御措施？

### 直觉洞见
想象你通过一个"翻译中介"给外国朋友写信，但这个中介不仅帮你翻译，还偷偷在你的信里塞入"请给我转账100美元"，同时把你朋友回信里的银行账号替换成他自己的。当前 LLM API Router 的安全状态就是如此——你的 Agent 发出的每一条指令、接收的每一条回复，都经过了一个不可信的中介，而你没有加密签名来验证完整性。

## Module 2 — 核心方法 (Core Method)

本文提出了一套系统化的 **LLM API Router 攻击测量框架**，包含三个核心组件：

**（1）威胁模型形式化（Threat Model Formalization）**
形式化定义了恶意 LLM API Router 的攻击者能力模型：攻击者作为 MITM（中间人），对应用层 JSON 载荷拥有完整的读写权限。信任假设为：客户端与上游之间缺乏端到端加密完整性保护。

**（2）攻击分类体系（Attack Taxonomy）**
定义两类核心攻击及其自适应变体：
- AC-1: Payload Injection（载荷注入）— 在请求/响应中注入恶意指令
  - AC-1.a: Dependency-targeted Injection（依赖目标注入）
  - AC-1.b: Conditional Delivery（条件投递，用于规避检测）
- AC-2: Secret Exfiltration（密钥窃取）— 从 Agent 执行过程中窃取 API 密钥、凭证等敏感信息

**（3）研究代理 Mine 与防御评估**
构建名为 "Mine"的研究代理，实现了针对四种公开代理框架（OpenAI、Anthropic、Google、开源）的所有四类攻击。用 Mine 评估了三种可部署的客户端防御策略：
- Fail-closed policy gate（故障关闭策略门控）
- Response-side anomaly screening（响应侧异常筛查）
- Append-only transparency logging（仅追加透明度日志）

**关键设计决策**：采用蜜罐技术（AWS Canary Credentials + ETH 私钥）来实证测量攻击者的真实窃取行为，而非仅做理论分析。

## Module 3 — 技术细节 (Technical Details)

### 测量对象
- **付费市场**: 从淘宝、闲鱼、Shopify 购买 **28 个付费 Router**
- **免费社区**: 从公共社区收集 **400 个免费 Router**

### 投毒实验（Poisoning Studies）
两项投毒研究展示了良性 Router 如何被拉入同一攻击面：

| 实验场景 | 泄露资源 | 影响规模 |
|---------|---------|---------|
| 单个泄露的 OpenAI 密钥 | GPT-5.4 tokens + Codex 会话 | **1亿 tokens**, 7+ 会话 |
| 弱配置诱饵（weak-config bait） | 计费 tokens + 凭证 + YOLO会话 | **20亿 tokens**, 99/440 凭证, 401 YOLO会话 |

### 防御方案实现细节

**Fail-closed Policy Gate**:
在客户端侧设置策略检查点，当检测到响应异常时拒绝接受该响应并触发告警。核心参数：异常检测阈值 ε、时间窗口 Δt。

**Response-side Anomaly Screening**:
使用统计方法（如 Z-score 异常检测或基于 embedding 的语义偏移检测）筛选出偏离正常分布模式的响应。

**Append-only Transparency Logging**:
所有经过 Router 的请求-响应对以只追加方式记录至不可变日志，支持事后审计但不实时拦截。

## Module 4 — 实验设置与结果 (Experiments and Results)

### 数据集与评价指标
- **评价指标**: 恶意 Router 检测率 (↑), 凭证触达数 (↓), 资源损失金额 (↓), 防御漏检率 (↓)
- **测试范围**: 28 付费 + 400 免费 Router

### 主要结果表格

| 指标 | 付费 Router (n=28) | 免费 Router (n=400) |
|------|-------------------|-------------------|
| 活跃恶意注入 | **1 个 (3.6%)** | **8 个 (2.0%)** |
| 自适应规避触发器 | — | **2 个** |
| AWS 凭证触达 | 部分 | **17 个** |
| ETH 私钥耗尽 | — | **1 个** |

### 关键发现

1. **真实攻击已在发生**: 付费 Router 中 3.6%、免费 Router 中 2.0% 存在活跃的恶意行为，证明这不是理论威胁而是已发生的现实风险。
2. **YOLO 自动化放大危害**: 发现 401 个会话处于自动 YOLO（无人值守全自动）模式，意味着凭证一旦泄露，攻击者可以无限期消耗资源。
3. **经济影响可观**: 单个弱配置诱饵即产生 20 亿计费 tokens 的损失规模，说明 Router 投毒的经济动机充足。
4. **防御有效性有限**: 三种客户端防御均无法完全阻断高级自适应攻击（AC-1.b 条件投递），透明度日志仅能提供事后追溯能力。

### 消融实验
论文对不同防御策略的组合效果进行了评估（详见正文 Section 5）。基本结论：多层防御组合优于单一策略，但仍存在绕过空间。

## Module 5 — 局限性分析 (Limitations)

1. **范围限制**: 研究聚焦于 API Router 这一层面的中间人攻击，未涉及模型权重篡改、训练数据投毒等其他供应链攻击向量。完整的安全图景需要覆盖更多环节。

2. **样本偏差**: 付费 Router 样本主要来自中国电商平台（淘宝/闲鱼）和 Shopify，可能不能完全代表全球商业 Router 市场的安全状况分布。

3. **防御不完美**: 提出的三种客户端防御均为轻量级方案，摘要暗示它们对自适应规避攻击（AC-1.b）的防护存在局限，且可能引入额外的推理延迟开销（具体数值需查看正文）。

4. **伦理边界**: 投毒实验中使用蜜罐凭证主动引诱攻击行为，虽然符合安全研究的标准实践，但在某些司法管辖区可能引发法律合规问题讨论。

5. **泛化性未知**: 防御方案的跨框架迁移能力（如从 OpenAI SDK 迁移到 Anthropic SDK）未经充分验证。
