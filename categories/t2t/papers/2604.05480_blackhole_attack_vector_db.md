## 模块一：研究背景与问题定义

### 动机
向量数据库（Vector Database）已经成为现代 AI 应用的检索骨干，支撑着 RAG、语义搜索、推荐系统等关键组件。然而，其安全性在很大程度上未被探索。如果攻击者能够在向量数据库中注入恶意向量，是否可以操纵几乎所有查询的检索结果？

### 科学问题
**高维嵌入空间的几何特性是否存在可被利用的系统性缺陷？能否通过少量恶意向量注入实现对大规模查询结果的全局性操控？**

### 直觉洞见
- 高维空间中存在"中心驱动的枢纽性"（centrality-driven hubness）现象：靠近质心的向量会成为大量其他向量的最近邻
- 在实际嵌入空间中，质心区域几乎是空的——这创造了一个天然的"真空地带"
- 如果将恶意向量注入这个真空地带，它们就像"黑洞"一样吸引查询，频繁出现在 top-k 结果中

## 模块二：核心方法

### 技术贡献
提出 **Black-Hole Attack**（黑洞攻击），一种针对向量数据库的投毒攻击：

1. **中心驱动的枢纽性理论**
   - 形式化了高维嵌入空间中"接近质心 = 成为枢纽"的数学关系
   - 揭示了嵌入空间几何缺陷与检索脆弱性的本质联系

2. **黑洞向量构造**
   - 计算存储向量的几何中心（质心）
   - 在质心附近生成少量恶意向量
   - 这些向量自然地吸引大量查询进入其邻域

3. **攻击效果验证**
   - 少量注入即可污染绝大多数查询结果
   - 攻击不需要了解具体查询内容

### 方法框架
```
向量数据库
├── 正常向量分布（高维空间中呈壳状分布）
├── 质心区域（几乎为空 → 真空地带）
│
└── 黑洞攻击：
    1. 估算向量集合的几何质心
    2. 在质心±ε范围内注入K个恶意向量
    3. 高维空间的枢纽效应 → 恶意向量成为大量查询的最近邻
    4. top-k检索结果被全局性污染
```

## 模块三：技术细节

### 核心数学原理

**枢纽性（Hubness）现象**：
- 在 d 维空间中，当 d → ∞ 时，所有点对之间的距离趋于相同（维度灾难）
- 但靠近数据分布中心的点，其到其他点的距离方差更小，更容易成为多个点的最近邻
- 这是高维几何的固有属性，与具体嵌入模型无关

**攻击向量生成**：
- 设 V = {v₁, v₂, ..., vₙ} 为数据库中的正常向量
- 计算质心 c = (1/n)Σvᵢ
- 生成攻击向量 a = c + δ，其中 δ 是小扰动
- 攻击向量数量 K ≪ N（远小于正常向量数量）

**为什么质心区域是空的？**
- 高维空间中，数据点倾向于分布在以原点为中心的超球面壳上
- 质心虽然是所有点的"平均位置"，但该位置附近很少有实际数据点
- 这种"中心为空、边缘密集"的分布是高维空间的普遍特征

### 攻击参数
- 注入向量数量 K：极少量即可（实验中具体比例见结果）
- 扰动范围 δ：控制恶意向量偏离质心的程度
- 目标：最大化恶意向量在 top-k 结果中的出现频率

## 模块四：实验设置与结果

### 实验设置
- 多种真实向量数据库场景
- 多种嵌入模型生成的向量集合
- top-k 检索（k=10）

### 核心结果

| 指标 | 结果 |
|------|------|
| 恶意向量出现在 top-10 的比率 | **高达 99.85%** |
| 注入向量占比 | 远小于数据库总量 |
| 对正常检索质量的影响 | 严重破坏检索准确性 |

### 防御评估
对现有的枢纽性缓解方法进行了评估：
- **结论**：现有方法要么显著降低检索准确性，要么提供的保护有限
- 这表明需要专门针对黑洞攻击开发新的防御机制

## 模块五：局限性分析

1. **攻击前提**：攻击者需要能够向向量数据库中写入数据（注入恶意向量），这在完全受控的数据库中较难实现
2. **质心估算**：在某些场景中，攻击者可能无法精确估算数据库向量的质心
3. **嵌入模型依赖**：不同嵌入模型产生的向量分布特性可能影响攻击效果
4. **防御研究不足**：论文指出现有防御不够，但未提出有效的新防御方案
5. **动态数据库**：数据库内容持续更新时，质心可能漂移，需要动态调整攻击向量

## 模块六：复现指南

### 环境要求
- Python 3.8+
- 向量数据库（如 Faiss、Milvus、Chroma 等）
- 嵌入模型（如 OpenAI、Sentence-Transformers 等）
- NumPy/SciPy 用于向量运算

### 复现步骤
1. 选择目标嵌入模型，生成正常向量集合
2. 计算向量集合的几何质心
3. 在质心附近生成 K 个攻击向量
4. 将攻击向量注入向量数据库
5. 执行大量随机查询，统计攻击向量在 top-k 结果中的出现频率
6. 评估现有枢纽性缓解方法的防御效果

## 📌 核心启示

**对 RAG/Agent 安全的影响**：
- **RAG 系统投毒**：黑洞攻击可以让恶意内容出现在几乎所有 RAG 查询的上下文中，间接操控 LLM 输出
- **供应链风险**：使用第三方向量数据库的 AI 应用面临数据投毒风险
- **防御优先级**：向量数据库需要引入异常检测——特别是对质心区域附近的异常向量的监控
- **维度安全**：高维嵌入的固有几何缺陷是系统性风险，不是某个具体实现的 bug
