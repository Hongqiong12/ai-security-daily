## Module 1 — 背景与问题 (Background and Problem)

### 动机 (Motivation)
过去关于 offensive cyber agent 的讨论，很多停留在“模型能不能写 exploit 脚本”或“某个 CTF 能做出多少题”。但这类结果常常把真正关键的系统变量混在一起：**模型能力、agent 架构、工具环境、提示策略、以及 API / 工具适配度**。如果这些变量没有被拆开，benchmark 分数就很难解释，更无法比较。

### 核心科学问题 (Core Scientific Question)
在真实攻防工具链参与的情况下，究竟是模型本身、prompt engineering，还是执行环境更决定 offensive cyber performance？不同前沿 LLM 在标准化 CTF benchmark 上的系统能力边界到底在哪里？Planner / Executor 异构搭配是否真能带来更好的成本—性能折中？

### 直觉洞见 (Intuitive Insight)
这篇论文最重要的洞见是：**在 offensive cyber 场景中，环境与工具链经常比 prompt trick 更重要**。当 agent 真的拿到 Kali Linux、100+ 渗透工具和命令可发现性之后，很多 fancy prompting 不但不增益，甚至会成为负担。换句话说，这类 benchmark 测的不是孤立模型，而是“model × agent × tools × environment”的系统能力。

## Module 2 — 核心方法 (Core Method)

### 方法总览
作者基于 **NYU CTF Bench** 与 **D-CIPHER** 做系统扩展，围绕四个研究问题展开：

1. **环境作用**：Ubuntu vs Kali Linux；
2. **prompt 策略作用**：generic prompts vs category-specific tips vs auto-prompting；
3. **跨模型比较**：10 个 frontier models、7 个 providers；
4. **多智能体配置比较**：Planner / Executor 同构 vs 异构搭配。

### 评测框架
- **Benchmark**：NYU CTF Bench 全量 **200 个 challenges**；
- **Agent framework**：D-CIPHER Planner–Executor 多智能体框架；
- **环境扩展**：自定义 Kali Linux Docker + **100+ security tools**；
- **工具可发现性**：加入 `ListCommandsTool` 与 `LookupCommandTool`。

### 方法定位
这不是 attack / defense 论文，而是一篇**能力基准与系统评测基础设施论文**。它把 offensive cyber benchmark 从“静态题库 + 单模型跑分”推进到“有工具环境、有 agent 架构、有成本分析”的系统级评测。

## Module 3 — 技术细节 (Technical Details)

### NYU CTF Bench 组成
200 个 challenge 来自 CSAW CTF (2016–2024)，分为：
- **Cryptography**：52
- **Reverse Engineering**：51
- **Binary Exploitation (Pwn)**：39
- **Misc**：24
- **Web**：19
- **Forensics**：15

### D-CIPHER 架构扩展
#### 1. 双 agent 协作
- **Planner**：负责高层策略与任务拆分；
- **Executor**：负责实际命令执行、脚本编写与结果分析。

#### 2. 环境层扩展
- **Ubuntu 22.04** 作为基础环境；
- **Kali Linux** 作为增强环境；
- Kali 中预装 **100+ 渗透测试工具**，例如 nmap、sqlmap、john、hashcat、binwalk、volatility。

#### 3. 工具发现能力
- `commands_documentation.csv` 保存工具及其说明；
- `ListCommandsTool` / `LookupCommandTool` 让 agent 能查询“有哪些工具、怎么用”。

### 统一实验配置
- 默认模型：Gemini 3 Pro（在环境 / prompt 因子实验中）；
- Temperature：**1.0**；
- Planner rounds：**30**；
- Executor rounds：**100**；
- 每题 nominal cost limit：**$5.00**；
- 每题 wall-clock timeout：**10 分钟**。

### 析因实验设计
对环境 / 提示策略 / auto-prompting 做 2×2×2 组合：
- Environment：Ubuntu / Kali
- Prompt：Generic / Category-specific tips
- Auto-prompting：Off / On

共 **8 种配置 × 200 题 = 1600 次 challenge run**。

## Module 4 — 实验设置与结果 (Experiments and Results)

### 关键结果一：Kali 环境带来实打实的增益
在 **Generic + No AutoPrompt** 条件下：
- **Kali：52.0%** solve rate
- **Ubuntu：42.5%** solve rate
- 提升 **+9.5 个百分点**

类别上最明显的改善出现在：
- **Forensics**：46.7% vs 20.0%
- **Reverse Engineering**：58.8% vs 49.0%

说明工具环境越关键的题型，环境差异越决定结果。

### 关键结果二：Prompt engineering 在强工具环境里经常是负收益
在 Kali 下：
- **Generic：52.0%**
- **Category-specific tips：40.0%**

也就是 tips 反而拉低 **12 个点**。作者解释为：当环境已经足够强时，提示词会过度限制 agent 的探索空间。

### 关键结果三：Auto-prompting 大多数时候适得其反
- Kali + Generic：**52.0% → 39.5%**（-12.5pp）
- Ubuntu + Tips：**44.5% → 16.0%**（-28.5pp）
- 唯一明显例外是 Kali + Tips：**40.0% → 48.5%**（+8.5pp）

总体结论是：auto-prompting 常常浪费预算并制造误导性初始分析。

### 关键结果四：模型排名
在最佳配置（Kali + Generic + No AutoPrompt）下：

| 模型 | Solve rate | 总成本 | 每 solve 成本 |
|------|-----------:|-------:|--------------:|
| **Claude 4.5 Opus** | **59.0%** | $249.92 | $2.12 |
| **Gemini 3 Pro** | **52.0%** | $44.23 | $0.43 |
| **Gemini 3 Flash** | **27.0%** | $2.69 | **$0.05** |
| GLM-5 | 19.5% | $22.09 | $0.57 |
| GPT-5.2-Codex | 18.0% | $9.23 | $0.26 |
| GPT-5.2 | 13.5% | $7.47 | $0.28 |
| DeepSeek-V3 | 6.5% | $0.19 | $0.01 |
| DeepSeek-R1 | 4.0% | $0.13 | $0.02 |
| Qwen 3.5 397B-A17B | 3.5% | $1.39 | $0.20 |
| Llama 3.3 70B | 2.5% | $0.03 | $0.01 |

作者因此给出三层结论：
- 追求最高 solve rate：**Claude 4.5 Opus**；
- 追求性能 / 成本平衡：**Gemini 3 Pro**；
- 极端预算敏感：**Gemini 3 Flash**。

### 关键结果五：同构 Planner–Executor 比异构更稳定
| Planner | Executor | Solve rate |
|---------|----------|-----------:|
| **Pro** | **Pro** | **52.0%** |
| Pro | Flash | 28.5% |
| Flash | Flash | 27.0% |
| Flash | Pro | 23.5% |

这说明 planner / executor 之间的**语义一致性与协同风格**，比“强 planner + 弱 executor”的直觉搭配更重要。

## Module 5 — 局限性分析 (Limitations)

1. **结果依赖于 D-CIPHER 架构**：换成别的 agent framework，绝对数字可能不同。  
2. **实验存在采样随机性**：temperature=1.0，且多数结果基于单次 run，应视为 indicative 而非最终稳态。  
3. **NYU CTF Bench 仍是静态、学术型 benchmark**：外部有效性不等于真实 offensive cyber operation。  
4. **存在 benchmark contamination 风险**：公开 writeup 可能已进入模型训练分布。  
5. **分数混合了多种因素**：不仅是模型 reasoning，还有 API 兼容性、工具调用适配与 agent 编排质量。

## Module 6 — 复现指南 (Reproduction Guide)

### 复现目标
复现重点是重建其“系统级 benchmark”条件，而不是只复现一个模型分数。

### 复现步骤
1. 准备 NYU CTF Bench 的 200 个 Dockerized challenges；
2. 搭建 D-CIPHER 风格的 Planner / Executor agent；
3. 分别配置 Ubuntu 与 Kali 两类环境，并在 Kali 中装入 **100+ 工具**；
4. 实现工具发现模块（列出可用命令、查询命令文档）；
5. 对环境 / prompt / auto-prompting 做 2×2×2 析因实验；
6. 再在最佳配置下对 10 个模型做 solve rate / total cost / cost per solve 评测；
7. 最后比较同构与异构 planner–executor 配置。

### 关键核验点
- Kali 是否稳定带来约 **+9.5pp** 的提升；
- Generic prompt 是否在强工具环境里优于 category tips；
- Auto-prompting 是否大多数时候为负收益；
- Claude 4.5 Opus、Gemini 3 Pro、Gemini 3 Flash 的性能 / 成本排序是否复现；
- Pro / Pro 是否显著优于异构组合。
