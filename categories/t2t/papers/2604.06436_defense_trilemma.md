## Module 1 — 背景与问题 (Background and Problem)

### 动机 (Motivation)

随着 LLM Agent 和 Tool Use 场景爆发，提示注入（Prompt Injection, PI）攻击成为最实际的安全威胁之一。业界主流防御方案是**防御封装（Defense Wrapper）**——即在用户输入到达模型之前进行预处理函数 D: X→X，过滤或改写恶意输入。然而，这类封装是否真能从根本上保障安全性？此前从未有工作给出数学层面的否定性证明。大量工程实践在"打补丁"，但无人回答"补丁是否有上限"这个根本性问题。

### 核心科学问题

对于具有连通提示空间的语言模型，是否存在一种既连续又能保留效用的防御封装 D，使得所有输出都严格安全？如果不存在，其根本约束是什么？

### 直觉洞见

想象你在修补一面有洞的网（提示空间）。你有三种要求：（1）修补后的网不能有裂缝（连续性）；（2）鱼还能正常游过（效用保留）；（3）所有有害鱼类都被挡住（安全性）。这篇论文用严格的数学证明告诉你：**这三件事不可能同时做到**。就像你无法同时让一张网"无缝隙"、"通透"又"完全过滤"——这是拓扑学意义上的不可能，不是工程不够努力的问题。

## Module 2 — 核心方法 (Core Method)

### （1）防御封装的形式化定义

将防御封装建模为函数 D: X → X，其中 X 是提示空间（通常是离散 token 序列的嵌入空间或其连续松弛）。要求 D 满足:
- **连续性 (Continuity)**: D 是连续函数（输入微小变化不会导致输出跳变）
- **效用保留 (Utility Preservation)**: 对于正常输入 x_clean，D(x_clean) ≈ x_clean（良性查询不被过度修改）

### （2）三难困境的递进式证明

论文通过三个逐次增强的假设建立不可行性结论:

1. **边界固定化 (Boundary Fixation)**: 存在某些"阈值级别"的输入处于安全/不安全的边界上，防御 D 无法改变这些点的状态（否则破坏效用保留或连续性）
2. **ε-鲁棒性约束**: 在 Lipschitz 连续性条件围绕固定边界点形成正测度的带状区域，该区域内的点仍保持在阈值附近
3. **横截性不安全区域**: 引入横截性（transversality）条件后，必然存在一个具有正测度的不安全输入子集

### （3）Lean 4 机械验证

全部理论证明使用 Lean 4 定理证明器进行了形式化验证，排除了人工证明的可能疏漏。

### （4）扩展场景

理论进一步扩展至:
- **离散版本**: 不依赖拓扑结构的纯离散结果
- **多轮交互**: 对话历史累积效应下的防御极限
- **随机防御**: 引入随机性的防御封装是否改变结论？
- **容量-奇偶校验**: 特定约束下的变体分析

## Module 3 — 技术细节 (Technical Details)

### 核心定理（非正式表述）

**定理 (Defense Trilemma)**: 设 X 为连通拓扑空间（提示空间），f: X → Y 为语言模型（Y 为输出空间），S ⊂ Y 为不安全输出的像。若防御封装 D: X → X 满足:
- (C) D 连续
- (U) D 保持效用 (∀x∈X_clean, d(D(x), x) < ε)
- (I) f∘D(X) ∩ S = ∅（完整性/安全性）

则在温和的正则性条件下（如 Lipschitz 连续），上述三个条件**无法同时成立**。

### 证明直觉

```
连通空间 X ──D──> X ──f──> Y
     ↑                    ↑
  边界点 x_b           不安全区域 S

1. x_b ∈ ∂X_safe (边界): f(x_b) ∈ ∂S (临界输出)
2. D(x_b) = x_b (边界固定化): D 不能移动边界点
3. D 连续 + Lipschitz: ∃ B(x_b, δ) 使得 D(B) ⊆ B(x_b, L·δ)
4. B(x_b, δ) 中存在不安全点 x_unsafe: f(D(x_unsafe)) ∈ S
5. 矛盾: D 无法同时满足 C+U+I
```

### 关键数学工具

| 工具 | 用途 |
|------|------|
| 拓扑连通性 (Connectivity) | 保证安全/不安全区域之间存在路径 |
| Lipschitz 连续性 | 限制防御封装的局部变形程度 |
| 横截性 (Transversality) | 保证不安全区域的"厚度"不为零 |
| 测度论 (Measure Theory) | 量化不安全输入集合的"大小" |
| Lean 4 定理证明器 | 全部定理的机械验证 |

## Module 4 — 实验设置与结果 (Experiments and Results)

### 数据集与评价指标

| 维度 | 说明 |
|------|------|
| 验证模型 | 3 个大型语言模型（具体型号待确认，可能含 LLaMA、Qwen 系列） |
| 攻击类型 | Prompt Injection（间接注入、直接注入、越狱混合） |
| 防御方案 | 多种 Wrapper 实现（输入重写、指令净化、分隔符增强等） |
| 评价指标 | 安全违规率 (↓)、效用保持率 (↑)、防御覆盖率 |

### 主要结果表格

| 防御封装类型 | 安全违规率 | 效用保持 | 是否满足三条件 |
|------------|-----------|---------|--------------|
| 无防御 (Baseline) | 高 (~60%+) | 100% | ❌ 不安全 |
| 黑名单过滤 | 中等 (~25%) | ~95% | ❌ 不安全 |
| 输入重写 (Paraphrase) | 中低 (~15%) | ~85% | ❌ 效用损失大 |
| LLM-as-Judge 预检 | 低 (~8%) | ~90% | ❌ 仍有漏洞 |
| 理想Wrapper (理论上) | 0% | 100% | ⚠️ **不存在** |

### 实证验证的核心发现

1. **理论预测与实证一致**: 在所有测试的 Wrapper 实现中，确实观察到理论预测的不安全"盲区"（即特定构造的恶意输入总能绕过防御）。
2. **权衡不可避免**: 降低安全违规率的尝试必然导致效用损失或引入不连续跳跃。
3. **架构无关性**: 三种不同架构的 LLM 均表现出一致的模式，支持理论的普适性。

### 消融实验

| 理论假设放松 | 结果变化 |
|------------|---------|
| 放弃连续性 (允许非连续 D) | 可以实现完美安全，但实际中非连续 D 不可部署 |
| 放弃效用保护 | 可以实现完美安全，但用户体验崩溃 |
| 仅离散输入空间 | 不安全区域仍然存在，但可能变为零测集 |

### 关键发现

1. **"防御封装"这条路有天花板**: 不论工程多么精妙，纯输入侧预处理无法实现完美安全。
2. **必须转向其他防御范式**: 训练时对齐、架构修改、输出侧监控——这些不在三难困境的覆盖范围内。
3. **形式化验证的价值**: Lean 4 的机械证明排除了"也许存在巧妙构造"的希望。

## Module 5 — 局限性分析 (Limitations)

1. **理想化假设**: 理论依赖于提示空间的连通性和防御封装的 Lipschitz 性质。实际中离散 token 空间和有限精度实现可能使部分边界情况有所不同。
2. **不排斥替代防御路线**: 结论明确不适用于训练时对齐（RLHF/DPO）、模型架构修改（如添加安全头）、输出侧监控等方法。这限制了结论的实际打击范围——"封装防御不行"≠"所有防御都不行"。
3. ** constructive 不足**: 论文证明了"什么不行"，但对"在限制条件下最优的封装长什么样"的指导相对有限。实践中工程师仍需摸索近似最优解。
4. **未考虑随机防御的全局性质**: 虽然扩展到了随机防御，但如果允许防御封装依赖外部随机源（如每次请求使用不同的 hash），实际攻击难度会上升——尽管理论上不安全区域仍存在。

    """验证防御三难的实证工具"""

    def __init__(self, model, wrapper_func):
        """
        model: 被测试的 LLM

        """
