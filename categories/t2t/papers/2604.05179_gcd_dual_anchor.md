## Module 1 — 背景与问题 (Background and Problem)

### 动机 (Motivation)

当前最强的 LLM 防御过滤器普遍存在一个致命缺陷：**过度拒绝（over-refusal）**。为了拦截越狱攻击，保守型过滤器将大量良性查询误判为有害，严重损害用户体验。此前的工作 GradSafe 使用单一"全接受"锚定令牌（如 "Sure"）来检测安全提示，但其决策边界过于脆弱——阈值的微小波动即可导致大量误报或漏报。更重要的是，即使 GradSafe 判定一个提示"安全"并放行，也无法保证后续自回归解码过程中不会产生有害内容——因为它只管"进门"，不管"出门"。

### 核心科学问题

能否设计一种免训练的解码时护栏，在显著降低误报率的同时，确保无论采样策略如何都能保证首 token 安全？

### 直觉洞见

想象机场安检。GradSafe 就像一个安检员只问一个问题："你是好人吗？"（单一锚点 "Sure"）。旅客说"是"就放行——但坏人也可以说"是"。GCD 则像是一个**双提问安检员**：先问"你是好人吗？"再问"你需要被拒绝吗？"（双锚点 "Sure" / "Sorry"）。只有两个答案一致时才做决定，大幅降低了误判。更进一步，如果判定可疑，GCD 不是简单拦住，而是**直接把旅客的手铐好再放行**（预注入拒绝 token）——这样不管旅客之后想干什么，第一步已经被锁死了。

## Module 2 — 核心方法 (Core Method)

### （1）Dual-Anchor Steering（双锚点转向）

GCD 同时监测两个锚定令牌的概率分布:
- **接受锚 (Accept Anchor)**: "Sure" —— 代表模型倾向于接受/执行该请求
- **拒绝锚 (Reject Anchor)**: "Sorry" —— 代表模型倾向于拒绝该请求

通过比较两个锚点的对数概率差异来划定更紧致的决策边界，而非依赖单一阈值的二元判断。

### （2）Gradient-Controlled Decision（梯度控制决策）

利用梯度信号（而非原始概率）来判断提示的安全性。计算损失函数相对于输入嵌入的梯度方向，判断其对两个锚点的"推动力"哪个更强。这种方法比朴素概率比较更加稳定。

### （3）Pre-injection Mitigation（预注入缓解）

这是 GCD 最具创新性的组件。当某个提示被标记为潜在不安全时，GCD 在恢复自回归解码前**预注入 1-2 个拒绝令牌**（如 "Sorry, I can't..."）。这一设计确保:
- 无论后续使用何种采样策略（greedy、top-p、temperature），首 token 必然是安全的
- 后续 token 的生成会被预注入序列引导至安全轨道（类似于强制前缀）

## Module 3 — 技术细节 (Technical Details)

### 数学形式化

**步骤 1 — 双锚点评分**

给定输入 prompt x 及其嵌入 e(x)，分别计算两个锚点 token 的条件对数概率:

p_acc(x) = log P(token="Sure" | x)
p_rej(x) = log P(token="Sorry" | x)

安全分数: s(x) = p_acc(x) - p_rej(x)

当 s(x) > τ (阈值) 时判定为安全，否则为不安全。

**步骤 2 — 梯度增强判决**

g_acc = ∇_e(x) L_acc, 其中 L_acc = -log P("Sure" | x)
g_rej = ∇_e(x) L_rej, 其中 L_rej = -log P("Sorry" | x)

梯度一致性指标 c(x) 作为阈值附近的 tie-breaker。

**步骤 3 — 预注入缓解**

若 x 被判定为不安全，构造强制前缀:

prefix = "Sorry, I can't help with that. "
output = Decode(model, prefix ⊕ [GENERATION_START])

由于自回归条件概率链式规则:
P(t₁, t₂, ...) = P(t₁) · P(t₂ | t₁) · ...

首 token t₁ 被锁定为拒绝响应的首字符，后续生成天然倾向安全轨道。

### 关键超参数

| 超参数 | 含义 | 推荐值 |
|-------|------|-------|
| τ (threshold) | 安全/不安全阈值 | 0.0 |
| mitigation_prefix | 预注入前缀 | "Sorry, I can't..." |
| demo_templates | 演示模板数 | ~20 |

## Module 4 — 实验设置与结果 (Experiments and Results)

### 数据集与评价指标

| 数据集 | 类型 | 用途 |
|--------|------|------|
| ToxicChat | 有毒内容检测 | 误报率 (FPR ↓) |
| XSTest-v2 | 敏感查询测试 | FPR + Recall |
| AdvBench (GCG) | 对抗性越狱 | 攻击成功率 (ASR ↓) |

### 主要结果表格

| 方法 | FPR on ToxicChat (↓) | ASR on AdvBench (↓) | Clean Acc (↑) | 延迟增量 |
|------|---------------------|--------------------|--------------|---------|
| 无防御 (Baseline) | 0% | ~89% | 100% | 0ms |
| GradSafe | ~12.4% | ~42% | ~91% | ~15ms |
| **GCD (本文)** | **~5.9% (-52%)** | **~38%** | **~94%** | **~17ms** |

*注: ASR 降低约 10% 相对于最强纯解码基线; FPR 降低 52% 相对于 GradSafe*

### 消融实验

| 设置 | FPR | ASR | 说明 |
|------|-----|-----|------|
| 完整 GCD | 最佳 | 最优 | 双锚 + 预注入联合最佳 |
| 仅双锚 (无预注入) | 低 | 中等 | 预注入贡献显著 ASR 降低 |
| 单锚 (退化为GradSafe模式) | 高 | 中高 | 验证双锚的必要性 |
| 无梯度增强 | 略高 | 类似 | 梯度主要改善边界稳定性 |

### 关键发现

1. **双锚决策显著降低误报**: 在保持召回率接近的情况下，FPR 较 GradSafe 降低 **52%**。
2. **预注入实现确定性首 token 安全**: 这是首个在理论上保证（而非统计上保证）首 token 安全的免训练方法。
3. **跨架构迁移成功**: 在 LLaMA-2-7B、Mixtral-8x7B、Qwen2-7B 上均有效，证明方法的通用性。

## Module 5 — 局限性分析 (Limitations)

1. **依赖锚点 token 的语义匹配**: "Sure"/"Sorry" 这类锚点的有效性依赖于模型训练分布中确实存在这些词与接受/拒绝行为的强关联。对于非英语或特殊微调模型，锚点选择需要重新校准。
2. **预引入理延迟**: 虽然 ~17-20ms 的增量很小，但在极高吞吐场景下仍可能累积。且预注入机制本质上是用"放弃回答"换"安全性"，对需要处理边缘 case 的商业系统可能过于保守。
3. **自适应攻击风险**: 若攻击者知道 GCD 的锚点 token 和阈值策略，可能构造刻意规避双锚检测的 prompt。论文未报告对抗性自适应攻击下的表现。
4. **无法防御所有攻击类型**: 对于不在训练分布中的新型攻击（如多模态越狱、长尾语义陷阱），双锚信号本身可能不可靠。

### 核心代码框架 (Core Code)

```python
class GradientControlledDecoding:
    def __init__(self, model, tokenizer,
                 accept_anchor="Sure",
                 reject_anchor="Sorry, I can't",
                 threshold=0.0,
                 mitigation_prefix="Sorry, I can't help with that request."):
        self.model = model
        self.tokenizer = tokenizer
        self.acc_tok = accept_anchor
        self.rej_tok = reject_anchor
        self.threshold = threshold
        self.prefix = mitigation_prefix

    def compute_safety_score(self, input_ids, attention_mask):
        """计算双锚点安全分数"""
        with torch.no_grad():
            logits = self.model(input_ids, attention_mask=attention_mask).logits
            last_logits = logits[:, -1, :]
            log_probs = torch.log_softmax(last_logits, dim=-1)

            acc_ids = self.tokenizer(self.acc_tok, add_special_tokens=False)["input_ids"]
            rej_ids = self.tokenizer(self.rej_tok, add_special_tokens=False)["input_ids"]

            p_acc = log_probs[0, acc_ids[0]].item()
            p_rej = log_probs[0, rej_ids[0]].item()

        return p_acc - p_rej

    def is_safe(self, prompt_text):
        inputs = self.tokenizer(prompt_text, return_tensors="pt")
        score = self.compute_safety_score(inputs["input_ids"], inputs["attention_mask"])
        return score > self.threshold, score

    def guarded_generate(self, prompt_text, max_new_tokens=200):
        safe, score = self.is_safe(prompt_text)
        if safe:
            inputs = self.tokenizer(prompt_text, return_tensors="pt")
            return self.model.generate(**inputs, max_new_tokens=max_new_tokens)
        else:
            mitigated = prompt_text + "\n\nAssistant: " + self.prefix
            inputs = self.tokenizer(mitigated, return_tensors="pt")
            return self.model.generate(**inputs, max_new_tokens=max_new_tokens, do_sample=False)
```
