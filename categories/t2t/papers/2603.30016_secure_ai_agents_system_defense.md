## 模块一：核心问题与动机

**研究问题**: 如何从系统架构层面（而非单点模型层面）防御间接提示注入攻击（Indirect Prompt Injection, IPI）？

**背景与动机**: 基于 LLM 的 AI 代理（AI Agents）已广泛部署，执行复杂的工具调用和外部数据交互任务。间接提示注入——将恶意指令嵌入 Agent 从外部来源（网页、文档、API 响应）读取的数据中——已成为 AI 代理安全的头号威胁。现有防御多集中在模型层面（提升 LLM 的鲁棒性），但这种单点防御存在根本性局限：LLM 无法在所有上下文下可靠区分指令和数据。作者认为，系统级架构才是 IPI 防御的正确抽象层次。

## 模块二：方法与技术贡献

**核心立场**（三大立场）:

**立场一：动态重规划与安全策略更新是必要的**
- 现实任务环境是动态的，静态安全策略无法覆盖所有情境
- 代理需要在执行过程中根据上下文调整安全决策，而非依赖固定规则
- 实现方式：将安全策略表述为代理可查询的结构化规范（policy-as-code）

**立场二：某些安全决策仍需 LLM，但须在约束系统设计中进行**
- 纯规则引擎无法处理语义模糊的安全判断
- 关键：严格限制 LLM 在安全决策中可观察的信息范围（sandboxed context）
- 实现方式：专用安全 LLM 模块，只接收必要上下文，不接收完整 Agent 对话历史

**立场三：人机交互应是核心设计要素**
- 对于固有模糊的情境（ambiguous situations），应优先触发人机确认而非自动决策
- 个性化风险偏好应影响代理的安全阈值

**系统架构要素**:
- **信任层次化**: 将代理执行环境分为可信/不可信区域，不可信数据只能通过受控接口进入
- **基于规则 + 模型的混合检查**: 规则引擎处理明确违规，LLM 模块处理灰色地带
- **结构化代理控制流**: 代理行为通过形式化规范约束，减少模型自由裁量

## 模块三：实验设计与评估

**论文类型**: 立场论文（Position Paper），无实验数据，但包含现有基准分析

**现有基准局限性分析**:
- 多数 IPI 基准（AgentBench、InjecAgent、ToolEmu）在固定环境中测试，无法反映动态部署场景
- 评估指标侧重攻击成功率，忽略代理效用（utility）在防御下的保持程度
- 基准测试结果可能产生虚假的安全感：在基准上安全的系统在现实中仍可能被突破

**设计建议**:
- 基准应包含动态环境（任务状态随时间变化）
- 同时评估安全性和效用的权衡曲线（security-utility Pareto frontier）

## 模块四：关键发现与洞见

**核心洞见**:
1. **模型层面防御的根本局限**: LLM 无法可靠地在所有上下文中区分指令和数据，这是语言模型的固有特性，不是可修复的 bug
2. **系统即防御**: 安全性需要从系统设计阶段介入，而非在模型训练后补丁
3. **基准的虚假安全感**: 当前 IPI 基准可能系统性高估了防御效果
4. **人机协作的不可替代性**: AI 代理在"语义模糊"决策中需要人类监督，而非完全自动化

**研究意义**:
本文代表了 AI 代理安全研究从"如何让 LLM 更鲁棒"到"如何设计安全的代理系统"的范式转变。

## 模块六：研究意义与展望

**直接影响**:
- 为 AI 代理系统工程师提供安全架构蓝图
- 指出现有基准的评估盲点，推动更现实基准的建立

**研究启示**:
- AI 安全研究需要引入传统软件安全（系统架构、访问控制、形式化验证）的工程方法
- 信任模型（trust model）是代理安全的核心抽象

**未来方向**:
- 开发形式化代理安全规范语言
- 建立涵盖动态环境、效用-安全权衡的新一代 IPI 基准
- 研究人机确认界面的设计原则，最小化对用户体验的影响

**安全等级**: 🟡 中等影响（立场论文，推动研究方向转型，无直接攻击代码）
