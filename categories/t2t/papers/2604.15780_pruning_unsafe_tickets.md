## Module 1 — 背景与问题 (Background and Problem)

### 动机 (Motivation)
当前 LLM 安全增强大多依赖 SFT、RLHF、DPO 或各类护栏提示词，本质上都是“再教模型怎么回答更安全”。问题在于，这类方法通常没有真的删除那些会触发危险输出的参数子结构，而只是让模型在分布内更倾向于拒答。因此一旦面对 jailbreak、量化、架构迁移或不同部署环境，模型内部残留的“不安全子网络”仍可能被重新激活。

### 核心科学问题 (Core Scientific Question)
如果把模型中的危险行为视为由稀疏参数子网络承载，是否可以不做昂贵重训练，只靠轻量的后处理剪枝，就显著降低不安全输出并提升抗越狱性，同时尽量保住模型效用？

### 直觉洞见 (Intuitive Insight)
论文借用 Lottery Ticket Hypothesis 的视角提出：模型内部并不需要“整体有毒”才会输出危险内容，可能只是少数关键组件在 unsafe 场景下被异常激活。若能定位并剪掉这些 **unsafe tickets**，模型会暴露出仍然可用的 **safety ticket**，从而以较低资源代价完成部署后安全修补。

## Module 2 — 核心方法 (Core Method)

### 方法总览
论文提出一个四阶段的无梯度安全剪枝框架：

1. **Behavior Profiling**
   - 先用目标模型自己生成 safe / unsafe 响应；
   - 再用安全分类器自动标注，并做聚类抽样，形成贴合模型真实行为的数据。

2. **Attribution Analysis**
   - 在 Wanda 的基础上改造 attribution；
   - 不看 prompt token，只看 response token 的激活贡献，定位与危险输出更相关的参数。

3. **Component Scoring**
   - 对比 unsafe 与 safe 数据上的组件激活强度；
   - 按比值打分，衡量某个组件是否更偏向驱动危险输出。

4. **Iterative Pruning**
   - 逐轮选择最危险组件做剪枝；
   - 提供 Greedy 和 Beam Search 两种策略；
   - 用保 safe、压 unsafe 的对比目标选路径。

### 方法定位
这篇工作不属于“再训练一个更安全模型”，而属于**部署后、参数级、资源友好的安全补丁**。它的价值不在理论最优，而在于：
- 不依赖梯度大规模反传；
- 对量化模型和 VLM 也可用；
- 推理阶段不引入额外 token 与额外延迟。

## Module 3 — 技术细节 (Technical Details)

### 阶段一：行为画像
- 用多样化 prompt 查询目标模型；
- 用 **Llama-Guard-4-12B** 标 safe / unsafe；
- 结合拒答短语过滤，减少误标；
- 分别对 safe / unsafe 响应做 K-means 聚类；
- 聚类数 **K = 32**；
- 固定响应长度 **l = 50 tokens**。

### 阶段二：归因分数
原始 Wanda 分数被改成只统计 response token 激活：

S' = |W| · Σ_response ||x_i||_2

然后分别得到：
- S'_u：unsafe 响应上的 attribution
- S'_s：safe 响应上的 attribution

定义对比重要性：

I = S'_u / (S'_s + ε)

再在组件内部做 z-score 标准化，得到最终组件分数。

### 阶段三：迭代剪枝
- 每轮组件内剪枝比例 **p = 10%**；
- 全局停止阈值 **ρ = 3%**；
- Beam Search 参数：**b1 = 5, b2 = 5**；
- 路径选择目标：

L = CE_safe - CE_unsafe

即尽量维持 safe 数据上的能力，同时让 unsafe completion 更难出现。

### 评测对象
**语言模型**：
- Mistral-7B-Instruct-v0.2
- Mistral-Nemo-Instruct-2407
- Mistral-Small-Instruct-2409
- Mistral-7B 8-bit quantized

**视觉语言模型**：
- LLaVA-v1.6-Mistral-7B

### 评测集
- HarmBench
- MM-SafetyBench
- JailbreakBench (JBB)
- MT-Bench

## Module 4 — 实验设置与结果 (Experiments and Results)

### 关键结果一：Mistral-7B 上 unsafe rate 大幅下降
| 方法 | Unsafe ↓ | Over-Refusal ↓ | Utility ↑ |
|------|----------:|----------------:|----------:|
| Base | 22.8% | 22.7% | 8.07 |
| DPO | 17.0% | 23.7% | 7.95 |
| CB | 0.00% | 100.0% | 8.16 |
| Goal | 0.00% | 59.7% | 7.33 |
| Greedy | 1.67% | 46.0% | 6.96 |
| **Beam** | **1.17%** | **46.0%** | **7.13** |

最重要的结论是：**Beam 把 Mistral-7B 的 Unsafe Rate 从 22.8% 压到 1.17%**，而不像 CB / Goal 那样几乎把模型推成“全拒答”。

### 关键结果二：更大模型与量化模型也成立
**Mistral-Nemo (12B)**：
- Base **24.2%** → Beam **4.33%**

**Mistral-Small (23B)**：
- Base **15.8%** → Beam **3.83%**

**8-bit Mistral-7B**：
- Base **20.7%** → Beam **3.83%**

这说明 unsafe tickets 并不是某个 7B 特例，而是跨模型、跨量化形态都存在的结构现象。

### 关键结果三：VLM 同样受益
在 **LLaVA-v1.6-Mistral-7B** 上：
- Base Unsafe **17.1%**
- Beam Unsafe **1.96%**
- Over-Refusal **26.3%**

与 CB 的 100% 过拒答相比，Beam 更像一个实用型部署方案。

### 关键结果四：抗越狱性显著改善
在 JailbreakBench 攻击上，Beam 相比 Base 都有大幅下降：

| 攻击 | Base ASR | Beam ASR |
|------|---------:|---------:|
| GCG | 53.2 | 17.5 |
| AutoDAN | 45.7 | 0.67 |
| PAIR | 54.0 | 21.8 |
| TAP | 51.2 | 13.5 |

最值得记的数字是 **AutoDAN 45.7 → 0.67**，说明这套剪枝不仅降低原生不安全输出，也显著压制常见 jailbreak 成功率。

### 关键结果五：部署期几乎零成本
Mistral-7B 上：
- Beam 构建显存 **18GB**；
- 构建时间 **2457s**；
- 推理额外 token **0**；
- 输入延迟 **34.9ms**，几乎与 Base 持平。

对比 Goal 虽然更安全，但需要额外 **775 tokens**，延迟显著增加。这意味着 Unsafe Tickets 更适合部署后修补，而非研究型长 prompt 护栏。

### 关键结果六：危险组件高度集中
剪枝位置分析显示，危险参数主要聚集在：
- **Self-Attn.O**
- **MLP 第二层**

例如 Mistral-7B 一共剪掉 **27 个组件**，其中：
- Self-Attn.O **18** 个
- MLP.2 **9** 个

这让“unsafe tickets”从隐喻变成了可观测的结构事实。

## Module 5 — 局限性分析 (Limitations)

1. **Beam Search 的离线构建成本仍不低**：虽然上线后零额外 token、几乎零延迟，但在大模型上构建时间会明显增长，例如 Mistral-Small 需要 **16722s**。
2. **归因仍是静态、局部的**：它更像基于响应阶段激活的相关性剪枝，而不是真正的因果级安全解释。
3. **分类器依赖会影响绝对数值**：safe / unsafe 标注依赖 Llama-Guard-4-12B，不同分类器可能导致量化结果浮动。
4. **过拒答问题仍未完全消失**：Beam 虽比 CB / Goal 好，但 over-refusal 仍常在 25%–46% 区间，说明“安全增强而不损害帮助性”还没完全解决。
5. **方法更适合作为后处理补丁，而非最终对齐方案**：它能显著压 unsafe，但不等价于从目标层彻底重写模型价值观。

responses = collect_model_outputs(model, prompts, max_new_tokens=50)
