## 模块一：背景与问题

### 动机

大型语言模型（LLM）存在记忆训练数据并在推理时复现这些数据的能力，这带来了严重的隐私泄露和版权侵犯风险。量化这种"提取风险"是 LLM 安全审计的核心任务之一。现有主流方法依赖**贪婪解码提取（greedy-decode extraction）**——给定前缀，检查模型是否会生成与训练数据逐字相同的后缀——但该方法存在根本性局限：它只能检测到完全精确的逐字复现（verbatim memorization），而无法捕捉"近逐字"（near-verbatim）复现——即对训练数据进行了少量字词修改的输出。这类近逐字输出在版权和隐私意义上与逐字复现同样危险，但在技术层面极难系统量化。此前的概率提取（probabilistic extraction）方法虽优于贪婪解码，但仍局限于逐字匹配，且近逐字后缀的组合空间极大，蒙特卡洛采样需要约10万次/序列，成本高到无法实用。

### 核心科学问题

能否设计一种确定性算法，以极低计算成本（约20次采样当量）为近逐字提取风险提供可靠的下界估计，从而使大规模的 LLM 隐私风险审计变得切实可行？

### 直觉洞见

近逐字提取风险量化问题类似于在一个巨大的"字典"中找到所有与目标词"相似"（允许少量拼写错误）的词的总出现概率——直接枚举的搜索空间是天文数字，但如果用约束性的树状搜索策略（束搜索），可以系统性地剪枝不可能的路径，以极低代价覆盖大部分重要概率质量。解码约束束搜索正是将这一直觉形式化为 LLM 解码过程中的概率下界估计算法。

## 模块二：核心方法

**（1）近逐字提取风险定义**
近逐字提取将匹配条件从"完全相同"放宽至"编辑距离 ≤ k"（允许最多 k 处字符级或词元级的替换/插入/删除）。近逐字提取概率定义为给定前缀 x₀ 时，模型生成所有与目标后缀 y* 编辑距离 ≤ k 的序列 y 的概率之和：

P_near(y* | x₀, k) = ∑_{y: edit(y, y*) ≤ k} P_model(y | x₀)

由于满足条件的 y 数量组合爆炸，直接求和不可行。

**（2）解码约束束搜索（Decoding-Constrained Beam Search）**
核心创新在于将束搜索与编辑距离约束相结合，在标准 token-by-token 解码过程中动态追踪每条路径与目标 y* 的"已用编辑距离预算"，剪枝超出预算 k 的路径，只保留概率最高的候选序列。该算法提供 P_near 的确定性下界，且计算成本与每序列约20次蒙特卡洛采样相当。

**（3）规模化审计应用**
基于上述算法，对不同规模和类型的 LLM 进行系统性近逐字提取风险分布扫描，揭示逐字方法无法观察到的风险模式。

## 模块三：技术细节

### 近逐字提取概率的精确定义

给定 LLM P_θ、前缀 x₀、目标后缀 y* 和编辑距离容忍度 k，近逐字提取概率为：

P_near(k)(y* | x₀) = ∑_{y ∈ N_k(y*)} P_θ(y | x₀)

其中 N_k(y*) = { y : edit_distance(y, y*) ≤ k }

### 解码约束束搜索算法

```python
def decoding_constrained_beam_search(model, prefix, target_suffix, k, beam_width=20):
    """
    对近逐字提取风险进行确定性下界估计
    
    Args:
        model: LLM
        prefix: 输入前缀 x₀
        target_suffix: 目标后缀 y*
        k: 编辑距离容忍度
        beam_width: 束宽度（约等于 MC 采样次数）
    
    Returns:
        lower_bound: P_near(k)(y* | x₀) 的下界估计
    """
    # 初始化：(当前序列, 已用编辑距离, 对数概率)
    beams = [([], 0, 0.0)]
    completed = []
    
    for step in range(len(target_suffix)):
        new_beams = []
        for seq, edits_used, log_prob in beams:
            target_token = target_suffix[step]
            
            # 获取下一 token 分布
            next_probs = model.next_token_probs(prefix + seq)
            
            # 情形1：精确匹配（编辑距离不增加）
            exact_log_p = log_prob + log(next_probs[target_token])
            new_beams.append((seq + [target_token], edits_used, exact_log_p))
            
            # 情形2：替换（编辑距离 +1）
            if edits_used < k:
                for tok, p in top_k_tokens(next_probs, beam_width):
                    if tok != target_token:
                        new_beams.append((
                            seq + [tok], 
                            edits_used + 1, 
                            log_prob + log(p)
                        ))
        
        # 按概率剪枝，保留 top-beam_width
        beams = sorted(new_beams, key=lambda x: -x[2])[:beam_width]
    
    # 汇总所有完成的路径概率
    return logsumexp([b[2] for b in beams])
```

### 与蒙特卡洛估计的对比

| 方法                   | 成本/序列    | 估计质量        | 类型       |
|-----------------------|------------|----------------|-----------|
| 贪婪解码               | 1次前向传播  | 仅逐字，不完整   | 确定性      |
| 蒙特卡洛采样           | ~100,000次  | 可靠，覆盖近逐字 | 随机        |
| **解码约束束搜索（本文）** | **~20次等量** | **可靠下界，覆盖近逐字** | **确定性** |

### 关键超参数

- k（编辑距离容忍度）：通常取1–5，k=1已能揭示贪婪解码无法发现的大量风险
- beam_width（束宽度）：约20已能提供充分可靠的下界估计
- 序列长度：通常评估50–100 token 的后缀段

## 模块四：实验设置与结果

### 数据集与评价指标

**评估数据集**：来自训练数据的文本序列（包括网页文本、代码、书籍摘录等多种类型），通过前缀-后缀对形式构建评估样本。

**评价指标**：
- 近逐字提取概率 P_near(k)（↑表示风险越高）
- 可提取序列数量（达到一定概率阈值的序列数）↑ 表示风险更大
- 逐字提取概率 P_verbatim（作为比较基线）

### 主要结果表格

| 评估方法               | 可识别序列数 | 单序列提取概率质量 | 计算成本      |
|----------------------|------------|----------------|-------------|
| 贪婪解码（基线）        | 较少         | 低（仅逐字）     | 1次传播/序列  |
| 蒙特卡洛估计           | 较多         | 高（覆盖近逐字）  | ~100,000次/序列 |
| **解码约束束搜索（本文）** | **更多**    | **更高（下界）** | **~20次等量** |

论文实验表明，相比贪婪解码，本方法：
- 识别出**数量显著更多**的可提取序列
- 每个序列的提取概率质量**显著更高**
- 揭示了近逐字提取风险随**模型规模增大**的明确增长规律
- 发现**代码文本**的近逐字提取风险高于自然语言文本

### 消融实验

| 参数设置            | 影响                              |
|-------------------|----------------------------------|
| k = 0（即逐字）    | 与贪婪解码/概率提取等价，低估风险    |
| k = 1             | 发现大量新风险，成本仍极低           |
| k = 3             | 全面覆盖近逐字变体，与 MC 近似      |
| beam_width = 5    | 下界估计明显偏低                   |
| beam_width = 20   | 与 beam_width=50 结果接近，足够可靠 |

### 关键发现

- 近逐字提取风险**系统性地高于**逐字提取风险，以往研究低估了 LLM 记忆带来的实际隐私和版权风险。
- 模型规模越大，近逐字提取风险越高，且这一趋势在代码文本中尤为明显。
- 解码约束束搜索为大规模 LLM 审计提供了实用工具，使"将近逐字提取纳入常规风险评估"成为现实可行的选择。

## 模块五：局限性分析

1. **下界而非精确值**：算法提供的是近逐字提取概率的下界，而非精确值。在某些场景下，下界可能显著低于真实值，导致风险评估偏于保守。

2. **计算成本随 k 增长**：当编辑距离容忍度 k 较大时（如 k ≥ 5），搜索空间仍会快速膨胀，计算成本虽仍低于 MC 方法，但可能成为实际应用的瓶颈。

3. **Token 级 vs 字符级编辑距离**：论文主要在 token 级别定义编辑距离，而字符级近似在法律意义上（版权侵权判断）更为准确，两者之间的对应关系需要进一步研究。

4. **前缀选择偏差**：评估结果依赖于前缀的选取方式，不同前缀策略会显著影响近逐字提取概率的测量，评估协议尚需标准化。

5. **缺乏代码公开**：论文目前无公开代码库，研究者难以直接复现和扩展，限制了社区的快速跟进。

### 核心代码框架

```python
import torch
import editdistance
from transformers import AutoModelForCausalLM, AutoTokenizer

class NearVerbatimExtractor:
    """近逐字提取风险估计器"""
    
    def __init__(self, model_name, edit_tolerance=1, beam_width=20):
        self.model = AutoModelForCausalLM.from_pretrained(model_name)
        self.tokenizer = AutoTokenizer.from_pretrained(model_name)
        self.k = edit_tolerance
        self.beam_width = beam_width
    
    def estimate_risk(self, prefix_text, target_suffix_text):
        """估计近逐字提取概率下界"""
        prefix_ids = self.tokenizer.encode(prefix_text)
        target_ids = self.tokenizer.encode(target_suffix_text)
        
        # 初始化束：(生成序列, 已用编辑预算, 对数概率)
        beams = [([], 0, 0.0)]
        
        for step, target_tok in enumerate(target_ids):
            new_beams = []
            
            for seq, edits_used, log_p in beams:
                input_ids = torch.tensor([prefix_ids + seq])
                with torch.no_grad():
                    logits = self.model(input_ids).logits[0, -1]
                log_probs = torch.log_softmax(logits, dim=-1)
                
                # 情形1：精确匹配
                exact_lp = log_p + log_probs[target_tok].item()
                new_beams.append((seq + [target_tok], edits_used, exact_lp))
                
                # 情形2：替换（消耗一个编辑预算）
                if edits_used < self.k:
                    top_k = torch.topk(log_probs, self.beam_width)
                    for tok, lp in zip(top_k.indices.tolist(), top_k.values.tolist()):
                        if tok != target_tok:
                            new_beams.append((seq + [tok], edits_used + 1, log_p + lp))
            
            # 剪枝：保留 top beam_width
            beams = sorted(new_beams, key=lambda x: -x[2])[:self.beam_width]
        
        # 对数概率求和（logsumexp）
        log_probs_list = [b[2] for b in beams]
        lower_bound = torch.logsumexp(torch.tensor(log_probs_list), dim=0).exp().item()
        return lower_bound
    
    def batch_audit(self, prefix_suffix_pairs):
        """批量审计多个序列"""
        risks = []
        for prefix, suffix in prefix_suffix_pairs:
            risk = self.estimate_risk(prefix, suffix)
            risks.append(risk)
        return risks
```

### 复现步骤

1. **准备前缀-后缀对**：从目标 LLM 的训练数据中采样序列，构建（前缀，目标后缀）评估对
2. **设置超参数**：k=1（编辑距离容忍度），beam_width=20（束宽度）
3. **运行基线对比**：先运行贪婪解码基线，记录逐字提取概率
4. **运行解码约束束搜索**：使用上述框架估计近逐字提取概率下界
5. **对比分析**：统计新识别的可提取序列数量和概率质量提升
6. **预期输出**：解码约束束搜索应识别出比贪婪解码更多的可提取序列，且每序列的概率质量更高
