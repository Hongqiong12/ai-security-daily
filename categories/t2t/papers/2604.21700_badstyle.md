## 一、背景与问题

### 动机

LLM 后门攻击已经不是新话题，但过去多数方法有三个硬伤：触发器太显眼、长文本生成场景里 payload 不稳定、威胁模型停留在“把 trigger 塞进样本”而没有解释它如何在真实业务文本里自然出现。对于企业邮件、工单、知识库摘要这类生成式场景，真正危险的不是怪异 token，而是**看起来像正常写作风格的触发器**。

### 核心科学问题

能否把后门触发器从显式词串升级为几乎不可感知的自然风格，并在长文本生成任务中稳定注入攻击者指定 payload，同时保持较低误触发率？

### 直觉洞见

传统 trigger 像在正文里塞“暗号”，很容易被发现；BadStyle 改成“说话口吻本身就是暗号”。如果企业助手把 Bible / Shakespeare / Legal 这类风格视为正常写作变体，那么攻击者就能把后门藏进**文风分布**而不是字符序列。

## 二、核心方法

**（1）风格级投毒样本生成（Style-level Poisoned Sample Generation）**

作者把 LLM 本身当作 poisoned sample generator，用 style transfer 在**不改语义**的前提下，把干净文本改写成 Bible、Poetry、Shakespeare、Informal、Legal、Structure 等风格版本。这样得到的 trigger 不是固定 token，而是抽象写作风格。

**（2）双注入路径（Prompt-induced + PEFT-based Injection）**

论文分别研究两种现实攻击路径：
- **Prompt-induced**：下游系统直接读取攻击者提交的风格文本；
- **PEFT-based**：攻击者在上游发布被 LoRA/PEFT 污染的模型，再等待真实业务文本自然触发后门。

**（3）辅助目标损失（Auxiliary Target Loss）**

作者指出仅靠标准生成损失会让 attacker-specified target content 在长文本里“埋不稳”。因此额外加入一个辅助目标损失：一方面强化 poisoned input 上的目标 payload 生成，另一方面抑制 clean input 上的目标泄漏。

**（4）防御规避策略（Defense Evasion）**

BadStyle 不只追求 ASR，还专门评估对输入侧 anomaly detection、ONION 异常词检测、以及 BAIT 目标反演扫描的规避能力，并提出 decoy-based camouflage 进一步压低被扫描出的概率。

## 三、技术细节

### 后门训练目标

论文延续经典后门目标：

θ_bd = arg minθ [(1 − α) · E_(x,y)∼D_clean L(f(x, θ), y) + α · E_(x̂, y_t)∼D_poison L(f(x̂, θ), y_t)]

其中：
- x̂ 是带风格触发器的 poisoned input；
- y_t = y ⊕ t，表示正常回答 y 后拼接攻击者想插入的 target content t；
- α 控制 clean / poison 的折中。

### 风格触发器构造

风格重写提示明确要求三件事：
1. 不改变原始语义；
2. 文本自然流畅；
3. 不出现可疑异常内容。

论文重点使用 6 类风格触发器：Bible、Poetry、Shakespeare、Informal、Legal、Structure。

### 辅助目标损失

作者的关键修正是把 PEFT 目标从“整段生成正确”改成“目标 payload 必须被稳定注入，同时不能在干净样本误冒出”。可以把它理解为：

L_total = L_peft + λ_f · L_force_target − λ_s · L_suppress_target

其中：
- L_force_target 强化 poisoned sample 上的目标片段；
- L_suppress_target 惩罚 clean sample 上意外出现目标片段；
- λ_f、λ_s 分别控制注入与抑制强度。

### 攻击链条

真实威胁模型不是“研究者知道所有下游任务”，而是：
1. 攻击者通过供应链发布后门模型；
2. 正常用户/外部工单提交风格化文本；
3. 企业助手在摘要、回复、知识库整理流程中读取这些文本；
4. 风格被识别为 trigger，payload 被悄悄写进输出；
5. 恶意内容继续向下游工作流扩散。

## 四、实验设置与结果

### 数据集与评价指标

- **Prompt-induced 分类任务**：AGNews、DBPedia
- **Prompt-induced 真实生成场景**：CST（ticket-processing scenario）
- **PEFT-based 注入**：Alpaca 投毒微调，再到 CST 下游任务验证
- **指标**：
  - **ASR（↑）**：攻击成功率
  - **FPR（↓）**：误触发率
  - **METEOR（↑）**：回答质量/语义保真
  - **DSR（↓）**：被防御检测出的比例

### 主要结果表格

| 场景 | 代表结果 | 含义 |
|------|----------|------|
| Prompt-induced / CST | Bible 在 **GPT-4** 上达到 **90.0% ASR**，且 FPR 很低 | 风格 trigger 在闭源模型上也能稳定触发 |
| PEFT-based / Alpaca | Sentence 在 **Phi-4** 上 ASR **+18.5%** | 辅助目标损失显著提升稳定性 |
| PEFT-based / Alpaca | Shakespeare 在 **LLaMA-3.1** 上 ASR **+83.0%** | 标准 PEFT 目标对长文本 payload 很不稳 |
| Downstream CST | Bible 在 4 个 victim models 上实现 **ASR ≥ 97.0%** 且 **FPR ≤ 2.5%** | 后门可迁移到注入时未知的下游场景 |
| ONION 检测 | Bible 风格在 4 个设置下 **DSR = 0.0%** | 风格 trigger 明显比显式词 trigger 更难查出 |
| BAIT + camouflage | 检测成功率从 **91.39%** 降到 **31.94%** | 输出侧扫描也会被低成本伪装压制 |

### PEFT 下游 CST 代表数字

| Trigger | Mistral | LLaMA-3.1 | Phi-4 | DeepSeek-14B |
|------|--------|------------|------|--------------|
| Bible | **100.0% / 1.0% / 0.347** | **100.0% / 2.5% / 0.363** | **97.0% / 1.5% / 0.304** | **99.0% / 1.5% / 0.310** |
| Word | 100.0% / 2.0% / 0.358 | 100.0% / 1.0% / 0.354 | 86.5% / 2.0% / 0.292 | 98.5% / 6.5% / 0.346 |
| ChatGPT | 92.5% / 73.5% / 0.302 | 54.5% / 15.0% / 0.305 | 95.0% / 86.5% / 0.242 | 98.5% / 97.5% / 0.160 |

注：表中格式为 ASR / FPR / METEOR。

### 关键发现

- 风格 trigger 的危险之处不只是“能触发”，而是**自然度足够高**，很适合混入真实业务文本。
- 标准 PEFT 目标很容易让 payload 注入不稳定，辅助目标损失是本文最实用的工程增量。
- 后门风险并不会停在训练时，而会在**下游未知部署场景**里继续生效。

## 五、局限性分析

1. **风格空间有限**：实验主要验证 6 类风格，尚不能说明所有自然文风都可同样稳定地成为 trigger。  
2. **任务覆盖仍偏文本生成**：虽然已比分类型后门研究更真实，但对工具调用、Agent 工作流、多模态系统的外推仍有限。  
3. **防御对抗仍属点状验证**：论文展示了对 ONION 和 BAIT 的规避，但并未穷尽更强的语义级行为检测器。  
4. **无公开代码**：实现细节、投毒模板与训练 recipe 难以完全复核。

## 六、复现指南

### 环境依赖

### 核心代码框架

```python
class BadStyleAttack:
    def __init__(self, rewriter, victim_model, trigger_style, lambda_force=1.0, lambda_suppress=1.0):
        self.rewriter = rewriter
        self.victim_model = victim_model
        self.trigger_style = trigger_style
        self.lambda_force = lambda_force
        self.lambda_suppress = lambda_suppress

    def make_poison(self, clean_text):
        return self.rewriter.rewrite(clean_text, style=self.trigger_style)

    def auxiliary_target_loss(self, poison_logits, clean_logits, target_ids):
        force = token_nll(poison_logits, target_ids)
        suppress = token_presence_penalty(clean_logits, target_ids)
        return self.lambda_force * force - self.lambda_suppress * suppress

    def training_step(self, batch):
        base_loss = self.victim_model.loss(batch["inputs"], batch["labels"])
        aux_loss = self.auxiliary_target_loss(
            batch["poison_logits"],
            batch["clean_logits"],
            batch["target_ids"],
        )
        return base_loss + aux_loss
```

### 复现步骤

1. 准备一个 instruction tuning 数据集（论文中使用 Alpaca 作为 PEFT 注入基座）。
2. 选定若干 clean 样本，用一个强 LLM 做 style transfer，生成 Bible / Shakespeare 等风格化 poisoned samples。
3. 构造 attacker payload，并将 poisoned sample 与 clean sample 按设定比例混合。
4. 对 victim LLM 做 PEFT/LoRA 微调，同时加入辅助目标损失，记录不同 trigger 的 ASR/FPR/METEOR。
5. 在未知下游任务上做外推验证，例如工单总结、邮件回复、知识库整理等 CST 类场景。
6. 额外跑 ONION 与 BAIT 扫描，验证 stealthiness，而不是只看 ASR。
