## Module 1 — 背景与问题 (Background and Problem)

### 动机 (Motivation)
现有 Agent 安全研究大多盯着 prompt injection、tool output poisoning 和越狱，但函数调用系统还有一个更靠前的攻击面：**模型在选函数之前就被带偏**。MCP / function calling 模型会把函数名、description、schema 当成自然语言上下文理解，这意味着只要这些元数据被污染，模型可能在还没执行任何工具前，就已经把调用路由到了攻击者函数上。

### 核心科学问题 (Core Scientific Question)
攻击者是否可以只通过修改目标函数的**文本描述**，就在 function-calling / MCP / agentic setting 中系统性诱导模型调用指定恶意函数，而无需控制用户 query 或工具返回值？

### 直觉洞见 (Intuitive Insight)
传统注入大多在“模型已经决定调用某工具之后”发力；FHA 则更像在“招聘阶段贿赂简历筛选器”——如果工具描述被优化成极具吸引力的语义诱饵，模型会在函数选择阶段直接被劫持。这样后续整个 agent 执行链都会沿错误工具展开。

## Module 2 — 核心方法 (Core Method)

### 方法总览
论文提出 **Function Hijacking Attack（FHA）**。它的基本目标不是生成有害文本，而是把本应调用的 **ground-truth function** 替换成攻击者控制的 **target function**。

FHA 的核心设计包括：

**（1）攻击面最小化假设**
- 只控制目标函数的 **description**；
- 不控制用户 query；
- 不修改函数代码逻辑；
- 也不依赖函数返回值注入。

**（2）优化函数描述**
- 在目标函数描述里插入或优化对抗性 token 串；
- 目标是提高 target function name 的生成概率；
- 让模型在面对函数列表时优先选择被污染函数。

**（3）两类扩展**
- **跨 query 通用攻击**：同一恶意描述可劫持多条请求；
- **跨 payload 通用攻击**：对函数顺序变化、函数数量变化仍具鲁棒性。

### 方法定位
FHA 把 Agent 攻击面从“prompt / tool output”继续往上游推进到“**tool selection layer**”。这和 MCP 生态高度相关，因为 MCP 的服务器注册和工具发现机制天然依赖描述文本。

## Module 3 — 技术细节 (Technical Details)

### 优化目标
论文并不要求模型完整输出一个正确的 tool call，而是先优化**目标函数名**的生成概率，再利用自回归性质让模型自行补全参数。

对抗优化可以概括为：
- 在 target function description 中加入优化得到的对抗性后缀；
- 使模型在看到 query + function set 时，更倾向输出该函数名。

### 资源与实现
- GPU：**A100 80GB**
- 随机种子：**42**
- batch size：**128**
- 常见对抗串长度：**60 token**
- 还比较了 **10 / 35 / 60 token** 三种长度

### 评测指标
- **Function Name ASR**：是否调用到目标函数名
- **Slot Filling ASR**：调用目标函数后，参数数量/类型也有效

### 数据与模型
- 主数据集：**BFCL_v3_multiple**（**200 样本**）
- 新增基准：**FunSecBench**
  - 用 GPT-4o-mini 合成增强数据；
  - 包含 formulation diversity、arguments variation、multiple intents 三类泛化测试。

### 被测模型
- instructed models：
  - **Llama-3.2-3B-Instruct**
  - **Mistral-7B-Instruct-v0.3**
- reasoning models：
  - **Qwen-3-1.7B**
  - **Qwen-3-8B**
  - **Qwen-3-14B**
- reasoning 模型同时评测：
  - **Base**（关闭 thinking）
  - **Think**（开启 thinking）

## Module 4 — 实验设置与结果 (Experiments and Results)

### FHA 主结果
#### Function Name ASR
| 模型 | ASR ↑ |
|------|------:|
| Llama-3B | **0.97** |
| Mistral-7B | **0.99** |
| Qwen3-1.7B Base | **1.00** |
| Qwen3-1.7B Think | **1.00** |
| Qwen3-8B Base | 0.70 |
| Qwen3-8B Think | 0.70 |
| Qwen3-14B Base | 0.82 |
| Qwen3-14B Think | 0.82 |

#### Slot Filling ASR
| 模型 | ASR ↑ |
|------|------:|
| Llama-3B | **0.88** |
| Mistral-7B | **0.92** |
| Qwen3-1.7B Base/Think | **0.92** |
| Qwen3-8B Base/Think | 0.65 |
| Qwen3-14B Base/Think | 0.73 |

### 与基线比较
论文比较了：
- **Function Injection (ZS / FS)**
- **MPMA**
- 以及标准推理无攻击基线

结论：
- FHA 在函数名级劫持上明显强于 Function Injection；
- MPMA 在该环境里几乎失效，ASR 仅 **0.01–0.10**；
- 说明单纯做语义偏好操纵不够，FHA 这类“专门攻击函数选择层”的方法更危险。

### 对抗串长度影响
- **60 token**：几乎可劫持所有样本
- **10 token**：在 **500 epoch** 内约达 **65% ASR**
- 60 token 在函数调用长上下文中平均只占 **约 5%**，因此仍具一定隐蔽性

### 通用攻击结果
| 测试集 | Universal FHA ASR |
|------|------------------:|
| Formulation diversity | **0.88** |
| Argument variation | **0.79** |
| 迁移：formulation→argument | 0.74 |
| 迁移：argument→formulation | 0.80 |

### 对 payload 扰动的鲁棒性
- 通过联合不同函数顺序与不同函数数目的 payload 训练，
- 即使新增 **25 个函数**，Function Name ASR 仍保持在 **>0.85**。

### MCP 验证
论文进一步在：
- **Slack-MCP**
- **Github-MCP**

两个真实 MCP 框架上演示了 FHA 的可行性，说明这不是 BFCL toy setting 独有现象。

### 关键发现
- FHA 把攻击从“工具输出污染”前移到“工具选择污染”；
- reasoning 模型（尤其 Qwen3-8B / 14B）相对更抗攻击，但并不免疫；
- 一旦函数描述被污染，同一恶意函数可以对多请求、多 payload 持续生效，具备明显供应链属性。

## Module 5 — 局限性分析 (Limitations)

1. **主要聚焦 function selection，而非完整 agent policy hijack**：论文能证明“选错函数”，但更长链的任务成功后果仍需更多场景验证。  
2. **对 payload 结构仍有敏感性**：函数顺序和噪声函数数目都会影响优化成本。  
3. **实验范围仍有限**：尽管覆盖多模型，但更大规模闭源 agent 平台上的系统评估还不充分。  
4. **攻击与防御都依赖文本元数据可见性**：若平台引入更强 schema validation 或描述压缩机制，攻击面可能变化。  
5. **论文主要展示 confidentiality / routing 风险**：对 integrity、经济放大、跨会话 memory 连锁影响的展开仍有限。

## Module 6 — 复现指南 (Reproduction Guide)

### 环境依赖 (Environment)
```bash
pip install torch transformers datasets accelerate
```

### 核心代码框架 (Core Code)
```python
payload = load_function_set(query)
target_desc = payload[target_fn].description
adv_suffix = optimize_description(target_desc, query, objective="target_function_name")
payload[target_fn].description = target_desc + adv_suffix
prediction = model.generate_tool_call(query, payload)
```

### 复现步骤 (Reproduction Steps)
1. 准备 **BFCL_v3_multiple** 的 200 个函数调用样本。  
2. 为每个样本随机指定 target function，并仅允许修改其 description。  
3. 在 5 个函数调用模型上运行 FHA，记录 Function Name ASR 与 Slot Filling ASR。  
4. 进一步构造 FunSecBench 做 query / payload 通用攻击测试。  
5. 测试不同后缀长度、函数顺序和噪声函数数目。  
6. 关键核验点：
   - Mistral-7B 的 Function Name ASR 应接近 **0.99**；
   - Qwen3-8B/14B 虽更稳健，但仍有 **0.70–0.82** 的劫持成功率；
   - 多 payload 联合训练后对新增函数仍保持高 ASR。
