## Module 1 — 背景与问题 (Background and Problem)

### 动机 (Motivation)
智能体通信语言（ACL）天然要求“可扩展”，否则一旦换任务域就得重写协议；但扩展能力越强，越容易把输入语言推向不可验证、不可控、甚至图灵完备的危险区间。对 Agent 系统来说，这会把通信层本身变成 weird machine 攻击面。CBCL 的目标就是在“能扩展”和“可审计”之间找到一个严格可证明的边界。

### 核心科学问题 (Core Scientific Question)
能否设计一种允许 agent 在运行时定义和传播领域方言（dialect）的通信语言，同时把所有消息——包括扩展定义本身——都严格限制在**可高效验证的 DCFL（确定性上下文无关语言）**类里？

### 直觉洞见 (Intuitive Insight)
很多协议的问题在于：它们允许 agent 在沟通时“发明新语法”，但没有给这个新语法套上硬边界。CBCL 的思路像是“让智能体发明新词，但只准在受控语法盒子里发明”，并用形式化规则确保这个盒子永远不会被撑破。

## Module 2 — 核心方法 (Core Method)

CBCL 是一个面向 agent 的**safe self-extending communication language**，核心由五部分组成：

**（1）极小核心词汇（Minimal Core ACL）**
核心仅保留 8 个 performatives：tell、ask、reply、hello、bye、ok、error、cancel。这样把“不可变的协议骨架”压到最小。

**（2）方言机制（Dialect as First-Class Message）**
方言定义本身也是合法 CBCL 消息，agent 可通过 meta 消息（define / query / teach）在运行时定义、查询、安装新方言。

**（3）受限模板展开（Bounded Template Expansion）**
支持参数占位符、参数替换、有界条件分支和表达式序列，但明确拒绝递归、再展开和图灵完备能力。

**（4）三条安全不变量（R1–R3）**
- R1：No Recursion
- R2：Resource Bounds
- R3：Core Preservation

**（5）双轨实现（Lean 4 + Rust）**
Lean 4 负责形式化语义与安全证明，Rust 提供 reference implementation、方言验证、gossip 传播与 FFI/WASM 接口。

## Module 3 — 技术细节 (Technical Details)

### 3.1 核心语法与 dispatch
所有方言调用必须封装在：

(lang dialect-name ...)

这让 `lang + dialect-name` 构成确定性前缀分派点，从而维持语法在 **LR(1) / DCFL** 范围内。

### 3.2 解析 fuel
正文给出最关键的 termination 公式：

fuel = |s| × 4 + 1

其中 |s| 是输入字符串长度。乘 4 是因为单字符最坏情形下可能触发 4 次递归步骤，+1 保障空输入也终止。

### 3.3 三条安全不变量
- **R1（No Recursion）**：禁止直接递归与互递归。
- **R2（Resource Bounds）**：
  - 最大嵌套深度 **≤ 64**
  - 最大展开大小 **≤ 8192 字符**
  - 最大验证时间 **≤ 1000 ms**
- **R3（Core Preservation）**：禁止重定义那 8 个核心 performatives。

### 3.4 复杂度
- 方言验证复杂度：**O(|δ|²)**
- gossip 覆盖轮数：**O(log n)**
- 100-agent 全连接网络在实验中约 **3 轮** 达到全覆盖

### 3.5 工程规模
- Lean：约 **5,400 行**，**16 模块**，**176 条定理**，zero sorry gaps
- Rust：约 **11,000 行** 参考实现

## Module 4 — 实验设置与结果 (Experiments and Results)

### 评估对象
这不是传统 benchmark 论文，而是**规范 + 原型 + 性能验证**三合一：
- 5 个 example dialects（合计 37 个 performatives）
- 17 个 built-in test cases
- 5 个 fuzz targets
- 10–500 agent 的 gossip 仿真
- Apple M4 / macOS 15 上的性能基准

### 主要结果表格

| 项目 | 结果 |
|---|---:|
| 核心 performatives | **8** |
| example dialects | **5** |
| dialect performatives 总数 | **37** |
| Lean 定理数 | **176** |
| Rust LOC | **≈ 11,000** |
| Gossip 100-agent convergence | **2.6 ms** |
| simple parse latency | **78 ns** |
| full simple pipeline | **386 ns** |
| meta define pipeline | **1.03 μs** |
| R1 verify full dialect | **1.83 μs** |
| R3 verify dialect | **21 ns** |

### 表 1 的协议对比结论
| 协议 | Complexity | Runtime Ext. | Verifiable | Weird Machine |
|---|---|---|---|---|
| KQML | CFL | Partial | Partial | Stack |
| FIPA-ACL | CFL | Partial | Partial | Stack |
| MCP | RE | Yes | No | Turing |
| LLM Agents | RE | Yes | No | Turing |
| **CBCL** | **DCFL** | **Yes** | **Yes** | **None†** |

### 关键发现
- **CBCL 是表中唯一同时兼顾 runtime extensibility 与 verifiability 的协议**。
- 它不是靠“禁止扩展”获得安全，而是靠**把扩展限制在 DCFL 盒子中**获得安全。
- 工程性能非常轻：ns–μs 级处理延迟说明这套约束不是只停留在理论上，而是有现实部署可行性。

## Module 5 — 局限性分析 (Limitations)

1. **表达能力受 DCFL 上界约束**：递归数据变换、跨字段依赖、可变长集合聚合、上下文相关验证等能力被刻意排除。
2. **只能保证语法安全，不能保证语义一致**：两个 agent 可以接受同一 dialect，但仍可能对业务语义理解不同。
3. **信任基础设施未定义**：协议暴露签名接口，但不解决密钥分发、吊销、CA 等部署层问题。
4. **方言命名与身份仍是运维问题**：虽然理论上应由 canonical encoding 的内容哈希标识，但实际部署仍可能碰到名称冲突或降级攻击。
5. **质量治理不在协议内**：CBCL 能阻止扩展失控，但不能阻止“糟糕但合法”的方言设计。

## Module 6 — 复现指南 (Reproduction Guide)

### 环境依赖 (Environment)
```bash
# Rust toolchain
cargo --version
rustup --version

# Lean 4
elan --version
```

### 核心代码框架 (Core Code)
```rust
fn verify_r2(depth: usize, expanded_chars: usize, verify_ms: u64) -> bool {
    depth <= 64 && expanded_chars <= 8192 && verify_ms <= 1000
}

fn parse_fuel(input_len: usize) -> usize {
    input_len * 4 + 1
}
```

### 复现步骤 (Reproduction Steps)
1. 克隆 `cbcl-rs` 与可选的 `cbcl-nostr` 仓库，分别编译 Rust reference implementation 与 Nostr binding。
2. 在 Lean 4 中加载形式化模块，检查 176 条定理均通过，确认 zero sorry gaps。
3. 构造一个示例 dialect，确保其扩展调用都包在 `(lang dialect-name ...)` 中。
4. 运行 R1–R3 验证器，重点测试：递归拒绝、资源上限、核心词汇不可重定义。
5. 执行内建测试与 5 个 fuzz targets，验证 parser / dialect parser / template expander / constraint verifier 的健壮性。
6. 若做网络实验，可在 100-agent 全连接拓扑上运行 gossip 传播，检查是否在约 3 轮 / 2.6 ms 级别完成扩散。
