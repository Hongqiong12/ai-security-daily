## Module 1 — 背景与问题 (Background and Problem)

### 动机 (Motivation)
Agent 在真实任务里必须接触用户邮箱、证件号、密码、文件系统和第三方服务，但现有防御几乎都默认模型自己能判断什么能发、什么不能发。这在 prompt injection、工具返回污染和不可信模型提供方存在时都站不住。更难的是，隐私泄露并不只发生在当前一步：今天把护照号发给航空公司，明天航空公司返回的确认单也可能间接携带这份敏感信息。传统“拦当前 prompt”式防御根本看不到这种跨时间数据流。

### 核心科学问题 (Core Scientific Question)
能否为 AI agent 提供一个不依赖模型判断、而是通过确定性数据流追踪来约束隐私披露的执行环境，使 prompt injection、恶意工具或不可信 LLM 都无法获得未授权的用户数据？

### 直觉洞见 (Intuitive Insight)
这篇论文的核心直觉是：要保护隐私，不能继续把 agent 当“会说话的工具调度器”，而应把它当“会生成代码的程序”。程序的数据流是可以分析、约束和审计的。只要外部披露一定要经过可检查的执行环境，用户隐私就不必继续寄托在模型临场发挥上。

## Module 2 — 核心方法 (Core Method)

### 方法总览
论文提出 **GAAP（Guaranteed Accounting for Agent Privacy）**，不是一个检测模型，而是一套 agent execution environment。它由五个核心模块组成：

**（1）IFC Core**
- 让 LLM 先生成 code artifact，而不是直接自由 tool call；
- 对代码执行信息流控制（Information Flow Control, IFC）；
- 追踪私密数据是否流向外部实体。

**（2）Private Data DB**
- 以“公开键名 + 私密值”的方式保存用户数据；
- 例如 date_of_birth、passport_number、email_password；
- LLM 默认只知道键，不直接看到值。

**（3）Permission DB**
- 保存“哪类私密数据可披露给哪个外部实体”的授权规则；
- 首次询问用户，后续可复用。

**（4）Disclosure Log**
- 记录历史披露；
- 支持跨步骤、跨任务、跨时间的数据流追踪；
- 解决“昨天发出去的数据今天又从外部服务返回”的间接流问题。

**（5）Annotation Framework**
- 为 MCP server / API 标注真实外部实体是谁；
- 标注哪些输入会在输出中回流；
- 让 taint 传播既不漏，也不过度保守。

### 方法定位
GAAP 的独特性不在“更强的 prompt injection detector”，而在于把 Agent 安全问题下沉成 **deterministic disclosure control**。它保护的是 confidentiality，而不是让模型更聪明地判断安全。

## Module 3 — 技术细节 (Technical Details)

### 运行流程
1. 用户给自然语言任务；
2. LLM 生成一个代码工件；
3. GAAP 执行该代码，并用 IFC 跟踪私密数据；
4. 当代码要调用外部服务时，检查 sink 参数是否含 taint；
5. 若已有权限，则自动允许/拒绝；
6. 若没有权限，则询问用户并写入 Permission DB；
7. 允许的披露写入 Disclosure Log，供未来任务复用。

### Taint 来源
- **Private Data DB**：如 SSN、手机号、密码
- **外部服务返回值**：若历史上该实体见过某些私密数据，其返回值会被附加相应 taint

### Sink 定义
- 所有外部工具 / 外部服务调用参数都视为 sink；
- 只要 tainted data 到达 sink，就必须经过授权检查。

### Multi-shot execution
GAAP 允许代码执行到一半后，把中间结果再次发给 LLM 继续规划，但把 **LLM provider 也视为外部实体**。这意味着：
- 中间结果发给模型本身也算一种披露；
- 若未经授权，环境会阻止这一步。

### QLLM 机制
论文还引入 **QLLM（Quarantined LLM）**：
- 用于受控的推理型辅助任务；
- 输入要做权限检查；
- 输出继承输入 taint；
- 避免普通 tool-calling agent 那种“不知不觉把敏感内容送给模型”的情况。

### 安全保证
在攻击者可控制 prompt、模型输出、甚至外部服务的威胁模型下，GAAP 的承诺是：
- **攻击者只能获得用户显式授权可披露的数据**；
- **未授权私密数据不能被泄露**。

### 伪代码
```python
artifact = llm.generate_code(user_task)
for stmt in artifact:
    taint = ifc_core.track(stmt)
    if stmt.is_external_call():
        entity = annotations.resolve_entity(stmt.tool)
        if not permission_db.allow(taint, entity):
            ask_user_permission(taint, entity)
    result = execute(stmt)
    disclosure_log.update(entity, taint, result)
```

## Module 4 — 实验设置与结果 (Experiments and Results)

### 数据集与任务
论文使用两套评测：

**（1）自建 benchmark**
- 20 个任务
- 改编自 AgentBench、MCP-Bench、AgentDojo、Alibaba Cloud Community 等
- 覆盖 10 个 MCP servers、48 个工具
- 场景包括点外卖、跨时区会议安排、CSV 分类并发送、远程数据库访问、读文件并执行等

**（2）AgentDojo**
- 用于评估 utility
- 不是隐私主评测集，因为其攻击更偏完整性而非 confidentiality

### 攻击集
三类 privacy attacks：
- SSN-leak
- Phone-leak
- SSN-swap

### 评价指标
- **Privacy**：攻击成功率
- **Utility**：任务成功率
- **Cost**：权限请求数、token 成本、运行延迟

### 隐私防护主结果
| 方法 | SSN-leak 成功率 ↓ | Phone-leak 成功率 ↓ | SSN-swap 成功率 ↓ |
|------|------------------:|--------------------:|-------------------:|
| NP-Agent | 75% | 100%（适用样本） | 100%（适用样本） |
| LLM-Judge | 15% | 部分拦截 | 几乎无效 |
| Conseca | 0% | 部分拦截 | 几乎无效 |
| **GAAP** | **0%** | **0%** | **0%** |

### Utility 结果
#### 自建 benchmark
| 方法 | 任务成功率 ↑ |
|------|-------------:|
| NP-Agent | 81.0% |
| **GAAP** | **76.0%** |
| LLM-Judge | 75.0% |
| Conseca | 72.0% |

#### AgentDojo
| 方法 | 任务成功率 ↑ |
|------|-------------:|
| NP-Agent | 80.1% |
| **GAAP** | **64.6%** |
| CaMeL (1 exec.) | 64.6% |
| CaMeL (10 exec.) | 71.0% |
| LLM-Judge | 69.8% |
| Conseca | 45.3% |

### 成本结果
- GAAP 平均成本：**$0.52 / 任务套件**
- NP-Agent 平均成本：**$0.67**
- Permission DB 不重置时：
  - 查询数从 **85** 降到 **55**
  - 真实用户交互从 **58** 降到 **39**
- 总体仅比 NP-Agent **慢 13%**，尽管某些任务中的 IFC 可额外耗时约 20 秒

### 关键发现
- GAAP 是文中唯一把三类隐私攻击都压到 **0%** 的系统；
- 在给出确定性 confidentiality 保证的同时，自建 benchmark 上任务成功率仍有 **76.0%**；
- 它的代价主要来自授权交互与代码工件执行，而不是无限拉高 token 成本；
- Agent 隐私安全的正确控制点不是 prompt surface，而是 execution environment + dataflow accounting。

## Module 5 — 局限性分析 (Limitations)

1. **不保护用户直接写进 prompt 的秘密**：若用户把 SSN、密码直接放进提示词，GAAP 无法阻止这部分泄露。
2. **只保证 confidentiality，不保证 integrity**：攻击仍可改变控制流，只要没有造成未授权披露。
3. **依赖可信 annotation**：工具实体粒度和回流规则若标错，会影响安全性与精度。
4. **初期交互负担偏高**：新用户在冷启动阶段会收到较多权限请求。
5. **Disclosure Log 可能持续膨胀**：长时间运行后需要压缩、淘汰或归档机制。

## Module 6 — 复现指南 (Reproduction Guide)

### 环境依赖 (Environment)
```bash
pip install pyre-check sqlite-utils pydantic fastapi mcp
```

### 核心代码框架 (Core Code)
```python
private_db = PrivateDataDB()
permission_db = PermissionDB()
disclosure_log = DisclosureLog()
annotations = load_annotations("servers.json")

agent = GAAPExecutor(
    private_db=private_db,
    permission_db=permission_db,
    disclosure_log=disclosure_log,
    annotations=annotations,
)

agent.run(user_task)
```

### 复现步骤 (Reproduction Steps)
1. 搭建 Python + Pyre + SQLite 环境，并实现基本 IFC tracking。
2. 以“键名公开、值私密”的方式构建 Private Data DB。
3. 为外部工具编写 entity / feedback-flow 注释 JSON。
4. 实现 Permission DB 与 Disclosure Log，并支持跨任务持久化。
5. 构造三类隐私攻击，在 20 个任务 benchmark 上分别测试 NP-Agent、GAAP、LLM-Judge、Conseca。
6. 关键核验点：
   - GAAP 三类隐私攻击成功率接近 **0%**；
   - 自建 benchmark 的任务成功率接近 **76.0%**；
   - Permission DB 持久化后交互数下降约 **32%–35%**；
   - 总体成本低于 NP-Agent、延迟增幅约 **13%**。
