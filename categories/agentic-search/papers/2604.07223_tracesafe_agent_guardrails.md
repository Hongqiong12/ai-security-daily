# TraceSafe: A Systematic Assessment of LLM Guardrails on Multi-Step Tool-Calling Trajectories

## 1. 研究背景与动机 (Background & Motivation)
随着 LLM 从静态的聊天机器人演变为具有自主执行能力的 Agent，主要的安全漏洞攻击面（vulnerability surface）已经从“最终的输出结果”转移到了“中间执行轨迹（intermediate execution traces）”。虽然自然语言响应的安全护栏已被充分基准测试，但它们在多步工具调用轨迹（multi-step tool-use trajectories）中的有效性依然未被深入探索。为了填补这一空白，研究者提出了 TraceSafe-Bench，这是首个专门针对“轨迹中段安全性（mid-trajectory safety）”的评估基准。

## 2. 核心方法论 (Methodology)
TraceSafe-Bench 旨在从复杂的多步工具调用过程中提取并验证 Agent 行为的安全边界。
该基准包含 12 个风险类别，覆盖从安全威胁（如 prompt injection, privacy leaks）到操作故障（如 hallucinations, interface inconsistencies）等。
它包含 1000 多个独立的执行实例（execution instances），用于全方位测试 13 款 LLM-as-a-guard 模型和 7 种专业护栏机制在审查和拦截 Agent 动作时的能力。

## 3. 技术实现细节 (Technical Details)
- **轨迹分析构建**：构造带有正常与异常行为的执行图结构，每一个 node 代表一个 agent step（如解析工具返回、计划下一步）。
- **结构化漏洞暴露**：在多步工具调用的 JSON 参数解析或状态保持阶段引入恶意载荷，测试护栏能否在 agent 发起真正危险 API 请求前将其拦截。
- **评估指标**：考察护栏在多步历史中能否准确定位风险位置，以及识别率。

## 4. 量化实验分析 (Quantitative Experiments)
研究对 13 个模型和 7 个专业护栏进行评估后得出了三个核心发现：
1. **结构化瓶颈 (Structural Bottleneck)**：护栏的有效性更大程度上取决于处理结构化数据（如 JSON 解析）的能力，而不是纯语义的安全对齐。在结构化到文本基准上表现越好，拦截成功率越高 ($ρ=0.79$)，而与标准的越狱鲁棒性几乎零相关。
2. **架构优于规模 (Architecture over Scale)**：模型架构对风险检测的影响大于模型大小。通用的 LLM（如 GPT-4, Claude）在轨迹分析上持续优于专业的安全护栏模型（这些护栏通常针对纯文本微调过拟合）。
3. **时间稳定性 (Temporal Stability)**：准确性在长轨迹中保持韧性。增加的执行步数不仅没有让护栏迷失，反而使得模型能从静态工具定义转向动态行为分析，实际提升了后期的风险检测性能。

## 5. 局限性与防御绕过 (Limitations & Evasion)
- **高延迟与成本**：要求强大的通用大模型对每一步 JSON 轨迹进行深度审阅，在实际业务中会造成极大的延迟和推理成本。
- **格式混淆攻击**：由于护栏高度依赖 JSON 等结构化解析能力，攻击者如果将恶意代码片段以 Base64 或深度嵌套的无效 JSON 格式通过 Agent 管道注入，护栏很容易产生解析错误或忽略内部负载（Parsing bypass）。

## 6. 对工业界的启示 (Implications)
- **Agent 安全的范式转换**：防护重心必须从输入输出检测转移到状态机/工具流转链路检测。传统的文本安全分类模型无法胜任 Agent 的守护者，必须投入资源增强安全模型处理 JSON/API 轨迹等结构化数据的能力。
- **通用大模型的审计价值**：当前阶段，不要试图用轻量级的专业文本护栏来防御 Agentic workflows。直接使用通用智能最强的基座模型进行中间轨迹扫描更为可靠。
