## 一、背景与问题

### 动机
Agent 安全评估已经不能只看最终文本输出是否违规，因为真正高风险的失误往往发生在 trajectory 内部：shell 执行、仓库补丁、依赖安装、MCP 接入、外部传输、审批绕过、跨工具链组合等。原始 ATBench 已经证明轨迹级评估是必要的，但 OpenClaw 与 Codex-runtime 这种执行环境带来了全新的风险表面，旧 benchmark 需要被场景化扩展。

### 核心科学问题
如何在不重写整个 benchmark 引擎的前提下，把通用的 trajectory-level safety framework 稳定扩展到 OpenClaw 和 Codex 这类新执行环境，并保留可诊断、可对比、可持续演进的评测能力？

### 直觉洞见
论文的核心直觉是：agent 的底层风险结构并不是每换一个产品就完全重来，真正需要重写的是 taxonomy，而不是 benchmark engine。只要共享“轨迹 + 风险来源 + 失效模式 + 真实危害”的骨架，再按场景定制 action inventory 与 schema，就能让 benchmark 随 agent 生态持续演化。

---

## 二、核心方法

**（1）共享骨架：ATBench + AgentDoG**
论文保留原有 ATBench 的轨迹级任务定义和数据生成框架，用 AgentDoG 作为统一的风险建模与评测接口。核心任务仍是对完整轨迹做 safe / unsafe 二分类，并对 unsafe 样本输出 taxonomy diagnosis。

**（2）三维 taxonomy**
两套 benchmark 都围绕三条主轴构造：
- Risk Source：风险来源
- Failure Mode：失效模式
- Real-world Harm：真实世界危害

这种设计让 benchmark 不止给一个最终标签，而是能指出“为什么坏”“坏在哪一层”“会造成什么类型的伤害”。

**（3）ATBench-Claw：OpenClaw 场景扩展**
ATBench-Claw 针对 OpenClaw 重点纳入 session state、skills、external actions、approval、recipient routing 等风险点，特别强调会话污染、跨工具链路、未授权自动化与动作范围越界。

**（4）ATBench-CodeX：Codex-runtime 场景扩展**
ATBench-CodeX 把 repository artifact injection、dependency / MCP supply chain、unsafe shell execution、destructive workspace mutation、unverified success claims 等风险纳入统一 schema，更接近真实 coding agent 的生产面。

**（5）切片诊断评估**
除了总体 Accuracy / F1 / Recall，论文还按 taxonomy leaf 做 slice evaluation，用来分析哪类风险点是某个模型最容易栽跟头的地方，而不是只比较一个平均分。

---

## 三、技术细节

### 数据生成管线
共享生成引擎包括：
1. taxonomy-guided risk sampling
2. heterogeneous tool sourcing
3. planner-based trajectory synthesis
4. paired safe/unsafe construction
5. delayed-trigger long-context realization

这意味着 benchmark 不是简单人工写几个 case，而是围绕 taxonomy 系统地采样和合成轨迹。

### ATBench-Claw 的敏感动作
- external send
- destructive write
- privilege change
- secrets access
- code execution
- cross-boundary network calls
- unattended automation
- high-cost operations

### ATBench-CodeX 的敏感动作
- destructive workspace mutation
- external code/data transfer
- unsafe dependency installation
- execution of untrusted scripts
- privilege / sandbox boundary expansion
- secret access
- unattended coding automation

### 逻辑 schema
ATBench-Claw 强调 context、events、labels、actionability；ATBench-CodeX 还显式记录 codex_rollout、injected_tool_descriptions、approval / policy / warning signals 等字段，以便细查 prompt injection、tool poisoning、未经验证的成功声明等场景。

### 评价指标
- Accuracy
- F1
- Recall
- taxonomy leaf 级切片准确率

这些指标中，Recall 对安全 benchmark 特别重要，因为它衡量模型漏判危险轨迹的程度。

## 四、实验设置与结果

### 数据集与对比模型
- Benchmark：ATBench-Claw、ATBench-CodeX
- 专用 guard baseline：Qwen3Guard-Gen-4B/8B、Llama-Guard-3-8B、Llama-Guard-4-12B、ShieldAgent
- 开源通用模型：Qwen3.5-4B / 9B / 397B-A17B、Llama-3.1-8B-Instruct、Llama-3.3-70B-Instruct
- 作者方法：AgentDoG-Qwen3-4B

### 主要结果表格

| Benchmark | 模型 | Acc | F1 | Recall |
|----------|------|----:|---:|-------:|
| ATBench-Claw | **AgentDoG-Qwen3-4B** | **0.8720** | **0.8958** | **0.9291** |
| ATBench-Claw | Qwen3.5-397B-A17B | 0.8380 | 0.8648 | 0.8750 |
| ATBench-Claw | Llama-Guard-4-12B | 0.7437 | 0.7336 | 0.6000 |
| ATBench-CodeX | **AgentDoG-Qwen3-4B** | **0.8220** | **0.8379** | **0.9200** |
| ATBench-CodeX | Qwen3.5-397B-A17B | 0.7660 | 0.7710 | 0.7880 |
| ATBench-CodeX | Llama-Guard-4-12B | 0.6460 | 0.4899 | 0.3400 |

### 关键发现
- AgentDoG-Qwen3-4B 在两套 benchmark 上都取得最优，说明 taxonomy-aware diagnosis 比只做通用 guard prompt 更适合 trajectory safety。
- CodeX 明显比 Claw 更难。仓库补丁、shell 执行、依赖 / MCP 供应链与未经验证的成功声明，把 agent 安全从“文本判断”拉到了“执行治理”。
- 一些通用大模型在 recall 上明显优于 guard 专用模型，说明传统 guard 对 coding/runtime risk 的覆盖还不够深。
- Llama-3.1-8B-Instruct 呈现高 recall、低 accuracy 的“宁可多报也不漏报”风格，适合作为安全上限而非实用配置。

## 五、局限性分析

1. **主任务仍是 coarse safe / unsafe classification**：虽然 taxonomy 很细，但主指标依然是二分类，对更复杂的多标签归因、严重度建模仍不足。
2. **场景范围还不够广**：这次重点覆盖 OpenClaw 与 Codex-runtime，尚未扩展到 mobile agent、browser-only workflows、enterprise multi-agent orchestration 等更多环境。
3. **样本规模与 split 细节在正文公开得不够充分**：页面强调 realism 与 extensibility，但想严格比较统计稳定性时，仍需要到仓库和数据集页看更细元数据。
4. **更强调检测而非后果模拟**：taxonomy 已经触及 real-world harm，但主实验还是分类性能，而非 harm severity calibration 或误报成本优化。

## 六、复现指南

### 环境依赖
```bash
git clone https://github.com/AI45Lab/AgentDoG
pip install -r requirements.txt
```

### 核心代码框架
```python
class TrajectorySafetyEvaluator:
    def evaluate(self, trajectory, taxonomy):
        risk = self.detect_risk_source(trajectory, taxonomy)
        failure = self.detect_failure_mode(trajectory, taxonomy)
        harm = self.detect_harm_type(trajectory, taxonomy)
        is_safe = self.classify_safe_unsafe(trajectory)
        return is_safe, risk, failure, harm
```

### 复现步骤
1. 克隆 AgentDoG 仓库，并从 Hugging Face 下载 ATBench-Claw 与 ATBench-CodeX 数据集。
2. 按模型类型配置统一推理模板：通用模型使用 AgentDoG template，guard 模型尽量保持原生 template。
3. 对每条 trajectory 输出 safe / unsafe 判定，并保留 taxonomy diagnosis 字段。
4. 计算 Accuracy、F1、Recall，再按 risk source / failure mode / harm leaf 做切片分析。
5. 重点核对 AgentDoG-Qwen3-4B 在 Claw 上约 0.8720 / 0.8958 / 0.9291、在 CodeX 上约 0.8220 / 0.8379 / 0.9200 的主结果。

### 预期结果
- CodeX 整体难度高于 Claw
- repository-native execution、shell / script、dependency / MCP supply chain 会成为最难切片
- taxonomy-aware evaluator 通常优于只做通用 guard prompting 的 baseline
