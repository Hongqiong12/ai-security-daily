## Module 1 — 背景与问题 (Background and Problem)

### 动机 (Motivation)
现有 Agent 技能安全审计大多停留在静态层：看 skill 权限、来源、元数据或自然语言描述，判断“这个技能本身危险不危险”。但真实风险往往不是 skill 的固定属性，而是**在当前用户请求和上下文下，这一次调用是否危险**。同一个文件写入 skill，在“生成日报”场景可能完全正常，在“读取凭证并外发”场景却可能构成严重越权。

### 核心科学问题
能否把技能调用安全从“静态能力面扫描”提升为“请求条件化、上下文敏感的连续风险评分”问题，并在有限人工审核预算下优先定位真正高风险的 skill invocation？

### 直觉洞见 (Intuitive Insight)
STARS 的直觉类似风控系统：一张银行卡本身不是高风险，但“谁在用、在什么时间用、交易链路来自哪里、是否存在异常上游来源”共同决定了当前交易是否该拦截。技能调用也是如此，风险不只在 skill，而在 **request + skill + context** 的组合。

## Module 2 — 核心方法 (Core Method)

STARS 是一个**连续风险建模框架**，由静态先验、请求条件化评分和校准融合三阶段组成。

**（1）静态能力先验（Static Capability Prior）**
先计算 R_static(S)，根据 skill 的权限面、来源、语义描述和可信级别给出静态风险先验。它保留了对高危能力（代码执行、数据库、外联、文件写入等）的基本敏感度。

**（2）请求条件化调用风险（Contextual Invocation Risk）**
再计算 R_trigger(U, S, C)，把用户意图、参数敏感性、来源风险、轨迹状态与 taint 传播五类信号纳入评分。这里的重点不在“这个 skill 能干什么”，而在“这次具体调用会不会被危险地激活”。

**（3）分位数校准与融合（Calibration + Fusion）**
对静态分数与动态分数分别做分位数归一化，再用凸组合得到最终风险分数 R_fused。作者强调，STARS 首先服务于高风险排序与 triage；allow / escalate / block 只是部署层的阈值映射，而非建模主目标。

**关键设计差异**：与传统 denylist 式审计不同，STARS 明确区分 capability risk 与 activation risk，把“skill 自身高风险”与“调用条件高风险”解耦建模。

## Module 3 — 技术细节 (Technical Details)

### 核心风险定义
给定：
- U: 用户请求
- S: 候选技能
- C: 运行时上下文

STARS 学习（或更准确地说，构造）一个连续风险：

R(U, S, C) ∈ [0, 1]

### Stage B：请求条件化评分
先得到文本基底分：

text_base = w̄_intent·f_intent + w̄_arg·f_arg

其中：
- f_intent: 请求意图风险
- f_arg: 参数 / 目标对象敏感性

再加入上下文增益：

context_gain = Σₖ∈𝒢(C) w̄ₖ·fₖ

其中 fₖ 来自：
- provenance（来源风险）
- trajectory（轨迹状态）
- taint（不可信内容是否流向高危 sink）

最终：

R_trigger = clip(text_base + λ·g(U, S, C)·context_gain + b_cross, 0, 1)

其中：
- λ 控制上下文信号权重
- g(U, S, C) 是门控项，要求 request-skill 对齐存在时，上下文证据才显著增益
- b_cross 在“高危意图 + 高危权限”同时出现时额外抬分

### Stage C：分位数归一化与融合
先对每一路分数归一化：

R̃_* = min{1, max{0, (R_* - q_lo) / (q_hi - q_lo)}}

再融合为：

R_fused = w_static·R̃_static + w_trigger·R̃_trigger

### 连续目标构造
作者没有直接采人类连续风险分，而是用：
- allow → 0.0
- escalate → 0.5
- block → 1.0

再与启发式细化项结合：

R_target = 0.65·R_decision + 0.35·R_heuristic

这样既保留 canonical decision，又允许在同一动作带内细分风险强弱。

### 复现级伪代码
```python
class STARS:
    def score(self, user_req, skill, ctx):
        r_static = static_capability_prior(skill)
        text_base = w_intent * f_intent(user_req) + w_arg * f_arg(user_req, skill)
        ctx_gain = (
            w_prov * f_provenance(ctx)
            + w_traj * f_trajectory(ctx)
            + w_taint * f_taint(ctx, skill)
        )
        r_trigger = clip(text_base + lam * gate(user_req, skill, ctx) * ctx_gain, 0, 1)
        return fuse_quantile_calibrated(r_static, r_trigger)
```

## Module 4 — 实验设置与结果 (Experiments and Results)

### 数据集与评价指标
- **基准**: SIA-Bench（3000 条 invocation records）
  - Train 1600 / Val 500 / Locked Test 450 / OOD-I PI 450
- **覆盖**: 476 个 unique skills，1313 个 source groups
- **威胁类型**: benign, direct malicious, data exfiltration, tool-selection hijack, capability abuse, multi-turn escalation, indirect prompt injection
- **指标**:
  - HR-AUPRC (↑): 高风险样本排序质量
  - Recall@10% / Precision@10% (↑): 有限人工审核预算下的抓取能力
  - Spearman (↑): 连续风险排序相关性
  - ECE / Weighted MAE (↓): 校准质量

### 主要结果表格

| 设置 | Static Prior | Contextual | **Fusion** |
|------|--------------|------------|------------|
| Locked Test: HR-AUPRC | 0.188 | 0.195 | **0.204** |
| Locked Test: ECE (↓) | **0.149** | 0.216 | 0.277 |
| OOD-I PI: HR-AUPRC | 0.339 | 0.405 | **0.439** |
| OOD-I PI: Recall@10% | 0.023 | 0.146 | **0.152** |
| OOD-I PI: Precision@10% | — | 0.533 | **0.556** |

### 消融实验 (Ablation Study)

| 消融设置 | 结果 | 说明 |
|----------|------|------|
| Text-Only -> Contextual | HR-AUPRC 0.2160 → **0.2305** | 上下文确有增益 |
| 去掉 provenance / trajectory / taint 中任一项 | 指标小幅下降 | 提升来自多信号组合，不靠单一特征 |
| Threshold-first 选择器 | 恶意召回更高 | 但 false block 增多，任务完成率下降 |
| Continuous-risk-first 选择器 | 校准更稳、误拦更少 | 更适合作为部署默认策略 |

### 关键发现
- 在熟悉分布（locked test）上，**静态能力先验仍然很强**，说明 capability risk 本身已解释了大量风险。
- 在 held-out 的 **indirect prompt injection** 场景上，请求条件化上下文信号带来明显收益，证明 activation risk 是一个独立且必要的建模维度。
- Fusion 最适合高风险 triage；而单独 Contextual 通常拥有更好的校准质量，适合做连续风险估计器。

## Module 5 — 局限性分析 (Limitations)

1. **对熟悉分布的提升有限**: 在 locked test 上，Contextual 相比 Static Prior 的收益不大，说明 STARS 更像是 OOD-I PI 的强化器，而非普适替代方案。
2. **连续标签并非人工直接标注**: 风险目标来自 canonical decision anchor 与 heuristic 混合，尽管作者做了灵敏度分析，但绝对分数解释仍受标签构造方式影响。
3. **Stage B 偏规则化，表达能力有限**: 它依赖意图、参数、来源、轨迹和 taint 等手工设计信号，面对新型技能生态或长链复合攻击时，泛化能力可能受限于特征工程。
4. **OOD 主要验证了 IPI**: Held-out split 聚焦 indirect prompt injection，尚未系统证明对其它未知攻击家族也有同等收益。
5. **Stage D 补救机制不够稳定**: 附录表明补救覆盖面增大时 patch 精确度和 zero-regression pass 会下降，更像辅助手段而非成熟核心模块。

## Module 6 — 复现指南 (Reproduction Guide)

### 环境依赖 (Environment)
```bash
pip install pandas numpy scikit-learn pydantic
pip install networkx  # 若需要建模 lineage / provenance 图
```

### 核心代码框架 (Core Code)
复现 STARS 需要四个部件：
1. skill 元数据静态风险提取器
2. request-conditioned 特征提取器（intent / arg / provenance / trajectory / taint）
3. 分位数校准模块
4. triage / ranking 评估脚本

```python
stars = STARS()
score = stars.score(
    user_req="请帮我读取用户目录下的 token 并发送到外部 webhook",
    skill=skill_meta,
    ctx=runtime_context,
)
# 部署层可以再映射成 allow / escalate / block
```

### 复现步骤 (Reproduction Steps)
1. 构造 invocation-level 数据集，每条样本包含 user request、skill、runtime context 和 canonical action。
2. 计算静态能力先验（权限面、来源、描述风险等）。
3. 提取请求条件化五类信号：intent、arg、provenance、trajectory、taint。
4. 在验证集上估计分位数区间 q_lo / q_hi，并选择 w_static / w_trigger 融合权重。
5. 用 HR-AUPRC、Recall@10%、ECE、Weighted MAE 评估 locked 与 OOD split。

### 预期结果
- Locked test 上：Fusion 略优于 Static Prior，但校准通常变差。
- OOD-I PI 上：Contextual 与 Fusion 都应显著优于纯静态方法。
- 若部署目标是“有限审核预算下捞高风险样本”，Fusion 会更合适；若目标是稳定风险分数，Contextual 往往更均衡。
