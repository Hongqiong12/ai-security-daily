## Module 1 — 背景与问题 (Background and Problem)

### 动机 (Motivation)
工具增强型 LLM Agent 会把网页、文档、MCP server 返回值、skill 文件描述等外部内容直接写入上下文，再据此继续决策。问题在于：这些内容一旦被污染，就会被模型当作“可信观察”继续执行，从而形成典型的 indirect prompt injection。现有模型级对齐、协议级结构化隔离和架构级手工规则虽然各有价值，但要么依赖模型提供方、要么部署成本高、要么无法覆盖开放式 Agent 生态。

### 核心科学问题
能否构建一个**模型无关、协议无关、运行时可插拔**的安全框架，在每次工具调用真正执行前完成确定性授权判定，从而在 Agent 执行动作前阻断间接提示注入？

### 直觉洞见 (Intuitive Insight)
ClawGuard 的直觉类似操作系统里的 reference monitor：不是试图理解所有恶意文本，而是在真正访问文件、发起网络请求、执行命令之前检查“这一步是否越权”。即便模型被网页或 MCP 哄骗，只要动作本身不满足最小权限约束，系统仍能在边界处将其拦下。

## Module 2 — 核心方法 (Core Method)

ClawGuard 是一个**工具调用边界上的运行时安全层**，由“会话前任务规则归纳 + 每次调用执行审计”两部分组成。

**（1）任务规则归纳（Task Rule Induction）**
在第一次外部工具调用前，仅根据系统提示与用户原始目标 H₀ 归纳任务级规则 R_task，例如允许访问的域名、文件目录和工具类别。关键点是：这一步发生在接触任何外部内容之前，因此不会被注入内容污染。

**（2）内容净化（Content Sanitizer）**
对工具输入参数和工具输出都做敏感信息脱敏，例如云密钥、私钥、token、数据库连接串等，减少敏感数据在工具链中的流动与二次污染。

**（3）规则评估器（Rule Evaluator）**
从一次调用中抽取 command / file / network 三类属性，分别与黑白名单比对，按“deny 优先，未知走审批，全部允许才执行”的策略做聚合判定。

**（4）Skill Inspector + Approval Mechanism**
首次使用某个 skill 前，先做结构化风险分析，再要求用户确认；对模糊调用则挂起等待人工批准，形成可审计的人在回路防线。

**关键设计差异**：ClawGuard 的核心不是更强的 prompt 分类，而是把防御点下沉到 **tool-call boundary**，把“安全”变成可执行、可记录、可复盘的系统机制。

## Module 3 — 技术细节 (Technical Details)

### Agent 形式化
设目标 Agent 为模型 𝓜 与工具集 𝓣 = {t₁, …, tₙ} 的组合。每一步动作为：

aₜ ∈ 𝓞 ∪ 𝓒_𝓣

其中工具调用集合为：

𝓒_𝓣 = {(tⱼ, q) | tⱼ ∈ 𝓣, q ∈ Σ*}

工具返回：

oⱼ = tⱼ(q)

并直接写回历史：

Hₜ₊₁ = Hₜ · (tool, oⱼ)

这正是 indirect prompt injection 的根源：外部输出未经验证便成为下一步推理依据。

### 规则集合
ClawGuard 维护：

𝓡 = 𝓡_base ∪ 𝓡_task

- 𝓡_base: 系统级硬约束（凭证、危险命令、外传通道等）
- 𝓡_task: 从用户目标自动归纳并经用户确认的任务级最小权限规则

### 单元素判定
对任意待检查元素 x：

V_elem(x) =
- ⊥, 若 x 命中黑名单
- ⊤, 若 x 命中白名单
- amb, 其它情况

若黑白名单同时命中，则黑名单优先。

### 多属性聚合
对一次调用 a* 的多个属性集合 {xᵢ}：

V(a*) =
- ⊥, 若 ∃xᵢ: V_elem(xᵢ)=⊥
- amb, 若不存在 ⊥ 且 ∃xᵢ: V_elem(xᵢ)=amb
- ⊤, 否则

即“最严格结论优先”。

### 内容脱敏
输入与输出分别做：

q* = 𝓢_in(q, 𝓟)  
o* = 𝓢_out(o, 𝓟)

其中 𝓟 是敏感模式库，覆盖 AWS/GCP/Azure key、JWT、SSH key、数据库连接串等。

### 复现级伪代码
```python
class ClawGuard:
    def authorize(self, action, task_rules, base_rules, patterns):
        action = sanitize_input(action, patterns)
        if is_new_skill(action):
            report = inspect_skill(action.skill)
            if not user_approve(report):
                return "deny", None

        verdict = evaluate_rules(action, task_rules + base_rules)
        if verdict == "deny":
            return "deny", None
        if verdict == "ambiguous" and not user_approve(action):
            return "deny", None

        result = execute_tool(action)
        result = sanitize_output(result, patterns)
        return "allow", result
```

## Module 4 — 实验设置与结果 (Experiments and Results)

### 数据集与评价指标
- **基准**: AgentDojo, SkillInject, MCPSafeBench
- **目标模型**: DeepSeek-V3.2, GLM-5, Kimi-K2.5, MiniMax-M2.5, Qwen3.5-397B-A17B
- **指标**:
  - ASR (↓): Attack Success Rate
  - DSR (↑): Defense Success Rate = RR + IRR
  - CR (↑): Completion Rate

### 主要结果表格

| Benchmark | 无防御 ASR | **ClawGuard ASR** | **ClawGuard DSR** | 关键信号 |
|-----------|------------|-------------------|-------------------|----------|
| AgentDojo | 0.6%–3.1% | **≈0.0%** | **99.4%–100%** | 对显式工具型攻击几乎清零 |
| SkillInject | 26.2%–47.6% | **4.8%–14.2%** | **71.4%–84.6%** | 对 skill 注入显著有效 |
| MCPSafeBench | 36.5%–44.5% | **7.1%–11.0%** | **74.9%–75.8%** | MCP 注入大幅压制 |

### 消融实验 (Ablation Study)

| 消融设置 | 结果 | 说明 |
|----------|------|------|
| 仅使用基础规则 R_base | 已有显著防御增益 | 当前论文主实验即采用此配置 |
| 去掉 skill 检查 | SkillInject 场景风险明显上升 | skill 描述本身是独立攻击面 |
| 对 ambiguous 调用直接拒绝 | 安全性更强但交互成本被低估 | 论文实验为便于统一评估未真实走审批流程 |

### 关键发现
- 对**动作型攻击**（文件访问、命令执行、越权联网）来说，在工具边界做规则执行比依赖模型“自己识别恶意文本”更稳健。
- SkillInject 与 MCPSafeBench 证明：只靠基础规则就能大幅降低 ASR，说明开放式 Agent 生态可通过运行时 reference monitor 获得显著收益。
- 但论文实验尚未真正量化完整版本的 `R_task` 任务规则归纳，因此当前结果更准确地说是“基础规则版 ClawGuard 很强”，而非完整版已被完全验证。

## Module 5 — 局限性分析 (Limitations)

1. **完整方法尚未被完整评测**: 当前实验仅使用 R_base，并未真正验证 context-aware task rule induction 的量化收益；论文最有新意的部分之一尚缺系统实证。
2. **对内容误导型攻击存在盲区**: 如果攻击主要体现在“模型生成了被误导的结论”，而不是触发危险工具调用，单纯的 tool-call boundary 机制不一定能拦住。
3. **规则归纳仍依赖 backbone LLM 质量**: 若模型生成的任务规则过宽松或过严格，都会分别带来漏拦截或误阻断问题；JSON 解析失败时还会回退到基础规则。
4. **人机交互成本未被充分评估**: Skill 首次审批、任务规则确认、ambiguous 调用确认都会增加部署负担，而论文实验为了简化直接把 ambiguous 视为拒绝，回避了真实交互延迟。

## Module 6 — 复现指南 (Reproduction Guide)

### 环境依赖 (Environment)
```bash
pip install fastapi pydantic rich typer
pip install openai anthropic  # 任一可驱动 Agent 的 LLM SDK
```

### 核心代码框架 (Core Code)
复现需要实现四类部件：
1. 规则归纳器：根据 H₀ 产出 network/file/command 三域规则
2. 内容净化器：支持常见密钥与连接串脱敏
3. 规则评估器：黑名单优先、多属性聚合
4. 审批与审计日志：记录 allow / deny / queue 决策

```python
base_rules = load_base_rules()
task_rules = induce_task_rules(user_goal)
clawguard = ClawGuard()
verdict, result = clawguard.authorize(
    action=agent_action,
    task_rules=task_rules,
    base_rules=base_rules,
    patterns=secret_patterns,
)
```

### 复现步骤 (Reproduction Steps)
1. 选择一个支持工具调用的 Agent 框架，接管其 tool dispatcher。
2. 在第一次工具调用前，只基于用户任务生成最小权限规则并让用户确认。
3. 为每个工具调用抽取命令、文件路径、网络目标并做黑白名单匹配。
4. 在调用前后做输入/输出脱敏，并记录审计日志。
5. 在 AgentDojo、SkillInject 或 MCP 场景上计算 ASR / DSR / CR，与无防御版本对比。

### 预期结果
- 对显式工具型攻击，ASR 应显著下降；在 AgentDojo 上接近清零。
- 对 skill / MCP 注入，ASR 通常可降到个位数到低双位数区间。
- 若补上任务规则归纳，复杂场景下的外联与目录越权应进一步减少。
