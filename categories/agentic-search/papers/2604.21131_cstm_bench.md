## 一、背景与问题

### 动机

现有 Agent guardrails 大多是 **session-bound** 的：每条消息单独判一次，上一轮、上一会话、上一代理做过什么，系统通常全忘了。但真正危险的攻击越来越不是“这一条消息就很恶意”，而是**把攻击分散到几十个 session 里，每条看起来都合理，只有串起来才危险**。

### 核心科学问题

能否构建一个专门评测 cross-session threats 的 benchmark，并比较不同检测架构在“跨 session 聚合威胁”上的能力边界，尤其是：
- 单 session 判断为什么会失效；
- full-log 长上下文相关器是否真的够用；
- bounded-memory reader 是否能在效果与实时成本之间找到更好的平衡。

### 直觉洞见

这篇论文最重要的判断不是“上下文窗口变大了”，而是：**安全检测的失败并不主要来自 token 不够，而来自相关器在冗长、稀释、被重写过的跨 session 轨迹里抓不住真正的高信号片段。**

---

## 二、核心方法

**（1）CSTM-Bench：跨会话威胁基准**

论文发布 **CSTM-Bench（Cross-Session Threat Memory Benchmark）**，把攻击按两条正交轴组织：
- **kill-chain stage**
- **cross-session operations ontology**：accumulate / compose / launder / inject_on_reader

并绑定到 **7 个 identity anchors**，使“是否违规”变成清晰的 policy predicate，而不是模糊主观判断。

**（2）三类 traffic 拆分**

数据被拆成：
- **Attack**
- **Benign-pristine**
- **Benign-hard**

这样 false alarm 不会被“纯干净样本”掩盖，能更真实地测出系统是否会把复杂但正常的历史模式误判成攻击。

**（3）两种 released shards**

- **dilution**：测信号稀释，攻击原则上仍可从单 session 里看出来；
- **cross_session**：用闭环 rewriter 把单 session 明显违规部分软化，使得每个 session 单独看都像正常工作，但跨 session 聚合后仍保留攻击结构。

这让 benchmark 能区分两类完全不同的失败：
1. per-session blind spots；
2. cross-session dilution。

**（4）三种 reader 架构对比**

- **Per-session judge**：每个 session 单独判；
- **Full-Log Correlator**：把所有 session 按时间拼成长日志，再一次性让 LLM 判；
- **Coreset Memory Reader**：维护容量受限的高信号 buffer，再把它喂给 correlator。

论文真正的创新点，不只是 benchmark，而是明确提出：**bounded-memory coreset reader 可能比 full log 更好**，因为 full log 虽然信息更多，但相关器会在长上下文里被稀释。

## 三、技术细节

### 复合指标设计

论文提出非补偿式综合指标：

CSTM = 0.7 · F1(CSDA@action, precision) + 0.3 · CSR_prefix

其中：
- **CSDA@action**：是否在 action-on-objective 落地前检测到攻击；
- **F1(CSDA@action, precision)**：衡量检测质量；
- **CSR_prefix**：ordered prefix stability，衡量 coreset buffer 的前缀稳定性，可作为 KV-cache 复用与实时 serving cost 的代理。

这点很关键：论文不是只盯 recall，而是把“能否线上跑得起”一起纳入目标。

### 数据与 taxonomy

- attack taxonomies：**26** 个
- 每个 shard 场景数：**54**
- 场景构成：**26 Attack + 14 Benign-pristine + 14 Benign-hard**
- 关键 cross-cutting tag：**inject_on_reader**

### 信息瓶颈视角

论文把跨 session 检测建模成：

message stream → reader bottleneck → correlator LLM → verdict

其中最重要的结构变量不是具体 ranker，而是：**有多少历史真的被允许进入 correlator**。作者认为这本质上是一个信息瓶颈问题，而不是简单“窗口更长就能看见更多”。

### reader 设置

- correlator 模型：**Claude Sonnet 4.6**
- coreset reader 默认容量：**K = 50**
- Full-Log Correlator：chronological full log，一次 batch 判定
- Coreset Memory Reader：同一 correlator，但输入改成 coreset-aware K=50 buffer

### 关键结构性风险：inject_on_reader

作者专门指出一个新攻击面：**inject_on_reader**。也就是说，单独看每个 session 都 benign，但把多条 benign-looking 历史拼起来之后，反而对 downstream correlator 自己形成 prompt injection。这不是传统“攻击 agent”，而是“攻击检测器”。

## 四、实验设置与结果

### 三个 reader、两个 shard 的 headline result（Table 15）

| Reader | Shard | CSDA@action | F1 | CSTM | FPR(pristine) | FPR(hard) |
|--------|-------|------------:|---:|-----:|--------------:|----------:|
| Per-session judge | dilution | **1.000** | 0.727 | 0.809 | 0.214 | 0.643 |
| Per-session judge | cross-session | **0.538** | 0.554 | 0.688 | 0.214 | 0.643 |
| Full-Log Correlator | dilution | 0.885 | 0.812 | **0.868** | **0.000** | 0.500 |
| Full-Log Correlator | cross-session | **0.500** | 0.600 | 0.720 | **0.000** | 0.500 |
| Coreset Memory Reader | dilution | **0.962** | **0.926** | 0.775 | **0.000** | **0.214** |
| Coreset Memory Reader | cross-session | **0.731** | **0.804** | **0.690** | **0.000** | **0.214** |

### 最重要的三条结论

**（1）Per-session judge 在 cross-session shard 上直接掉半边**

- dilution：**26/26** attack recall
- cross-session：**14/26** attack recall

即：攻击只要被 rewrite 到“单轮看起来像正常工作”，session-bound 检测基本就废了。

**（2）Full-Log Correlator 也并不稳**

- dilution：**23/26**
- cross-session：**13/26**

论文反复强调：这不是 context window 不够。因为所有场景都**远低于 1M token window**。也就是说，full log 的失败是**信号稀释 + 语义重写迁移失败**，不是简单 truncation。

**（3）K=50 Coreset Memory Reader 是唯一跨两个 shard 都稳住的 reader**

- dilution：**25/26 attack recall，F1 = 0.926**
- cross-session：**19/26 attack recall，F1 = 0.804**

它是唯一一个在 dilution 与 cross-session rewriting 下都还能维持较高 recall 的 reader。尽管 composite CSTM 因 CSR_prefix 记账方式未必始终最高，但在核心 efficacy 指标 F1 上，它是最强架构。

### “roughly half recall” 不是口号，而是核心负结果

论文最强的负面结论是：
- session-bound judge 从 26/26 掉到 14/26；
- Full-Log Correlator 从 23/26 掉到 13/26；

两种纯 LLM correlator 都在 cross-session rewriting 下**大约掉半数 recall**。这说明“把日志全拼起来再丢给大模型”并不是 cross-session defense 的终局。

### coreset 的额外优势

- 在 dilution shard 上恢复了 Full-Log Correlator 会漏掉的 aggregate-only 模式，如 **T05 Mosaic Attack** 和 **T10 Temporal Evasion**；
- Benign-hard FPR 从 **0.50 降到 0.214**；
- 在 cross-session shard 上还能独特恢复 **T21 / T24** 这类其它 LLM-only reader 都漏掉的场景。

### 关键发现

1. 现有安全检测的主要盲区是 **state resets**，不是单条 prompt 分类精度。  
2. Full-log correlation 不等于真正的 cross-session robustness。  
3. bounded-memory coreset 是当前最有前景的结构方向。  
4. 真实线上系统必须同时优化 detection quality 与 serving stability。

## 五、局限性分析

1. **数据规模仍是 first-order characterization**：每个 shard 54 个场景，适合刻画结构趋势，但不足以做细粒度模型排名。  
2. **目前主要用单一 correlator 家族评估**：核心 reader 对比基于 Claude 系列，跨 provider 泛化仍待更大规模验证。  
3. **reference admitter 只是一个起点**：K=50 不是理论最优，只是证明 bounded-memory 方向有效。  
4. **benchmark 强在检测侧，不直接覆盖完整防御闭环**：例如 lineage-preserving storage、tool provenance、multi-agent memory governance 仍需后续系统工作补上。

### 核心代码框架

```python
class CoresetReader:
    def __init__(self, correlator, capacity=50):
        self.correlator = correlator
        self.capacity = capacity
        self.buffer = []

    def update(self, message):
        score = ranking_score(message)
        self.buffer = admit_or_evict(self.buffer, message, score, k=self.capacity)
        self.buffer = stable_sort(self.buffer)

    def verdict(self):
        return self.correlator.inspect(self.buffer)
```

### 复现步骤

1. 从 Hugging Face 下载 `intrinsec-ai/cstm-bench`，分别读取 dilution 与 cross_session 两个 split。  
2. 实现三类 reader：per-session judge、full-log correlator、K=50 coreset reader。  
3. 固定 correlator 模型与 prompt，仅改变输入形态，避免把架构差异和模型差异混在一起。  
4. 统计 Attack recall、F1、Benign-pristine FPR、Benign-hard FPR 与 CSR_prefix。  
5. 复现 Table 15 的 headline matrix，并重点检查：cross_session 上两类 LLM-only reader 是否出现近半 recall drop。  
6. 若要走生产化方向，应继续优化 ranker 的 prefix stability，并把 provenance / lineage 信号接入 reader，而不是只让 correlator看纯文本历史。
