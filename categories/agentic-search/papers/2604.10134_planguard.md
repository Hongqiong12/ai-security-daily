## Module 1 — 背景与问题 (Background and Problem)

### 动机 (Motivation)

间接提示注入（Indirect Prompt Injection, IPI）对 Agent 的威胁比对普通聊天模型更大，因为 Agent 不只是“被诱导说错话”，而是会真的去调用工具、发请求、读写外部状态。现有防御很多停在 pre-processing：删关键词、过滤网页文本、扫描 prompt。这种做法的问题是，它根本没有验证模型**最终打算执行什么动作**。

PlanGuard 的切入点很准确：真正该防的不是“上下文里有没有恶意字符串”，而是“Agent 最终要执行的动作是否仍然忠于用户原始意图”。如果动作已经偏了，再干净的输入过滤都没意义。

### 核心科学问题 (Core Scientific Question)

能否在不修改底层大模型、也不依赖额外训练的前提下，通过规划隔离与运行时一致性验证，让 Agent 在面对 indirect prompt injection 时仍只执行与用户意图一致的工具调用？

### 直觉洞见 (Intuitive Insight)

PlanGuard 的思路像“先写授权清单，再核对现场执行”。它先让一个隔离的 Planner 只根据用户原始请求列出**合法动作集合**，然后对实际要执行的 tool call 做两层检查：先查“是不是授权工具”，再查“参数偏差是不是合理格式差异”。不是去猜网页里有没有恶意话术，而是直接问：**你现在这步动作，原始用户真的授权了吗？**

## Module 2 — 核心方法 (Core Method)

PlanGuard 是一个 training-free runtime defense，核心由三个组件组成。

**（1）上下文隔离 Planner（Isolated Planner）**
Planner 只读取**用户原始指令**，不读取外部检索结果、网页内容、邮件正文等不可信上下文。它输出一个 reference action set，作为该任务合法动作的上界。这样做的本质是把“合法计划”与“受污染上下文”解耦。

**（2）Stage I：硬约束校验（Hard Constraint Verification）**
当 Agent 准备调用工具时，PlanGuard 先检查该调用是否属于 Planner 授权集合。若工具名、动作类型或高层行为不在授权范围内，直接拦截。这个阶段追求的是 **zero-ASR security floor**，即宁可保守，也不能让未授权行为漏过去。

**（3）Stage II：Intent Verifier（意图一致性验证）**
仅靠 Stage I 会导致较高误报，因为很多合法调用只是在参数格式上与 planner 输出略有不同。因此作者加入 Intent Verifier，专门区分：
- 正常参数变体（benign formatting variance）
- 恶意 hijacking / parameter drift  

这样 PlanGuard 能在维持 0% ASR 的同时，把 FPR 压下来。

**（4）Context Isolation 的设计原则**
PlanGuard 最重要的不是某个 classifier，而是设计原则：**先隔离规划，再验证执行**。也就是把 Agent 的安全问题从“内容审计”改写成“计划-执行一致性审计”。

## Module 3 — 技术细节 (Technical Details)

### 形式化定义

设用户原始请求为 u，不可信外部上下文为 c_ext，Planner 仅基于 u 生成合法动作集合：

A_ref = Planner(u)

Agent 在实际环境中生成候选动作 a_exec。PlanGuard 的第一层校验为：

allow_stage1(a_exec) = 1[a_exec ∈ A_ref]

若 Stage I 通过，则进入 Stage II，验证参数与意图是否一致：

score_intent = Verifier(u, a_exec)
allow_stage2(a_exec) = 1[score_intent ≥ τ]

最终放行规则：

allow(a_exec) = allow_stage1(a_exec) ∧ allow_stage2(a_exec)

### 双阶段设计的意义

- **Stage I**：保证不会发生 planner 从未授权的工具调用；
- **Stage II**：在不牺牲安全底线的前提下，减少误报并保留正常功能。

### PlanGuard 伪代码

```python
class PlanGuard:
    def __init__(self, planner, intent_verifier, threshold=0.8):
        self.planner = planner
        self.intent_verifier = intent_verifier
        self.threshold = threshold

    def allowed_actions(self, user_instruction):
        return self.planner.plan(user_instruction)

    def stage1(self, tool_call, ref_actions):
        return tool_call.name in ref_actions

    def stage2(self, user_instruction, tool_call):
        score = self.intent_verifier.score(user_instruction, tool_call)
        return score >= self.threshold

    def guard(self, user_instruction, tool_call):
        ref_actions = self.allowed_actions(user_instruction)
        if not self.stage1(tool_call, ref_actions):
            return False
        if not self.stage2(user_instruction, tool_call):
            return False
        return True
```

### 关键设计点

| 组件 | 作用 | 论文信号 |
|---|---|---|
| Isolated Planner | 只依赖用户原始请求建立 reference set | 从根上切断外部污染对 planning 的影响 |
| Hard constraints | 确保 unauthorized tool call 直接被拒绝 | Stage I 即可做到 0% ASR |
| Intent Verifier | 区分 benign format drift 与真实 hijack | 负责把 FPR 从 27%/38% 拉低到 0.97%/3.28% |
| Training-free | 无需重新训练底模 | 方便嵌入已有 agent pipeline |

## Module 4 — 实验设置与结果 (Experiments and Results)

### 数据集与评价指标

| 设置 | 内容 | 指标 |
|---|---|---|
| InjecAgent benchmark | 评估间接提示注入攻击 | ASR ↓, FPR ↓ |
| Direct Harm (DH) subset | 诱导 agent 执行危险动作 | ASR ↓, FPR ↓ |
| Data Stealing (DS) subset | 诱导 agent 泄露敏感信息 | ASR ↓, FPR ↓ |

### 主要结果表格

| 方法 | ASR | FPR | 说明 |
|---|---|---|---|
| Vanilla Agent | **72.8%** | — | 论文摘要给出的总体基线 |
| Stage I Only | **0.0%** | DH: **27.00%**, DS: **38.01%** | 安全足够，但误报不可接受 |
| PlanGuard (Stage I + II) | **0.0%** | DH: **0.97%**, DS: **3.28%** | 在保持零攻击成功的同时大幅降低误报 |
| 摘要整体结论 | **72.8% → 0%** | **1.49% 平均可接受误报率** | training-free 且 model-agnostic |

### 细粒度结果解读

论文指出，Vanilla Agent 在两个子集上的脆弱性并不一致：
- DH 子集基线 ASR 为 **56.90%**；
- DS 子集基线 ASR 为 **88.67%**；  
说明“数据窃取型 IPI”比“直接危险动作型 IPI”更容易稳定触发。

而 PlanGuard 的强点在于：
- **Stage I 单独就能把 ASR 清零**，说明“动作是否被授权”这个约束非常强；
- 但 Stage I 误报太高，不适合实际部署；
- 加上 Intent Verifier 后，安全性不降，FPR 却下降到 **0.97% / 3.28%**，这才具备工程可用性。

### 消融实验 (Ablation Study)

| 消融设置 | 结果 | 说明 |
|---|---|---|
| 去掉 Stage II，只保留 Stage I | ASR 仍为 0，但 FPR 很高 | 说明仅靠硬约束会过度保守 |
| 不用 Isolated Planner，直接信任上下文 | 易被 IPI 劫持 | 说明 planning 必须与外部内容解耦 |
| 不做层级验证，仅做单条规则过滤 | 对 DS 类攻击更脆弱 | 单层规则无法处理参数级偏移 |

### 关键发现

- **Agent 安全的关键不只是过滤输入，而是验证“计划-执行一致性”。**  
- **Context Isolation 非常有效**：把 planner 从外部内容中隔离出来，能直接把未授权工具调用压到 0。  
- **两阶段设计是必要的**：Stage I 保证安全下限，Stage II 把系统从“能防”变成“能用”。

## Module 5 — 局限性分析 (Limitations)

1. **Planner 质量决定上限**：如果 Isolated Planner 自己对用户意图理解错了，后续所有校验都建立在错误 reference set 上。  
2. **更适合显式 tool-use agent**：PlanGuard 对动作显式、参数结构化的 agent 最自然；对于高度自由的 long-form planning agent，action normalization 会更难。  
3. **主要验证的是单轮 runtime consistency**：对多轮长轨迹中逐步累积的 subtle hijacking，当前实验仍偏有限。  
4. **依赖额外 verifier 组件**：虽然不训练底模，但系统复杂度增加，需要额外部署 Planner 和 Intent Verifier。  
5. **对 planner 输出粒度敏感**：参考动作集合太粗会漏风险，太细则 FPR 又会上升，工程上需要校准。

### 核心代码框架 (Core Code)

```python
class IsolatedPlanner:
    def plan(self, user_instruction: str) -> set[str]:
        # 只读取 user instruction，不读取网页/邮件/工具返回
        return infer_authorized_actions(user_instruction)


class IntentVerifier:
    def score(self, user_instruction: str, tool_call: dict) -> float:
        return estimate_consistency(user_instruction, tool_call)


class RuntimePlanGuard:
    def __init__(self, planner, verifier, tau=0.8):
        self.planner = planner
        self.verifier = verifier
        self.tau = tau

    def allow(self, user_instruction, tool_call):
        ref = self.planner.plan(user_instruction)
        if tool_call["name"] not in ref:
            return False
        return self.verifier.score(user_instruction, tool_call) >= self.tau
```

### 复现步骤 (Reproduction Steps)

1. 选择一个可调用工具的 agent 框架，并确保能记录每一步 tool call。  
2. 在不可信外部内容进入执行前，单独建立只读 user instruction 的 Planner。  
3. 为每次 tool call 生成 reference action set，并实现 Stage I 名称/动作级硬约束。  
4. 再实现参数级 Intent Verifier，对参数偏差给出一致性分数。  
5. 在 InjecAgent 风格 benchmark 上分别测试 DH / DS 子集，统计 ASR 和 FPR。  
6. 重点对比 Vanilla、Stage I Only、PlanGuard Full 三个设置，验证“0% ASR + 低 FPR”是否同时成立。  
7. 若部署到生产环境，建议把 planner 输出与 verifier 结果一并打日志，便于后续失败样本分析。
