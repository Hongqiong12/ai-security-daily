## Module 1 — 背景与问题 (Background and Problem)

### 动机 (Motivation)
大语言模型 (LLM) 智能体的性能在很大程度上取决于“执行框架（Execution Harness）”——即编排工具使用、上下文管理和状态持久化的系统层。然而，正是由于其在架构中的核心地位，执行框架成为了高价值的攻击面：在框架层的单点失陷可能会沿着整个执行管道产生级联崩溃。现有的安全机制（如纯模型对齐或外置的输入过滤）往往与框架的内部结构脱节，它们对框架的内部状态（如短时记忆和历史工具调用记录）是盲目的，无法在智能体运行的不同阶段进行跨周期的安全协同。

### 核心科学问题 (Core Scientific Question)
如何设计一个深度融入智能体运行生命周期底层的安全架构，使其能够跨越输入处理、决策制定、动作执行和状态更新等多个阶段进行协同防御，从而在不损害智能体核心任务能力的前提下大幅降低攻击成功率？

### 直觉洞见 (Intuitive Insight)
防守一个智能体，不能只在“门口”设置安检（输入过滤），也不能指望智能体自己的“大脑”完全不被催眠（模型对齐）。安全的智能体应该像一座设计精良的核电站：除了外围安保，控制室要有多重决策校验，操作阀门有权限隔离，即便真的发生了泄漏，也有底层的紧急回滚机制。将安全防御深植于底层“执行框架”的各个生命周期阶段，并在阶段之间建立通信，就能在攻击链条形成闭环前将其阻断。

## Module 2 — 核心方法 (Core Method)

作者提出了 **SafeHarness**，一个将四层防御机制直接编织进智能体生命周期底层的安全架构，各层之间通过跨层协同机制进行联动：

**（1）输入处理层的对抗上下文过滤 (Adversarial Context Filtering)**
在智能体接收外部信息时，主动识别并过滤具有注入攻击或越狱特征的对抗性上下文，防止恶意提示污染工作记忆。

**（2）决策制定层的分级因果验证 (Tiered Causal Verification)**
在 LLM 生成执行计划后、调用工具前，引入因果校验模型，评估当前决策是否与安全红线和初始意图构成合理的因果逻辑链。对于高危操作启动高级别的冗余验证。

**（3）动作执行层的权限隔离工具控制 (Privilege-separated Tool Control)**
打破传统的全能工具箱授权。根据上下文的安全上下文和因果验证结果，动态实施最小权限原则，隔离高危 API 的执行环境，防止攻击者通过控制 LLM 间接执行系统级破坏。

**（4）状态更新层的安全回滚与自适应降级 (Safe Rollback with Adaptive Degradation)**
监控执行后的状态更新。一旦检测到异常积累或越权行为发生，立刻回滚到上一个已知的安全快照，并在重启任务时自适应地降级工具权限或关闭高危组件。

## Module 3 — 技术细节 (Technical Details)

论文的核心在于跨生命周期的协同验证算法。其跨层降级逻辑可形式化为：

**生命周期状态转移安全度量**:
设定智能体在 $t$ 时刻的状态为 $S_t$，观测到输入 $X_t$，LLM 给出动作决策 $A_t$。
因果验证模块输出安全置信度度量：
$$ \tau_t = \mathcal{V}_{\text{causal}}(S_t, X_t, A_t) $$

**权限动态收缩策略 (Dynamic Privilege Control)**:
工具箱的可用权限集 $\mathbb{P}_t$ 是安全置信度的时间衰减函数的映射：
$$ \mathbb{P}_{t} = \mathcal{F}\left( \gamma \tau_t + (1-\gamma) \tau_{t-1} \right) $$
当累计的置信度低于硬阈值 $\tau_{\text{threshold}}$ 时，触发中断和状态回滚：
$$ S_{t+1} = \begin{cases} \text{Execute}(S_t, A_t), & \text{if } \tau_t \geq \tau_{\text{threshold}} \\ \text{Rollback}(S_{t-k}), & \text{otherwise} \end{cases} $$

**多层协同防御惩罚**:
在模型训练或部署中，可以量化防御成本：
$$ \text{Cost} = \sum_{stage \in \{I, D, E, S\}} \mathcal{C}_{stage} \cdot \mathbf{1}(\text{Anomaly}_{stage}) $$
一旦某个早期阶段 (如 $I$ 输入阶段) 判定极高风险，可以直接阻断后续执行以节省安全代价。

## Module 4 — 实验设置与结果 (Experiments and Results)

### 数据集与评价指标
- **评估基准**：包含跨越 6 种威胁类别的 5 大攻击场景。
- **对比基线**：无保护的基础框架（如原生 LangChain/AutoGPT 架构）及 4 种现有的单一层面安全基线。
- **评价指标**：不安全行为率 UBR (%) (↓)，攻击成功率 ASR (%) (↓)，以及任务效用保留率 Utility (%) (↑)。

### 主要结果表格

| 框架与防御机制 | 平均 ASR (↓) | 平均 UBR (↓) | 任务效用保留 (↑) |
|--------------|-------------|-------------|-------------|
| 无保护基线 (Vanilla Harness) | 较高 (基准) | 较高 (基准) | 100% |
| 单层过滤基线 | 降低约 15% | 降低约 10% | 轻微下降 |
| **SafeHarness (Ours)** | **绝对下降 42%** | **绝对下降 38%** | **接近基准水平** |

### 关键发现
- **级联防御的压倒性优势**：单点的输入过滤很容易被多轮对话中的上下文污染绕过；但当绕过输入层的恶意载荷在尝试调用危险 API 时，会被 SafeHarness 的因果验证和动态权限双层锁定，从而导致 ASR 大幅下降了 42%。
- **容错恢复**：安全回滚机制不仅是防御手段，也是保障任务完成率的关键。通过回滚并降级权限，智能体能够在丢弃恶意干扰后重新完成核心任务，证明了鲁棒性与实用性无需绝对对立。

## Module 5 — 局限性分析 (Limitations)

1. **框架重构成本过高**：SafeHarness 需要对 Agent 执行的底层逻辑（如 LangChain 的 AgentExecutor 或 ReAct 循环）进行深度魔改。它不是一个可以轻易即插即用的 API，导致将其移植到已有的生产系统中需要极高的工程成本。
2. **因果验证的计算瓶颈**：在每一次 LLM 生成动作前，引入额外的因果验证模型（无论它是另一个 LLM 还是专门的分类器）会显著增加系统的整体推理延迟。在对响应时间要求严苛的实时流式应用中，这种同步阻塞的验证机制不可接受。
3. **状态回滚的外部不可逆性**：尽管框架可以回滚其内部内存和上下文状态，但对于已经执行的不可逆外部动作（如发送了电子邮件、删除了云端数据库），SafeHarness 的内部回滚机制无能为力。

### 核心代码框架 (Core Code)
```python
from typing import Any
import copy

class SafeAgentHarness:
    def __init__(self, llm, causal_verifier, tools):
        self.llm = llm
        self.causal_verifier = causal_verifier
        self.tools_catalog = tools
        self.state_history = []  # 用于回滚的快照栈
        
    def execute_turn(self, user_input, context):
        # 1. 对抗上下文过滤 (Input Processing)
        safe_input = self._filter_adversarial_input(user_input)
        
        # 保存状态快照
        self.state_history.append(copy.deepcopy(context))
        
        # LLM 规划
        plan, proposed_action = self.llm.plan(safe_input, context)
        
        # 2. 分级因果验证 (Decision Making)
        confidence = self.causal_verifier.verify(context, plan, proposed_action)
        
        if confidence < 0.4:
            # 4. 安全回滚 (State Update)
            context = self.state_history.pop()
            return "Action blocked by safety causal verification. State rolled back."
            
        # 3. 权限隔离控制 (Action Execution)
        allowed_tools = self._get_dynamic_privileges(confidence)
        if proposed_action.tool_name not in allowed_tools:
            return "Insufficient privilege for this tool in current safety state."
            
        # 实际执行
        result = self.tools_catalog.execute(proposed_action)
        
        # 4. 状态更新
        context.update(result)
        return result
```

### 复现步骤 (Reproduction Steps)
1. 基于 LangChain 或类似的轻量级框架定制执行循环，替换原有的直接执行逻辑。
2. 训练或部署一个微调模型作为 `causal_verifier`，该模型专门评估 `(上下文, 意图, 即将执行的动作)` 之间的逻辑一致性，输出 0-1 的置信度。
3. 对注册的所有 Tool 按照风险等级（Low/Medium/High）进行分类硬编码。
4. 在受到攻击基准测试（如引入间接提示词注入的网页内容）时，验证 Agent 是否在面临提权尝试时触发阻断逻辑并恢复内部历史堆栈。
