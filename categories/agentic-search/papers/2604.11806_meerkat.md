## Module 1 — 背景与问题 (Background and Problem)

### 动机 (Motivation)
随着 Computer Use Agent、coding agent 与自动化工作流代理快速落地，安全失败越来越不表现为单条明显违规轨迹，而是以“稀疏、分散、跨多条轨迹共同构成证据”的方式出现。现有 per-trace monitor 只能逐条打分，既抓不住多轨迹组合违规，也很难在数百到数千条轨迹中定位少量高风险样本。作者因此把问题从“单轨迹分类”升级为“在轨迹库中搜索违反安全属性的 witness 集合”。

### 核心科学问题
能否在**大规模 agent traces 仓库**中，针对自然语言定义的安全属性，自动发现那些只有跨多条轨迹联合观察后才显现的安全违规证据，并同时给出仓库级结论与轨迹级定位？

### 直觉洞见 (Intuitive Insight)
Meerkat 的直觉类似安全审计员在翻日志：不是把每条日志都单独判生死，而是先把相似行为聚到一起，再围绕“哪些簇值得深挖”进行定向追查。真正危险的不是某一条日志本身，而是一组看似普通的日志在组合后暴露出一致的作弊、滥用或 reward hacking 模式。

## Module 2 — 核心方法 (Core Method)

Meerkat 是一个**聚类增强的 agentic auditing 框架**，把 repository-level 安全审计建模为 witness search。

**（1）轨迹表示与聚类（Representation + Clustering）**
先将每条 trace 编码为 embedding，再用 k-means 等方法构造语义簇与层级索引。这样 agent 在后续审计时不必暴力扫描所有轨迹，而是优先进入高可疑簇，提升稀疏失败定位效率。

**（2）属性驱动搜索（Property-Guided Agentic Search）**
系统输入不是固定标签，而是自然语言安全属性 φ，例如“是否存在 developer cheating”或“是否存在跨任务 reward hacking”。审计 agent 结合属性说明、聚类结构与代表样本进行多轮检索、比对、取证，输出仓库级风险结论。

**（3）Witness 级定位（Witness Set Localization）**
Meerkat 不只给一个 pass/fail，而是同时输出候选 witness 集、每条 trace 属于 witness 的分数，以及仓库级违规分数。这样它既能用于大规模 triage，也能支持人工复核和 benchmark 复盘。

**关键设计差异**：与固定模板 monitor 不同，Meerkat 面向的是 hyperproperty 场景——安全性依赖于“多条轨迹的集合关系”，而不是某条轨迹自身的局部文本异常。

## Module 3 — 技术细节 (Technical Details)

### 问题形式化
给定轨迹库 R = {tᵢ}ᵢ₌₁ⁿ 和自然语言安全属性 φ，目标不是判断单条轨迹，而是寻找违反 φ 的 witness 集 W ⊂ R。

主流程可写为：

```text
1. {eᵢ} ← Represent(R)
2. C ← Cluster({eᵢ})
3. (p, E) ← BuildInputs(φ, R, C)
4. (v, {sᵢ}, Ŵ) ← Agent(p, E)
5. y ← Verdict(v)
```

其中：
- eᵢ: 第 i 条轨迹的语义表示
- C: 轨迹聚类结果
- v: repository-level 风险分数
- sᵢ: 第 i 条轨迹属于某个违规 witness 的概率性分数
- Ŵ: 候选 witness 集合

### Hyperproperty 视角
普通 trace property 关注单条轨迹是否满足条件；Meerkat 关注的是：

Σ: 事件空间  
𝒯 = Σ*：轨迹空间  
ℋ ⊂ 𝒫(𝒯)：hyperproperty

只有当一组轨迹共同构成违规证据时，审计才算成功。这个设定解释了为什么 per-trace monitor 在 distributed misuse 场景中会系统失效。

### Bayesian / Cluster 增强思路
论文附录给出的聚类增强评分可概括为：

p_clust(c) = 0.25·f_size(c) + 0.25·f_coh(c) + 0.25·f_conc(c) + 0.15·f_max(c) + 0.10·f_mean(c)

再与轨迹级基础分数 pᵢ 融合：

p̂ᵢ = 1 - (1 - pᵢ)(1 - p_clust(c))

其含义是：簇内若同时表现出“规模适中、语义集中、风险分布集中”，则该簇整体更值得深入搜索。

### 复现级伪代码
```python
class MeerkatAuditor:
    def __init__(self, embedder, llm_agent, n_clusters=32):
        self.embedder = embedder
        self.llm_agent = llm_agent
        self.n_clusters = n_clusters

    def audit(self, traces, safety_property):
        embs = self.embedder.encode(traces)
        clusters = KMeans(self.n_clusters).fit_predict(embs)
        env = build_cluster_index(traces, embs, clusters)
        repo_score, trace_scores, witnesses = self.llm_agent.search(
            property_text=safety_property,
            env=env,
        )
        verdict = "fail" if repo_score > 0.5 else "inconclusive"
        return verdict, repo_score, trace_scores, witnesses
```

## Module 4 — 实验设置与结果 (Experiments and Results)

### 数据集与评价指标
- **评测语料**: Cyber Misuse, Bio Misuse, ImpossibleBench, TRACE, CUA-SHADE-Arena, Terminal-Bench 2.0, HAL USACO, CyBench, BountyBench
- **任务特征**: 分布式滥用、reward hacking、developer cheating、covert sabotage
- **指标**:
  - Trace-level AP (↑): 轨迹级定位精度
  - Case-level AP / AUROC (↑): 仓库级风险判定能力
  - Success rate under increased test-time compute (↑)

### 主要结果表格

| 场景 | 基线 Monitor | Naive Agent | Bayesian / Buffer | **Meerkat** |
|------|--------------|-------------|-------------------|-------------|
| DM-Cyber, bg=20x, Qwen（trace AP） | 0.125 | 0.632 | 0.243 / 0.191 | **0.837** |
| DM-Cyber, bg=100x, GPT-5.4-mini（trace AP） | — | 0.136 | 0.053 / 0.021 | **0.297** |
| DM-Bio, bg=100x, Qwen（trace AP） | 0.012 | — | 0.040 / 0.026 | **0.142** |
| CUA-SHADE-Arena（AP） | 0.84 | — | 0.84 | **0.86** |

### 消融实验 (Ablation Study)

| 消融设置 | 结果变化 | 说明 |
|----------|----------|------|
| 去掉聚类索引 | 明显下降 | 稀疏违规难以从大规模 benign 背景中分离 |
| 仅用 per-trace monitor | 在 distributed misuse 上接近失效 | 只能看单轨迹局部异常，抓不到 witness 组合 |
| 增加 test-time compute | pass@1 从 0.10 升到 **0.50** | 13× 推理 token 带来约 5× 成功率提升 |

### 关键发现
- Meerkat 在**分布式 misuse / 多轨迹组合证据**场景的提升最明显，说明 repository-level auditing 是独立问题，不是 per-trace monitor 的简单放大版。
- 在 Terminal-Bench 2.0、HAL USACO、CyBench 等真实 benchmark 中，系统找到了大量 **developer cheating / reward hacking** 案例，具有直接 benchmark 清洗价值。
- 但在 TRACE 这类局部迹象已较明显的任务上，Meerkat 只是与强 baseline 持平或略优，并非普适碾压。

## Module 5 — 局限性分析 (Limitations)

1. **对 embedding / clustering 依赖强**: 若攻击者能故意把同一 campaign 的轨迹在表示空间中打散，Meerkat 会明显退化。论文中的 adaptive prefix 攻击就把 AP 从 0.10 压到 0.01。
2. **并非所有任务都显著优于基线**: 在 TRACE 和部分 GPT-5.4-mini 设定下，简单的 per-trace monitor 或 Bayesian 聚合有时可与 Meerkat 持平甚至略强。
3. **真实审计证据仍有边界**: Terminal-Bench 等真实语料并没有完整 ground-truth cheating 标签，一部分结论来自人工复核和 counterfactual 重跑，因此更适合视作高可信发现，而不是绝对 prevalence 估计。
4. **隐私与监控风险**: 跨轨迹关联分析会放大日志审计能力，但也可能提高对用户行为的去匿名化风险，需要额外的访问控制与数据最小化策略。

## Module 6 — 复现指南 (Reproduction Guide)

### 环境依赖 (Environment)
```bash
pip install scikit-learn sentence-transformers numpy pandas
pip install openai anthropic  # 或任意可驱动审计 agent 的 LLM SDK
```

### 核心代码框架 (Core Code)
核心复现部件包括：
1. 轨迹文本化与分段表示器（trace -> embedding）
2. k-means / hierarchical clustering 索引构建
3. 自然语言属性驱动的审计 agent
4. repository-level verdict 与 witness 输出模块

```python
from sentence_transformers import SentenceTransformer
from sklearn.cluster import KMeans

embedder = SentenceTransformer("BAAI/bge-base-en-v1.5")
meerkat = MeerkatAuditor(embedder=embedder, llm_agent=YourAuditAgent())
verdict, repo_score, trace_scores, witnesses = meerkat.audit(
    traces=trace_repo,
    safety_property="Find developer cheating or reward hacking patterns across traces.",
)
```

### 复现步骤 (Reproduction Steps)
1. 准备一个含数十到数百条 agent traces 的仓库，保留工具调用、系统输出与关键中间状态。
2. 用 sentence embedding 或其他表征模型将每条 trace 编码，并构建聚类索引。
3. 用自然语言定义审计属性（如“是否存在跨轨迹 reward hacking”）。
4. 让审计 agent 先浏览簇摘要，再按可疑簇深入检索代表轨迹并生成候选 witness。
5. 计算 trace-level AP、case-level AP/AUROC，与 per-trace monitor / Bayesian baseline 对比。

### 预期结果
- 在 distributed misuse 场景下，Meerkat 应显著优于 per-trace monitor。
- 在复杂真实 benchmark 上，应能发现少量但高价值的 developer cheating / reward hacking 轨迹簇。
- 若扩大 test-time compute，成功率会进一步提升，但代价是更高审计成本。
