## 一、背景与问题

### 动机
Deep Research Agent 正在从问答工具变成真正的长程研究协作者：它们要处理多模态用户文件、跨网页证据、长篇结构化写作和引用对齐。但当前评测要么依赖实时网页，难以复现；要么过于离线、过于干净，缺少真实研究场景里的噪声、误导文档与多文件证据链，导致 benchmark 与真实部署脱节。

### 核心科学问题
如何构造一个既真实、又可控、还能稳定复现的 deep research benchmark，让模型必须联合用户文件与外部证据完成长篇研究报告，同时保证答案可验证、评估可重复？

### 直觉洞见
论文的关键直觉是：真正要保留的是 open-web 研究任务的复杂性，而不是 live web 自身的不稳定性。因此作者为每个任务单独构建静态 research sandbox，把 supportive pages、distractor pages、noise pages 一起放进去，再通过 reverse construction 从证据反向生成 query，用“静态但复杂”的方式逼近真实研究环境。

---

## 二、核心方法

**（1）DR3-Eval benchmark**
数据层由三部分构成：真实用户文件、按任务独立构建的静态 sandbox corpus，以及根据证据链反向生成的 query。这样既能保留研究任务的复杂性，又能确保 benchmark 在不同时间重复运行时仍然稳定。

**（2）五阶段数据构造**
从真实需求 grounding 出发，先蒸馏搜索路径，再构建 sandbox，随后反向生成 query，最后做严格 QC。关键不是“从网上随便抓点材料”，而是把可验证性嵌入任务生成流程本身。

**（3）DR3-Agent**
作者实现了一个配套 agent：主代理负责全局计划与长报告生成，RAG Search Sub-agent 负责在 sandbox 中做多轮检索，File Reader Sub-agent 负责读取长文本用户文件。它不是简单 RAG，而是明确的 Plan–Act–Observe 多代理流程。

**（4）多维评测体系**
指标分成两大块：
- Information Seeking：IR_UF、IR_SC、CC
- Report Generation：FA、IF、DQ

这让 benchmark 不只问“最终写得像不像”，而是同时问“证据找全了吗、必要引用到位了吗、事实对了吗、有没有真的按要求做”。

## 三、技术细节

### 数据构造流程
1. 收集真实用户提供的研究材料，先做自动 PII 清洗，再人工复核。
2. 用 Gemini-2.5-Pro 生成候选关键词，并把关键词分为 signal 与 noise 两类。
3. 每个关键词最多抓取 100 条结果，构建 supportive / distractor / noise 三类网页，形成静态 sandbox。
4. 根据已知证据链反向构造 query，确保任务需要联合 user files 与 sandbox 才能解出。
5. 用四条 QC 准则筛选：Implicit Guidance、Synthesis Necessity、Insight Novelty、Interpretative Unambiguity。

### 数据集统计
- 最终任务数：100
- 语言：50 英文 + 50 中文
- 领域：Technology、Economy、Humanities，共 13 个子领域
- 68% 任务是多模态
- 平均每题 2.24 个用户文件，最多 6 个
- 512k 设置下，平均每题约 465.5 个网页

### 上下文规模
benchmark 提供 32k、64k、128k、256k、512k 五个 sandbox 规模，用来测试随着噪声与长上下文增长，模型是否还能维持检索质量与 grounded writing。

### 指标定义
- IR_UF：用户文件 insight 覆盖率
- IR_SC：sandbox 证据 insight 覆盖率
- CC：必要引用文档覆盖率
- FA：事实准确性
- IF：指令遵循度
- DQ：分析深度与逻辑质量

### 代理配置
- 主代理最大 10 turns
- RAG 子代理最大 5 turns
- 文件读取子代理最大 3 turns
- 默认 embedding：OpenAI text-embedding-3-small
- 文本 judge：GPT-5.1，temperature = 0
- 多模态辅助 judge：Gemini-2.5-Pro

## 四、实验设置与结果

### 对比模型
- Claude Sonnet 4
- GLM-4.7 / GLM-4.6
- Gemini-2.5-Pro
- GPT-4.1
- Qwen3-235B-A22B / 32B / 30B-A3B

### 主要结果表格（Avg. 总分）

| 模型 | Avg.@64k | Avg.@128k | Avg.@512k |
|------|---------:|----------:|----------:|
| **Claude Sonnet 4** | **70.7** | **67.5** | **65.6** |
| **GLM-4.7** | **69.8** | **66.9** | **64.1** |
| GLM-4.6 | 66.8 | 64.5 | 62.3 |
| Gemini-2.5-Pro | 61.5 | 60.6 | 57.0 |
| GPT-4.1 | 51.9 | 51.3 | 50.9 |
| Qwen3-235B-A22B | 51.1 | 49.6 | 48.7 |

### 关键数字
- 最强模型 Claude Sonnet 4 在 64k 也只有 70.7，说明 DR3-Eval 不是“换个 prompt 就能刷高分”的软 benchmark。
- Claude Sonnet 4 随上下文规模从 64k 增长到 512k，平均分从 70.7 降到 65.6；GLM-4.7 从 69.8 降到 64.1，说明长上下文噪声仍显著侵蚀 deep research agent 的能力。
- 以 Claude Sonnet 4 @64k 为例：IR_UF = 58.8，IR_SC = 55.3，CC = 64.7，但 FA = 87.0、IF = 87.4，说明高分模型并非“不会写”，而是“找全证据并引用到位”仍是主要瓶颈。
- 自动评分与人工评分的一致性较高：Pearson r = 0.78，Pairwise Agreement = 0.89，说明该自动评测管线有较强可用性。

### 关键发现
- 当前 frontier model 在 deep research 任务上的最大短板依然是 information seeking，而不是 surface-level instruction following。
- 只会把报告写得像模像样，并不等于 grounded；GPT-4.1 就表现出 IF 很高但 FA 较低的“形式完成、证据不足”特征。
- 静态 sandbox 与 live web 的整体结果差异很小，说明作者成功做到了“牺牲少量开放性，换来大量可复现性”。

## 五、局限性分析

1. **任务规模仍有限**：100 个任务对高质量长报告 benchmark 已经不小，但对真实 deep research 的长尾场景仍远远不够。
2. **语言范围有限**：当前只覆盖中文与英文，多语种研究工作流的复杂性尚未进入评测。
3. **sandbox 不是 live web 的完全替身**：虽然结果接近，但动态网页变化、搜索排序抖动、失效链接、交互式页面等因素仍被抽离掉了。
4. **评测依赖专有 API**：数据构造与自动评审依赖 GPT-5.1、Gemini-2.5-Pro、Gemini-2.5-Flash 等服务，复现成本和版本漂移风险都不低。

## 六、复现指南

### 环境依赖
```bash
git clone https://github.com/NJU-LINK/DR3-Eval
pip install -r requirements.txt
```

### 核心代码框架
```python
class DeepResearchEvaluator:
    def run_task(self, query, user_files, sandbox):
        evidence = self.retrieve(query, sandbox)
        file_notes = self.read_files(user_files)
        report = self.write_report(query, evidence, file_notes)
        return self.evaluate(report, evidence, user_files)
```

### 复现步骤
1. 克隆项目并下载 benchmark 数据，包括 100 个任务、用户文件、任务级 sandbox corpus 与评测元数据。
2. 搭建 Python 环境，配置 OpenAI / Gemini 等 API key，并建立每个任务的 sandbox 检索索引。
3. 按论文默认配置运行 DR3-Agent：主代理 10 turns，RAG 子代理 5 turns，文件阅读子代理 3 turns。
4. 对 64k、128k、512k 等上下文规模分别跑评测，计算 IR_UF、IR_SC、CC、FA、IF、DQ。
5. 重点核对 Claude Sonnet 4 与 GLM-4.7 在 64k / 128k / 512k 下的 Avg. 分数，以及高 IF 但不一定高 FA 的模型失衡现象。

### 预期结果
- Claude Sonnet 4 与 GLM-4.7 应位于第一梯队，但在 512k 下也无法接近满分
- IR_UF / IR_SC / CC 会明显低于 FA / IF，反映 evidence retrieval 才是真瓶颈
- sandbox-vs-live-web 对比不应出现大幅排名翻转
