# TraceSafe: A Systematic Assessment of LLM Guardrails on Multi-Step Tool-Calling Trajectories

**ArXiv ID**: 2604.07223v1
**Domain**: Agentic Search / Tool-calling Security / Guardrails

## 1. 背景与动机 (Background & Motivation)
随着基于大语言模型的智能体（LLM Agents）在自动化执行任务和调用外部工具（Tool-calling）方面的广泛部署，其安全风险从“单轮对话”扩展到了“多步轨迹（Multi-step Trajectories）”。现有的安全护栏（Guardrails）大多是为单次 Prompt-Response 设计的，难以应对恶意请求在多步工具调用中被拆解、伪装和分片执行的复杂情况。TraceSafe 提出了首个针对大模型智能体多步工具调用轨迹的系统性安全评估框架。

## 2. 核心方法 (Methodology)
TraceSafe 构建了一套多步攻击范式，并提出了针对轨迹的安全评估指标：
1. **攻击策略解构 (Trajectory Poisoning)**：
   - **目标分解**：将一个有害目标（如“窃取敏感文件”）拆分为多个看似无害的工具调用步骤（如：1. 列出目录；2. 读取缓存；3. 压缩文件；4. 发送网络请求）。
   - **延迟触发**：在前几步建立合法性掩护，直到最后一步才组装恶意载荷。
2. **基准测试集 (TraceSafe-Bench)**：包含 500+ 个涉及系统管理、网络请求、数据库操作的多步恶意轨迹。
3. **护栏评估维度**：从单步拦截率（Step-level Interception）、轨迹级上下文感知（Trajectory-level Context Awareness）以及误报率（False Positive Rate on Normal Tools）进行量化评估。

## 3. 技术细节 (Technical Details)
- **多状态马尔可夫决策过程 (MDP)**：将智能体的执行建模为 MDP，攻击目标定义为到达特定的非安全状态 $S_{unsafe}$。
- **四种攻击变体**：
  - *Direct Chain*：直接的连续高风险操作。
  - *Obfuscated Chain*：利用 base64 编码或脚本混淆隐藏工具参数。
  - *Data Exfiltration*：合法的读取 + 非法的外部发送。
  - *Privilege Escalation*：利用已有工具漏洞横向越权。
- **评估护栏对象**：测试了 Llama-Guard-3、Nemo-Guardrails 以及基于 GPT-4 的启发式审查器。

## 4. 量化实验 (Quantitative Experiments)
- **单步防御的失效**：面对被拆解的多步攻击（Obfuscated Chain），传统的单次 Prompt 护栏的拦截率暴跌至 18.5%，因为单一工具调用（如 `execute("ls -la")`）本身不具备明显的恶意特征。
- **上下文感知的优势**：引入轨迹历史（历史 Action 和 Observation）作为判断依据的 TraceSafe-Evaluator，其拦截率提升至 82.4%。
- **不可恢复的灾难**：实验表明，一旦智能体在多步执行中突破了前置护栏，其后续步骤被系统拦截的概率呈指数级下降。

## 5. 局限性分析 (Limitations)
- **评测静态性**：测试集主要基于预设的轨迹（Static Trajectories），未能完全模拟真实世界中智能体遇到拦截时的动态自适应绕过（Adaptive Evasion）。
- **计算开销**：在实际业务中，对每一步工具调用都进行附带长上下文历史的安全审查，会带来极大的延迟和 Token 成本。

## 6. 核心启示 (Takeaways)
- **智能体安全的范式转换**：Agent 的安全防线必须从“输入输出审查”升级为“状态机级别的轨迹审计（Stateful Trajectory Audit）”。
- **红队测试新思路**：“分片执行与上下文毒化”将成为 Agent 越狱的主流手段，防御系统需要具备跨 Step 的意图追踪与拼图还原能力。
