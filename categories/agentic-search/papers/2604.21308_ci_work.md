## 一、背景与问题

### 动机

企业 LLM Agent 真正危险的地方，不是它会不会“胡说八道”，而是它能否在检索大量内部上下文后，**把该说的信息说出来，同时把不该说的敏感内容稳稳压住**。这和日常助手隐私 benchmark 很不一样：企业工作流里同一任务常常要同时读取多封邮件、会议纪要、草稿文档和组织结构信息，essential context 与 sensitive context 是纠缠在一起的。

### 核心科学问题

能否构建一个面向企业工作流的 Contextual Integrity 基准，系统评测 LLM Agent 在复杂检索场景中是否既能完成任务，又不违反组织层级与角色边界造成信息泄漏？

### 直觉洞见

很多“高 utility”企业 Agent 其实只是更会把检索到的内容往外讲，而不是更会做细粒度边界裁剪。CI-Work 的关键价值，在于把“任务完成度”和“隐私合规度”拆开量化，直接暴露二者之间的结构性张力。

## 二、核心方法

**（1）CI-grounded 企业工作流建模**

论文以 Contextual Integrity 为理论底座，把企业代理任务建模为 sender → recipient 的信息流问题，覆盖 **5 类组织流向**：Upward、Downward、Lateral、Diagonal、External。这样一来，agent 的错误不再只是“答错”，而是“把不该流向某个角色的信息流了过去”。

**（2）Essential / Sensitive 双集合划分**

每个任务实例都不是简单问答，而是包含 dense retrieval context。作者把 agent 检索到的上下文显式划分为：
- **Essential Set**：完成任务必须传达的信息；
- **Sensitive Set**：一旦泄露就违反 CI 的信息。

这样就能同时衡量 utility 与 privacy，而不是只看单一 success rate。

**（3）高保真企业基准构建流程**

CI-Work 采用四阶段流水线：
1. Task-oriented seed generation；
2. Contextual entries generation；
3. Case episode generation；
4. Trajectory simulation and evaluation。

作者先人工设计种子，再用 LLM 自迭代扩展，确保生成的数据既符合业务逻辑，也符合组织层级约束。

**（4）压力测试与扩展评估**

除了主实验，论文还考察三类现实放大因素：
- context quantity / length 增大；
- user pressure（implicit / explicit）；
- model scaling / reasoning effort / prompt defense。

这使它不仅是一个 benchmark，也是在回答“企业部署里什么因素会把泄漏进一步放大”。

## 三、技术细节

### 风险模型

设企业数据仓为 D，agent 为 A，工具集为 T，任务指令为 I。agent 在执行过程中通过工具调用获得累计上下文集合：

E = ⋃_(t=1)^T o_t

然后把 E 划分为两个不相交集合：

E_ess ∩ E_sens = ∅

其中：
- E_ess：完成任务所必需的信息；
- E_sens：若向目标 recipient 披露则违反隐私规范的信息。

最终风险发生在 agent 的最终输出 a_f 同时：
1. 为了完成任务传达了 essential information；
2. 但也把 E_sens 中的内容一并泄露出去。

### benchmark 构建规模

- task-oriented seeds：**25** 个人工种子 + **100** 个 LLM 扩展种子，共 **125** 个；
- contextual entries：默认每个实例 **4 sensitive + 4 essential**；
- contextual entries 总量：**1,000**；
- agent 执行框架：**ReAct**；
- benchmark generation 与 evaluation 主力模型：**GPT-5.2**。

### 三类核心指标

- **Leakage Rate (LR, ↓)**：输出中泄露敏感信息的比例；
- **Violation Rate (VR, ↓)**：是否违反 contextual integrity 的总体比例；
- **Conveyance Rate (CR, ↑)**：是否成功传达 essential information。

CI-Work 的关键不是单独压低 LR，而是观察：

高 CR 往往伴随更高 LR / VR

这正是论文想抓的 privacy–utility trade-off。

## 四、实验设置与结果

### 数据集与评价指标

- **流向维度**：Upward / Downward / Lateral / Diagonal / External
- **模型**：4 个开源 + 5 个闭源 frontier LLMs
- **指标**：LR（↓）/ VR（↓）/ CR（↑）
- **额外分析**：context 数量、长度、user pressure、model scaling、reasoning effort、prompt defense

### 主结果（Table 1）

| 模型 | 平均 LR ↓ | 平均 VR ↓ | 平均 CR ↑ | 关键观察 |
|------|----------:|----------:|----------:|----------|
| GPT-5 | 11.21 | 27.83 | 93.04 | utility 高，但泄漏并不低 |
| GPT-4.1 | 12.07 | 25.91 | **94.57** | conveyance 很强，但隐私代价显著 |
| Grok-3 | **26.66** | **50.87** | 94.97 | 整体最危险 |
| DeepSeek-R1 | **6.08** | **15.80** | 53.08 | 泄漏最低，但 utility 也明显不足 |
| Kimi-K2 | 14.62 | 31.06 | **96.18** | 高 utility 同样伴随较高风险 |

论文总括性的 headline 很硬：
- **violation rates = 15.8%–50.9%**
- **leakage 最高可到 26.7%**

### 方向性差异

- **Upward** 方向的泄漏与违规显著高于 **Downward**；
- **External** 方向通常泄漏更低，说明模型对“公司外部”有一定粗粒度警惕；
- 但它们仍不擅长判断更细的 role–information compatibility，例如某个内部报价、内部草案是否应该给特定角色看。

### 压力与扩展实验

**（1）User pressure 会把问题放大成 lose–lose**
- implicit pressure 已会明显抬高 leakage / violation；
- explicit pressure 近乎使 baseline violation **翻倍**；
- 更反常的是 conveyance 也开始下降，说明模型在“用户催得更紧”时并没有更好完成任务，而是同时损害 privacy 与 utility。

**（2）Inverse Scaling 现象**

论文明确观察到企业场景中的 **inverse scaling**：更大的模型 utility 更高，但 privacy leakage 反而更严重。原因并非大模型“不懂规则”，而是它们更擅长从长上下文里找到有用内容，也更容易在用户导向下把敏感信息一并带出。

**（3）Prompt defense 效果有限**

| 设置 | LR ↓ | VR ↓ | CR ↑ |
|------|-----:|-----:|-----:|
| w/o defense | 11.21 | 27.83 | **93.04** |
| Prompt Defense | 8.96 | **21.31** | 81.01 |
| CI-CoT | **8.95** | 22.13 | **84.90** |

结论很直接：prompt-level mitigation 能缓解，但**不能从根上解决**，且会明显损失任务 utility。

### 关键发现

1. 企业 Agent 的隐私问题不是边缘 case，而是主流失败模式。  
2. 更高任务 utility 往往正相关于更高 leakage / violation。  
3. model scaling 和 reasoning effort 都不是根治方案。  
4. 真正需要的是 **context-centric architecture**，而不是继续迷信“大一点的模型会自己变安全”。

---

## 五、局限性分析

1. **评估仍依赖 LLM-based generation 与 evaluation**：虽然已尽量高保真，但 benchmark 构造本身仍不可避免地带有生成式偏差。  
2. **企业场景覆盖还不够宽**：目前重点是典型工作流与组织层级，不等于完整企业数据生态。  
3. **主要刻画 retrieval-heavy 情形**：对更复杂的 tool-use / multi-agent / long-horizon workflow 还可继续扩展。  
4. **防御侧结论偏现象学**：论文清晰证明“现有方法不够”，但完整 context-centric defense 仍未实现。

## 六、复现指南

### 环境依赖

pip install openai anthropic datasets pandas networkx

### 核心代码框架

```python
class CIWorkEvaluator:
    def __init__(self, agent, retriever, judge):
        self.agent = agent
        self.retriever = retriever
        self.judge = judge

    def run_case(self, instruction, datastore):
        trajectory = self.agent.react(instruction, tools=self.retriever.tools, datastore=datastore)
        retrieved = union_observations(trajectory)
        essential, sensitive = partition_context(retrieved, instruction)
        final_answer = trajectory[-1]["final_answer"]
        return {
            "leakage_rate": leakage_rate(final_answer, sensitive),
            "violation_rate": ci_violation(final_answer, sensitive),
            "conveyance_rate": conveyance(final_answer, essential),
        }
```

### 复现步骤

1. 根据论文的五类 organizational directions 构造 sender / recipient / task seeds。  
2. 为每个任务生成 essential entries 与 sensitive entries，并保证它们在语义上相互纠缠。  
3. 用 ReAct agent 在 retrieval 环境中执行任务，记录完整轨迹与最终回答。  
4. 分别计算 LR / VR / CR，并按 flow direction、context length、entry quantity 做切片分析。  
5. 额外加入 implicit / explicit user pressure、不同模型规模与 CI-CoT/prompt defense，对比 privacy–utility trade-off 是否恶化。  
6. 若用于真实企业部署，应把此 benchmark 进一步扩展到 memory、tool side effects、cross-agent handoff 等更真实的累积式泄漏路径。
