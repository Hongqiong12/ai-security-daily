## Module 1 — 背景与问题 (Background and Problem)

### 动机 (Motivation)
MCP 安全最近的讨论很容易只盯着“模型会不会被 prompt injection”。但真正把系统做烂的，往往不是模型理解错一句话，而是服务器开发者在协议层、schema 层和跨工具转发层留下了一串低级却高危的坑。论文的出发点非常工程化：与其继续做“模型有没有中招”的抽象 benchmark，不如把**开发者常犯的 MCP 安全失误**做成可复现、可验证、可回归测试的实验框架。

### 核心科学问题 (Core Scientific Question)
能否设计一个**协议感知（protocol-aware）**的 MCP 安全测试框架，把工具元数据投毒、木偶服务器、多模态图像到工具链攻击等多向量风险统一到同一评测体系里，并基于协议 trace 而非 agent 自述来判断安全失败？

### 直觉洞见 (Intuitive Insight)
MCP Pitfall Lab 的核心直觉是：Agent 经常会“说得好像没问题”，但 trace 里其实已经把敏感数据发出去了，或者把攻击者控制的参数真正传给了高风险 sink。只要继续相信 agent 的自述，就会系统性漏掉真实风险。因此，这篇论文把评测锚点从“模型怎么描述自己做了什么”换成了“**协议层到底发生了什么**”。

## Module 2 — 核心方法 (Core Method)

### 方法总览
MCP Pitfall Lab 是一个四层组合式框架：

**（1）Scenario 层**
- 定义真实工作流、初始状态与高风险 sink；
- 论文实例化了 3 个场景：邮件、文档、加密货币。

**（2）Attack Family 层**
- 统一建模三类攻击：
  - Tool Poisoning（工具元数据投毒）
  - Puppet Server（木偶/恶意服务器）
  - Multimodal Image-to-Tool Chain（图像到工具链攻击）

**（3）Prompt Variant 层**
- 同一攻击在不同提示表述与包装形式下生成多个变体。

**（4）Objective Validators 层**
- 不依赖 agent 自述；
- 基于 MCP trace 与环境状态判定：
  - **Vconf**：是否发生机密性泄露；
  - **Vint**：是否发生未授权高影响动作。

### 开发者失误分类（P1–P6）
论文把问题具体化为 6 类 pitfall：
1. **P1 Tool description as policy**
2. **P2 Overly permissive schema**
3. **P3 Cross-tool forwarding**
4. **P4 Image-to-tool leakage**
5. **P5 Missing audit logs**
6. **P6 Unvalidated high-risk inputs**

其中：
- **Tier-1（静态可检查）**：P1 / P2 / P5 / P6
- **Tier-2（依赖 trace / dataflow）**：P3 / P4

### 方法定位
这篇论文最有价值的地方，是把 MCP 安全从“继续测模型”推进到“**测协议、测工具、测开发者实现质量**”。这意味着 Agent 安全评测开始真正贴近工程交付面。

## Module 3 — 技术细节 (Technical Details)

### 评测核心对象
每次运行都会产出两类关键工件：
- **结构化 run report**：汇总 validator 结论和支持证据；
- **JSONL 协议 trace**：记录 tool discovery、tool call、tool result、跨工具数据流、图像工件 provenance 等事件。

### 风险评分
论文将风险按严重性加权：
- high = **2**
- medium = **1**
- low = **0.5**

总体风险分数 capped 到 **0–10**。

### 成本效益指标
论文定义：

CE = Δrisk / (1 + log10(ΔLOC))

其中：
- Δrisk：加固前后风险改善量
- ΔLOC：为加固额外改动的代码行数

### 实验环境
- 系统：**macOS, Apple M2**
- Python：**3.10**
- MCP 框架：**FastMCP 2.14.0**
- Agent backbone：**GPT-4.1-mini**
- MCP control plane、tool servers、validator 均为 **in-process**

## Module 4 — 实验设置与结果 (Experiments and Results)

### 数据规模
- **3 个场景**：emailsystem / documentsystem / ETHPriceServer
- 每个场景各有：
  - **Baseline**（脆弱版 server）
  - **Hardened**（加固版 server）
- 共 **6 个 server variants**
- 攻击模板：**12 个 singular templates**，来自 **11 个 attack subfamilies**
- 总 submissions：**108**
- 总 agent runs：**324**（每个 submission 做 **3 次 trial**）

### 静态分析结果（6 servers, 36 binary labels）
| 类别 | 描述 | Precision | Recall | F1 |
|------|------|----------:|-------:|---:|
| P1 | Tool description as policy | 1.0 | 1.0 | **1.0** |
| P2 | Overly permissive schema | 1.0 | 1.0 | **1.0** |
| P3 | Cross-tool forwarding | 0.0 | 0.0 | 0.0 |
| P4 | Image-to-tool leakage | 0.0 | 0.0 | 0.0 |
| P5 | Missing audit logs | 1.0 | 1.0 | **1.0** |
| P6 | Unvalidated high-risk inputs | 1.0 | 1.0 | **1.0** |
| Aggregate | 宏平均 | 0.67 | 0.67 | **0.67** |

静态分析平均耗时仅 **5.2 ms**。

### 基线 vs 加固版问题数
| 场景 | 版本 | Total | HIGH | MEDIUM |
|------|------|------:|-----:|-------:|
| emailsystem | Baseline | 8 | 4 | 4 |
| emailsystem | Hardened | 0 | 0 | 0 |
| documentsystem | Baseline | 9 | 5 | 4 |
| documentsystem | Hardened | 0 | 0 | 0 |
| ETHPriceServer | Baseline | 12 | 7 | 5 |
| ETHPriceServer | Hardened | 0 | 0 | 0 |
| **All** | **Baseline** | **29** | **16** | **13** |
| **All** | **Hardened** | **0** | **0** | **0** |

### 风险与成本效益
| 场景 | Risk_base | Risk_hard | Δlog% | Δval% | ΔLOC | CE |
|------|----------:|----------:|------:|------:|-----:|---:|
| emailsystem | 10.0 | 0.0 | +100% | +25% | 32 | 3.99 |
| documentsystem | 10.0 | 0.0 | +100% | +50% | 15 | 4.60 |
| ETHPriceServer | 10.0 | 0.0 | +100% | +40% | 35 | 3.93 |
| **Mean** | **10.0** | **0.0** | **+100%** | **+38%** | **27** | **4.17** |

### Trace vs agent 自述偏差
在 emailsystem 的 **19 runs** 中：
- 总偏差率：**12/19 = 63.2%**
- 对发生 sink action 的运行：**12/12 = 100%**
- 全部偏差均为 **D5: Sink Misattribution**，即 agent 会描述“完成了发送”，但不准确报告真实 MCP sink 调用细节。

### 关键发现
- MCP 安全的高危点大量来自**开发者实现细节**，而非模型本身；
- 只做静态分析足以清空 Tier-1 问题，但 P3 / P4 必须做 trace / dataflow 级验证；
- 平均只需 **27 行代码** 的加固，就能把风险从 **10.0 清零到 0.0**；
- **不能再依赖 agent 自述做审计**，trace 才是唯一可信锚点。

## Module 5 — 局限性分析 (Limitations)

1. **主要覆盖 confidentiality 与高风险动作，不等于完整 Agent 安全闭环**：对更复杂 integrity / availability 问题仍需扩展。  
2. **场景数仍有限**：目前只有邮件、文档、加密货币三类工作流，更多 enterprise MCP 场景尚未纳入。  
3. **Tier-2 仍需要动态运行**：像 cross-tool forwarding、image-to-tool leakage 这类问题无法仅靠静态分析捕获。  
4. **当前仍未覆盖更大规模生态审计**：如 registry 级 server 搜索、批量 CI 接入、自动修复等仍有空间。  
5. **修复效果依赖推荐加固项被正确实现**：例如 schema 约束（M1–M3）在论文实例里其实仍未全部落地。

## Module 6 — 复现指南 (Reproduction Guide)

### 环境依赖 (Environment)
```bash
pip install fastmcp pydantic sqlite-utils python-dotenv
```

### 核心代码框架 (Core Code)
```python
scenario = load_scenario("emailsystem")
server = build_server(variant="baseline")
trace = []

for attack in attack_templates:
    result = run_agent_with_validators(server, scenario, attack)
    trace.append(result.trace)
    report = validators.evaluate(result)
```

### 复现步骤 (Reproduction Steps)
1. 搭建 3 个场景的 baseline / hardened MCP servers。  
2. 实现 P1–P6 的静态规则检测。  
3. 为 Tool Poisoning、Puppet Server、Image-to-Tool 三类攻击构造模板与 validator。  
4. 每个 submission 运行 **3 次**，保存 JSONL trace 与 run report。  
5. 统计静态分析、风险分数、LOC 改动与 CE。  
6. 关键核验点：
   - Tier-1 问题应可达到 **F1 = 1.0**；
   - baseline 总问题数应接近 **29**，hardened 为 **0**；
   - self-report / trace 偏差率应在 sink-action 样本上接近 **100%**。
