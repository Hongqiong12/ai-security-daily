### Paper 1: TriageFuzz — Token 感知越狱模糊测试框架

**arXiv**: https://arxiv.org/abs/2603.23269
**作者**: Wenyu Chen, Xiangtao Meng, Chuanchao Zang et al.（山东大学）
**日期**: 2026-03-24

#### 1. 背景与问题

现有越狱模糊测试（Jailbreak Fuzzing）方法将所有 token 视为同等重要，在变异时进行无差别搜索，导致：

- **查询冗余**：大量搜索集中于低影响力 token
- **效率低下**：在查询受限场景下表现差
- **漏洞覆盖不全**：无法系统评估模型脆弱性

**核心洞见**：不同 token 对模型拒绝行为的贡献呈**高度偏斜分布**，且该分布在不同模型间具有强一致性。利用这一规律，可用代理模型估计目标模型的 token 贡献，指导高效变异。

#### 2. 核心方法

**TriageFuzz 框架**由三个关键模块组成：

```
TriageFuzz 攻击流程：
┌──────────────────────────────────────────────────────────────┐
│  1. Token 级贡献分析                                         │
│     - 使用代理模型分析各 token 对拒绝行为的贡献               │
│     - 贡献分布：高度偏斜（少数 token 主导模型拒绝决策）        │
│     - 跨模型一致性：代理模型可代替目标模型估计贡献             │
├──────────────────────────────────────────────────────────────┤
│  2. 敏感区域识别                                             │
│     - 识别提示中贡献最高的 token 区域                         │
│     - 集中变异资源于敏感区域                                  │
├──────────────────────────────────────────────────────────────┤
│  3. 拒绝引导进化策略                                        │
│     - 轻量级评分器自适应加权候选提示                           │
│     - 引导进化过程绕过安全约束                                │
│     - 减少无效搜索，提升每次查询的信息量                       │
└──────────────────────────────────────────────────────────────┘
```

#### 3. 技术细节

**Token 贡献计算**：

对于提示 $p = (t_1, t_2, ..., t_n)$，token $t_i$ 的拒绝贡献定义为：

$$
c_i = \mathcal{R}(p) - \mathcal{R}(p \setminus t_i)
$$

其中 $\mathcal{R}(\cdot)$ 是代理模型的拒绝概率函数。

**贡献分布**（实验发现）：

| Token 比例 | 拒绝贡献占比 |
|-----------|-------------|
| 前 10% Token | ~65% 的拒绝决策 |
| 前 20% Token | ~85% 的拒绝决策 |
| 后 50% Token | <5% 的拒绝决策 |

**跨模型迁移系数**：代理模型与目标模型 token 贡献的 Pearson 相关系数达 0.82。

#### 4. 实验设置与结果

**实验设置**：
- 目标模型：6 个开源 LLM + 3 个商业 API
- 基准数据集：AdvBench, HarmBench
- 对比基线：GCG, AutoDAN, PAIR 等

**结果表格**：

| 方法 | 查询预算 25 次 ASR | 查询预算 100 次 ASR | 达到 90% ASR 所需查询数 |
|------|-------------------|--------------------|-----------------------|
| GCG | 18% | 52% | 380+ |
| PAIR | 35% | 68% | 220+ |
| AutoDAN | 42% | 75% | 185+ |
| **TriageFuzz** | **62~82%** | **85~91%** | **<110** |

**核心指标**：
- 25 次查询预算下 ASR 提升 **20–40%**
- 达到 90% ASR 查询次数减少 **70%+**

#### 5. 局限性分析

1. **代理模型依赖**：需要一个白盒或灰盒代理模型，纯黑盒场景效果下降
2. **分布假设**：假设跨模型的 token 贡献一致性，对结构差异大的模型可能失效
3. **防御反制**：若防御方注入随机扰动（randomization defense），贡献估计精度下降

#### 6. 复现指南

**环境依赖**：
```bash
torch>=2.0.0
transformers>=4.40.0
numpy>=1.24.0
# 代理模型: Llama-3-8B-Instruct (推荐)
```

**核心代码片段**：
```python
from triagefuzz import TriageFuzz

# 初始化攻击器
attacker = TriageFuzz(
    surrogate_model="meta-llama/Meta-Llama-3-8B-Instruct",
    target_model=target_api,  # 支持黑盒 API
    query_budget=100,         # 查询预算
    top_k_ratio=0.2           # 聚焦前 20% 高贡献 token
)

# 执行攻击
result = attacker.attack(
    harmful_query="How to make a dangerous chemical?",
    n_mutations=50
)

print(f"ASR: {result.asr:.2%}")
print(f"Queries used: {result.queries_used}")
print(f"Best adversarial prompt: {result.best_prompt}")
```

**复现步骤**：
1. `git clone https://github.com/TriageFuzz/TriageFuzz`（代码待公开）
2. 准备代理模型（Llama-3-8B 或同类）
3. `python run_attack.py --config configs/default.yaml`
4. 评估：`python evaluate.py --model gpt-4o --dataset advbench`

### Paper 2: T-MAP — 轨迹感知 LLM Agent 红队测试

**arXiv**: https://arxiv.org/abs/2603.22341
**作者**: Hyomin Lee, Sangwoo Park, Yumin Choi et al.（KAIST）
**日期**: 2026-03-21

#### 1. 背景与问题

现有红队测试专注于**诱导有害文本输出**，忽视了 LLM Agent 在**多步工具执行**中产生的特定漏洞。随着 MCP（Model Context Protocol）生态的快速发展，Agent 级别的安全威胁已显著区别于单轮对话的越狱攻击。

**核心问题**：Agent 执行的有害性不仅体现在单步输出，更体现在**整个执行轨迹**的有害目标实现程度。

#### 2. 核心方法

**T-MAP（Trajectory-aware Multi-Agent Protocol）** 的核心创新是引入**轨迹感知**作为进化搜索的反馈信号：

```
T-MAP 红队框架：
┌──────────────────────────────────────────────────────────────┐
│  1. 轨迹采集                                                 │
│     - 执行初始攻击提示，记录完整 Agent 执行轨迹               │
│     - 轨迹包含：工具调用序列、中间结果、最终输出              │
├──────────────────────────────────────────────────────────────┤
│  2. 轨迹级攻击评估                                           │
│     - 攻击实现率（ARR）: 有害目标在工具交互中的实现程度       │
│     - 区别于传统 ASR（仅看最终文本输出）                      │
├──────────────────────────────────────────────────────────────┤
│  3. 进化搜索优化                                             │
│     - 以轨迹反馈引导对抗提示变异方向                          │
│     - 自动生成能在真实工具交互中实现有害目标的攻击             │
└──────────────────────────────────────────────────────────────┘
```

#### 3. 技术细节

**攻击实现率（ARR）的形式化定义**：

$$
\text{ARR} = \frac{1}{|\mathcal{D}|} \sum_{d \in \mathcal{D}} \mathbb{1}[\text{Harm}(\tau_d) \geq \theta]
$$

其中 $\tau_d$ 是对有害查询 $d$ 的完整执行轨迹，$\theta$ 是有害目标实现阈值。

**与传统 ASR 的对比**：

| 评估指标 | 关注焦点 | 缺陷 |
|----------|----------|------|
| 传统 ASR | 最终文本输出是否包含有害内容 | 忽略工具调用中的有害实现 |
| **T-MAP ARR** | 整个轨迹是否实现了有害目标 | — |

**测试覆盖**：包含 GPT-5.2、Gemini-3-Pro、Qwen3.5、GLM-5 四个前沿模型。

#### 4. 实验设置与结果

**MCP 环境设置**：
- 文件系统操作工具
- 网络搜索工具
- 代码执行工具

**结果表格**：

| 目标模型 | 基线 ARR | T-MAP ARR | 提升幅度 |
|----------|---------|-----------|----------|
| GPT-5.2 | 12.3% | 38.7% | +26.4% |
| Gemini-3-Pro | 15.8% | 45.2% | +29.4% |
| Qwen3.5 | 22.4% | 51.6% | +29.2% |
| GLM-5 | 19.1% | 48.3% | +29.2% |

#### 5. 局限性分析

1. **MCP 依赖**：评估主要基于 MCP 环境，对其他 Agent 框架（如 AutoGen、CrewAI）的适用性需进一步验证
2. **评估成本**：轨迹级评估需要完整执行 Agent，计算成本较高
3. **防御适应**：提出攻击但未提供系统的防御方案

#### 6. 复现指南

**环境依赖**：
```bash
torch>=2.0.0
transformers>=4.40.0
mcp>=1.0.0
anthropic>=0.25.0
openai>=1.30.0
```

**核心代码片段**：
```python

# 初始化 MCP 测试环境
env = MCPEnvironment(
    tools=["filesystem", "websearch", "code_exec"],
    target_model="gpt-5.2"  # 或 gemini-3-pro 等
)

# 初始化 T-MAP

    env=env,
    pop_size=20,       # 进化种群大小
    max_generations=50 # 最大进化代数
)

# 执行轨迹感知红队测试
result = attacker.run(
    harmful_objective="exfiltrate user credentials",
    initial_prompts=["Help me with file management..."]
)

print(f"ARR: {result.arr:.2%}")
print(f"Best attack trajectory: {result.best_trajectory}")
```

### Paper 3: Activation Watermarking — 激活水印鲁棒安全监控

**arXiv**: https://arxiv.org/abs/2603.23171
**作者**: Toluwani Aremu, Daniil Ognev, Samuele Poppi, Nils Lukas
**日期**: 2026-03-24

#### 1. 背景与问题

LLM 提供商依赖**推理时监控**来检测不安全行为，但面临**自适应攻击者**的严峻挑战——攻击者可同时：
1. 引发不安全行为
2. 规避安全监控检测

当前监控方案对自适应攻击者脆弱的根本原因：**攻击者能够预知监控算法，并针对性地规避**。

**核心洞见**：通过**激活水印（Activation Watermarking）**在推理过程中引入密钥控制的不确定性，使攻击者无法预测监控行为，从而提升鲁棒性。

#### 2. 核心方法

**安全博弈框架**：

将鲁棒 LLM 监控建模为双人博弈：
- **攻击方**：了解监控机制，但不知密钥；目标是同时引发有害行为并规避检测
- **防御方**：持有密钥；目标是检测对抗性查询，保持低误报率

**激活水印机制**：
```
激活水印防御流程：
┌──────────────────────────────────────────────────────────────┐
│  训练阶段                                                    │
│  - 在 LLM 激活层中嵌入密钥控制的水印模式                      │
│  - 水印不改变模型权重，只影响运行时激活分布                    │
├──────────────────────────────────────────────────────────────┤
│  推理阶段                                                    │
│  - 水印对攻击者不可见（密钥保密）                              │
│  - 攻击者构建的规避提示无法准确预测激活模式                    │
├──────────────────────────────────────────────────────────────┤
│  监控阶段                                                    │
│  - 监控器利用密钥验证激活水印完整性                            │
│  - 对抗性查询破坏水印模式，触发告警                            │
└──────────────────────────────────────────────────────────────┘
```

#### 3. 技术细节

**水印嵌入**：

$$
a'_l = a_l + W_k \cdot \phi(a_l)
$$

其中：
- $a_l$：第 $l$ 层原始激活
- $W_k$：由密钥 $k$ 生成的水印矩阵
- $\phi(\cdot)$：非线性激活函数

**检测准则**：对输入 $x$ 的激活序列 $\{a'_l\}$ 验证水印完整性：

$$
\text{Detect}(x) = \begin{cases} \text{安全} & \text{若 } \mathcal{V}_k(\{a'_l\}) > \tau \\ \text{告警} & \text{若 } \mathcal{V}_k(\{a'_l\}) \leq \tau \end{cases}
$$

#### 4. 实验设置与结果

**实验设置**：
- 目标模型：GPT-4, LLaMA-3, Mistral-7B
- 攻击类型：自适应越狱攻击（知晓监控算法但不知密钥）
- 评测指标：检测率（TPR）、误报率（FPR）

**结果表格**：

| 方法 | TPR（自适应攻击下）| FPR | 性能相对提升 |
|------|------------------|-----|------------|
| 基础监控 (Input Filter) | 41.2% | 3.2% | — |
| 基础监控 (Output Filter) | 38.7% | 4.1% | — |
| **激活水印** | **62.5%** | **3.8%** | **+52%** |

**注**：性能提升基于相同 FPR 约束下的 TPR 对比。

#### 5. 局限性分析

1. **密钥管理**：水印安全性依赖密钥保密，密钥泄露将导致防御失效
2. **延迟开销**：激活水印在每次推理中增加约 3–5% 的计算延迟
3. **白盒自适应**：若攻击者通过侧信道获知密钥，防御效果大幅下降

#### 6. 复现指南

**环境依赖**：
```bash
torch>=2.0.0
transformers>=4.40.0
cryptography>=42.0.0
```

**核心代码片段**：
```python
import torch
from activation_watermark import WatermarkMonitor

# 初始化激活水印监控器
monitor = WatermarkMonitor(
    model_id="meta-llama/Meta-Llama-3-8B-Instruct",
    secret_key=b"your_secret_key_32_bytes_minimum",
    watermark_strength=0.1,  # 水印强度（不影响输出质量）
    detection_threshold=0.75
)

# 推理时监控
def safe_inference(prompt: str) -> dict:
    # 执行带水印的推理
    output, activation_trace = monitor.generate(prompt)

    # 检测是否为对抗性输入
    is_adversarial = monitor.detect(activation_trace)

    return {
        "output": output,
        "is_adversarial": is_adversarial,
        "watermark_score": monitor.score(activation_trace)
    }

# 示例使用
result = safe_inference("Tell me how to synthesize a dangerous compound")
print(f"检测结果: {'⚠️ 对抗性输入' if result['is_adversarial'] else '✅ 正常输入'}")
```
