### 论文一：SecureBreak — 构建安全可靠模型的基准数据集

**《SecureBreak -- A dataset towards safe and secure models》**

arXiv: https://arxiv.org/abs/2603.21975 | 领域: cs.CR / cs.AI / cs.CL | 作者: Marco Arazzi, Vignesh Kumar Kembu, Antonino Nocera | 发布: 2026-03-23 | GitHub: 暂无公开代码库

#### 一、背景与问题

大型语言模型在商业部署中的安全对齐机制并不充分——越狱（Jailbreaking）和提示注入（Prompt Injection）攻击可以绕过现有对齐，导致有害内容输出。已有研究主要聚焦于对齐方法本身，而对"如何检测对齐失败时的有害输出"关注不足。

**科学问题**：如何构建一个高质量的 LLM 有害输出检测数据集，使微调后的 LLM 能够作为"最终防御层"？

**直觉洞见**：安全对齐是充分但不充分的条件，纵深防御需要在生成阶段独立增加检测层；而这一层的质量取决于训练数据的质量。

#### 二、核心方法

SecureBreak 是一个专为 LLM 安全检测设计的有标注数据集，核心设计哲学是**保守策略**：对边界案例倾向有害判定，确保安全覆盖率优先于精确率。数据集支持两种用途：（a）训练阶段安全对齐的质量反馈；（b）部署阶段的生成后过滤器训练与评估。

#### 三、技术细节

数据构建的四阶段流程：原始数据收集（来自已知攻击案例）→ 自动初筛 → 人工精标（保守策略）→ 跨标注者一致性验证。

安全分类器形式化目标：

$$\mathcal{C}(o) = \begin{cases} 1 & \text{若 } o \in \mathcal{H}（\text{有害集合}） \\ 0 & \text{若 } o \in \mathcal{S}（\text{安全集合}） \end{cases}$$

保守策略的核心假设：$\text{FN（漏报）} \gg \text{FP（误报）}$ 代价，因此标注向 $\mathcal{H}$ 偏移。

#### 四、实验设置与结果

| 实验设置 | 主要发现 |
|---------|--------|
| 零样本预训练 LLM | 在 SecureBreak 上性能优于 baseline |
| 微调后 LLM | 有害内容检测性能显著提升 |
| 跨类别泛化 | 多危害类别覆盖均有效 |

#### 五、局限性分析

数据集规模未公开；静态构建无法应对新型攻击；多语言覆盖不明；分布外鲁棒性有待验证。

### 论文二：SFCoT — 推理过程中的主动安全校准

**《SFCoT: Safer Chain-of-Thought via Active Safety Evaluation and Calibration》**

arXiv: https://arxiv.org/abs/2603.15397 | 领域: cs.CR / cs.AI | 作者: Yu Pan, Wenlong Yu, Tiejun Wu, Xiaohu Ye, Qiannan Si, Guangquan Xu, Bin Wu | 发布: 2026-03-16 | GitHub: 暂无公开代码库

#### 一、背景与问题

链式推理（CoT）已成为 LLM 复杂任务推理的核心技术，但也为越狱攻击提供了新的攻击面：攻击者可在中间推理步骤逐步引导模型产生有害输出，而现有防御只在最终输出阶段拦截，忽视了推理链中的风险积累。

**科学问题**：能否在推理链的每个中间步骤实时评估安全性，并通过动态干预从源头阻止有害输出？

**直觉洞见**：类比人类思考中的自我审查——在"内心"步骤中就中止有害念头，而非等到"说出口"再被拦截。

#### 二、核心方法

SFCoT 的三组件架构：

1. **三层安全评分**：从语义安全 $\sigma_{sem}$、意图合规 $\sigma_{intent}$、行为风险 $\sigma_{risk}$ 三维评估每个推理步骤
2. **多视角一致性验证**：多视角投票，避免单点误判
3. **动态干预**：安全分低于阈值时触发语义重写、路径重定向或推理终止

#### 三、技术细节

综合安全评分：

$$\sigma(s_t) = \alpha \cdot \sigma_{sem}(s_t) + \beta \cdot \sigma_{intent}(s_t) + \gamma \cdot \sigma_{risk}(s_t), \quad \alpha + \beta + \gamma = 1$$

干预触发条件：$\text{Intervene}(s_t) = \mathbf{1}[\sigma(s_t) < \tau]$

多视角一致性验证的投票机制：$\hat{\sigma}(s_t) = \text{majority\_vote}(v_1(s_t), \ldots, v_K(s_t))$

#### 四、实验设置与结果

测试于 HarmBench、AdvBench、JailbreakBench；攻击方法覆盖 GCG、AutoDAN、PAIR、TAP。

| 方法 | ASR（↓） | 良性准确率 | 推理开销 |
|------|---------|----------|--------|
| 无防御 | 58.97% | 基准 | — |
| 输出过滤 | 34.21% | 基准 | 低 |
| **SFCoT（本文）** | **12.31%** | 接近基准 | 中等 |

消融：完整三层（12.31%）> 去掉一致性验证（19.7%）> 仅语义层（28.4%）

#### 五、局限性分析

实时评估引入计算开销；阈值调优敏感；自适应攻击可能规避检测。

#### 六、复现指南

```python
class SFCoTEngine:
    def __init__(self, safety_scorer, threshold=0.7):
        self.scorer = safety_scorer; self.threshold = threshold
    
    def generate_safe_cot(self, prompt, model, max_steps=10):
        cot_steps, context = [], prompt
        for i in range(max_steps):
            step = model.generate_step(context)
            if self.scorer.evaluate(step) < self.threshold:
                step = self.intervene(step, context)
            cot_steps.append(step); context += f"\nStep {i+1}: {step}"
            if model.is_complete(step): break
        return "\n".join(cot_steps)
```

### 论文三：LLM Agent Security — 形式化上下文安全框架

**《A Framework for Formalizing LLM Agent Security》**

arXiv: https://arxiv.org/abs/2603.19469 | 领域: cs.CR / cs.AI | 作者: Vincent Siu, Jingxuan He, Kyle Montgomery, Zhun Wang, Neil Gong, Chenguang Wang, Dawn Song | 发布: 2026-03-19 | GitHub: 暂无公开代码库

#### 一、背景与问题

LLM Agent 的安全性高度上下文相关：同一行为在不同授权场景下安全属性截然不同。现有防御缺乏上下文感知能力，导致过度防御损害效用，或防御不足留下漏洞。

**科学问题**：如何形式化 LLM Agent 的上下文安全属性，建立统一的攻防分析框架，使防御设计更精准？

**直觉洞见**：借鉴操作系统访问控制（Bell-LaPadula 模型），将 Agent 安全属性映射到任务、行为、数据流和授权边界的四元组描述。

#### 二、核心方法

四个上下文安全属性 + 预言函数：

| 属性 | 含义 |
|-----|------|
| P1 任务对齐 | 追求且仅追求经授权的目标 |
| P2 行为对齐 | 单步行为服务于当前授权目标 |
| P3 来源授权 | 仅执行来自认证来源的指令 |
| P4 数据隔离 | 信息流遵守权限边界 |

预言函数 $O_i(a, c)$：判定属性 $P_i$ 在上下文 $c$ 中是否被行为 $a$ 违反。

#### 三、技术细节

攻击-属性映射表：

| 攻击类型 | 违反属性 |
|---------|---------|
| 直接提示注入 | P3 |
| 间接提示注入 | P1 + P3 |
| 越狱 | P1 + P2 |
| 任务漂移 | P1 |
| 内存污染 | P3 + P4 |

效用损失正比于防御范围的过度扩展：

$$\text{UtilityLoss} \propto |\{c : O_i(a,c) = \text{violated but } a \text{ is legitimate}\}|$$

上下文感知防御通过缩小防御作用域降低效用损失。

#### 四、实验设置与结果

理论框架论文，主要通过映射验证：覆盖 AgentDojo（P1+P2）、InjecAgent（P3）、PrivacyLens（P4）、SafeToolUse（P1+P2强化）等现有工作。

#### 五、局限性分析

预言函数实现复杂；多智能体协作场景尚未充分覆盖；静态权限框架难以适应动态权限场景。

#### 六、复现指南

```python
class LLMAgentSecurityFramework:
    def __init__(self, authorized_tasks, authorized_sources):
        self.authorized_tasks = authorized_tasks
        self.authorized_sources = authorized_sources
    
    def evaluate_security(self, action, context):
        violations = []
        if context.get("task") not in self.authorized_tasks:
            violations.append("P1_task_alignment")
        if context.get("source") not in self.authorized_sources:
            violations.append("P3_source_authorization")
        return violations
```

### 论文四：PromptSan — NSFW 分类器引导的提示净化

**《NSFW-Classifier Guided Prompt Sanitization for Safe Text-to-Image Generation》**

arXiv: https://arxiv.org/abs/2506.18325 | 领域: cs.CV | 作者: Yu Xie, Chengjie Zeng, Lingyun Zhang, Yanwei Fu | 发布: 2025-06-23 | GitHub: 暂无公开代码库

#### 一、背景与问题

文本到图像模型可被恶意提示诱导生成 NSFW 内容（色情、暴力等）。现有防御或需修改扩散模型（概念擦除），或仅在输出端检测（被动事后）。PromptSan 另辟蹊径，在**输入端**净化有害提示，代价最低且无需修改模型。

#### 二、核心方法

两种净化变体：

**PromptSan-Modify**：迭代替换有害词汇 → 直至通过文本 NSFW 分类器

**PromptSan-Suffix**：训练优化后缀令牌，附加到有害提示后同时满足文本和图像 NSFW 分类器

#### 三、技术细节

Suffix 优化目标：

$$\min_{S} \mathcal{L}_{text}(P_{harm} \oplus S) + \lambda \cdot \mathcal{L}_{image}(\text{Diffuse}(P_{harm} \oplus S))$$

双分类器联合约束确保净化效果在文本和图像两个层面均有效，而非仅处理文本表面。

#### 四、实验设置与结果

基准：SneakyPrompt、I2P、Ring-A-Bell 测试集；指标：DSR（防御成功率）、FID、CLIP Score

| 方法 | DSR（↑） | FID 影响（↓越好） |
|------|---------|-----------------|
| ESD | 81.5% | +5.3 |
| **PromptSan-Modify** | **85.7%** | +1.3 |
| **PromptSan-Suffix** | **89.2%** | +1.8 |

相比概念擦除类方法，对良性生成质量影响最小（FID 提升最低）。

#### 五、局限性分析

专门对抗优化的提示可能失效；语义边界模糊时净化损失合理语义；推理延迟增加；主要支持英文。

#### 六、复现指南

```python
def optimize_suffix(harmful_prompt, nsfw_text_clf, nsfw_img_clf, pipe, k=10, steps=500):
    S = torch.randn(k, 768, requires_grad=True)
    opt = torch.optim.Adam([S], lr=0.01)
    for t in range(steps):
        p = harmful_prompt + " " + decode(S)
        loss = nsfw_text_clf(p).logits[0,1]
        if t % 50 == 0:
            loss += 0.5 * nsfw_img_clf(pipe(p).images[0])
        opt.zero_grad(); loss.backward(); opt.step()
    return S.detach()
```

### 论文五：Modular LoRA — 防止微调后安全对齐崩溃

**《Safety Alignment Backfires: Preventing the Re-emergence of Suppressed Concepts in Fine-tuned Text-to-Image Diffusion Models》**

arXiv: https://arxiv.org/abs/2412.00357 | 领域: cs.AI / cs.CV | 作者: Sanghyun Kim, Moonseok Choi, Jinwoo Shin, Juho Lee | 机构: KAIST | 发布: 2024-11-30 | GitHub: 暂无公开代码库

#### 一、背景与问题

T2I 模型通过概念擦除（ESD 等）实现安全对齐，但在**良性数据集上微调**后，安全机制意外失效、有害概念重现——这一"安全对齐回弹"（Safety Alignment Backfire）问题在 T2I 领域尚未充分研究。

**科学问题**：如何设计安全感知的微调方法，使 T2I 模型在学习新任务时保留安全对齐效果？

#### 二、核心方法

**Modular LoRA** 将 LoRA 参数空间拆分为两个物理独立的模块：
- **Safety LoRA**（$\Delta W_{safe}$）：冻结，编码安全约束
- **Task LoRA**（$\Delta W_{task}$）：与 Safety LoRA **正交**，自由学习新任务

推理时合并：$W_{final} = W_{pretrained} + \Delta W_{safe} + \Delta W_{task}$

#### 三、技术细节

正交梯度更新规则（防止任务梯度覆盖安全方向）：

$$A_{task}^{(t+1)} = A_{task}^{(t)} - \eta \nabla \mathcal{L}_{task} - \underbrace{\text{proj}_{A_{safe}} \left(\eta \nabla \mathcal{L}_{task}\right)}_{\text{去除平行安全方向的分量}}$$

LoRA 秩 $r$ 的合理选取决定安全与任务的容量分配。

#### 四、实验设置与结果

评估基于 ESD/MACE/UCE 安全对齐后，使用 DreamBooth-style 良性数据微调，测量裸露等有害概念的重现率。

| 微调方法 | HGR（↓） | CLIP Score（↑） |
|---------|---------|----------------|
| 标准 LoRA | 67.3% | 0.312 |
| Adapter | 51.2% | 0.298 |
| **Modular LoRA** | **8.4%** | **0.309** |

消融：完整（8.4%）> 仅冻结（23.7%）> 无正交约束（45.1%）

#### 五、局限性分析

秩选择敏感；正交约束为近似；多概念擦除时 Safety LoRA 容量可能不足；无法防范未对齐过的新型有害概念。

#### 六、复现指南

```python
# 正交约束梯度投影核心实现
def orth_grad(task_grad, safety_weight):
    dot = (task_grad * safety_weight).sum()
    norm = (safety_weight ** 2).sum()
    return task_grad - (dot / norm) * safety_weight

# 在每次微调反向传播后应用
for tp, sp in zip(task_lora.parameters(), safety_lora.parameters()):
    if tp.grad is not None:
        tp.grad.data = orth_grad(tp.grad.data, sp.data)
```
