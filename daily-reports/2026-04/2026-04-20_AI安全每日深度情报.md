## 今日概览

今天最强的主线不是“又多了几篇 attack / benchmark”，而是三个模态都在逼近同一个更底层的问题：**安全边界并不是孤立概念、孤立 prompt 或孤立工具的属性，而是会被组合语义、研究语境与执行载体共同重写的系统性边界**。

在 **T2I** 方向，这个信号最明显。**TwoHamsters** 直接证明：即便每个原子概念单独都安全，组合后仍可能稳定生成侮辱、仇恨、羞辱或危险内容；而且模型越强、越会精确执行提示，反而越脆弱，FLUX 在 MCCU 场景下的成功率高达 **99.52%**。与此同时，**TICoE** 则把概念擦除从“单 prompt 删概念”推进到“图文协同精准外科手术”：用 CCCM 覆盖文本变体、用 HVRL 压制视觉误伤，在 gun 擦除上做到 **ASR 0.00 / UDA 0.02 / P4D 0.04**，说明 T2I 安全正从“能不能删”转向“能不能在组合语义与相似概念干扰下删得准”。

在 **T2T** 方向，今天两篇论文共同揭示安全边界的两种软化机制。**Into the Gray Zone** 指出，安全研究语境本身就会成为一类解锁信号，多轮语境构造后的 Jargon 攻击把 7 个主流模型的平均 ASR 推到 **99.0%**，包括 GPT-5.2 的 **93%** 与 Claude Sonnet / Opus 的 **100%**。而 **Pruning Unsafe Tickets** 则从参数子结构层面给出一个很实用的反方向回答：如果危险行为集中在少量 unsafe tickets 上，那么部署后剪枝就能显著降低不安全输出。它把 Mistral-7B 的 Unsafe Rate 从 **22.8%** 压到 **1.17%**，并把 AutoDAN ASR 从 **45.7** 压到 **0.67**。这意味着 T2T 安全现在一边暴露“研究语境灰区”，一边开始探索“结构性后处理修补”。

在 **Agentic** 方向，风险进一步从 prompt surface 下沉到生态和基础设施。**HarmfulSkillBench** 说明开放 skill 生态中，危险能力本身就是攻击面：在 **98,440** 个 skills 中识别出 **4,858** 个 harmful skills，而仅仅把 harmful skill 暴露给 agent，就会让平均 harm score 从无技能时的 **0.27** 升到 **0.76**。**LogJack** 则把运维日志正式确认为高价值间接注入载体：Llama 3.3 70B 在 Active 调试条件下的日志劫持率高达 **86.2%**，且 **6/8** 个模型可被推进到 RCE，Azure / GCP 输入侧 guardrail 对日志上下文中的 payload 近乎全部失效。换句话说，Agent 安全已经不能只盯“工具有没有漏洞”，而必须把**技能生态治理 + 遥测数据可信性 + 执行动作边界**放到同一张风险图里。

## T2I 论文（文生图安全）

### 1. TwoHamsters: Benchmarking Multi-Concept Compositional Unsafety in Text-to-Image Models

**《TwoHamsters：面向文生图模型的多概念组合不安全基准》**

arXiv: https://arxiv.org/abs/2604.15967 | 领域: cs.CR / cs.CV | 作者: Chaoshuo Zhang et al. | 发布: 2026-04-17 | GitHub: 暂未公开

#### 一、背景与问题
现有 T2I 安全机制默认“危险 = 某个显式危险词或显式危险视觉概念”，但现实里很多风险来自**组合关系**：单个概念都无害，拼在一起才形成羞辱、仇恨、性暗示或危险语义。论文要回答的是：现有模型、过滤器和概念擦除方法在这种 MCCU（Multi-Concept Compositional Unsafety）风险上到底有多脆弱？

#### 二、核心方法
作者首先形式化 MCCU，然后构造 **TwoHamsters benchmark**：共 **17.5k prompts**、覆盖 **5.8k 高风险场景**、组织成 **10 类风险 / 51 个概念对**。随后统一评测 **10 个 T2I 模型、7 个安全过滤器、9 个概念擦除方法**，并训练 MCCU-ViT / MCCU-VLM 作为专用自动评估器。

#### 三、技术细节
基准采用七阶段构造流程：taxonomy + LLM 双路扩展、CLIP 文本去重（阈值 **0.95**）、组合图与单概念控制图的跨模态有效性筛选、LLM 先验理由标注、双盲人工审核等。核心指标包括 **MDR**（越高越安全）、**SCR**（单概念保真）和 **NCR**（非目标概念保留）。

#### 四、实验设置与结果
| 对象 | 关键结果 |
|------|----------|
| 前沿 T2I 模型 | **FLUX.1 的 MCCU 成功率高达 99.52%**，SD-v3.5 也达到 **96.55%** |
| 安全过滤器 | **LLaVA-Guard Recall 仅 41.06%**，SD Safety Checker 为 **0.00%** |
| 自动评估器 | MCCU-ViT Recall **99.49**，MCCU-VLM Recall **98.63** |

#### 五、局限性分析
1. 组合语义空间天然爆炸，51 个 concept pairs 仍远未穷尽真实风险。  
2. benchmark-specific evaluator 在 OOD 上相关性明显下降。  
3. 概念擦除实验只测了 4 个高频组合，覆盖面仍有限。  

#### 六、复现指南
按 10 类 taxonomy 组织 concept pairs，用 LLM 扩展隐式组合 prompts，先做 CLIP 去重和跨模态有效性筛选，再统一评测 10 个模型与 7 个过滤器。复现时重点核对：**FLUX MCCU 成功率约 99.52%、LLaVA-Guard Recall 约 41.06%、SD Safety Checker 近乎失效**。

### 2. Beyond Text Prompts: Precise Concept Erasure through Text-Image Collaboration

**《超越文本提示：通过图文协同实现精准概念擦除》**

arXiv: https://arxiv.org/abs/2604.15829 | 领域: cs.CV / cs.CR | 作者: Jun Li et al. | 发布: 2026-04-17 | GitHub: https://github.com/OpenAscent-L/TICoE.git

#### 一、背景与问题
现有概念擦除常在两个方向失衡：只靠文本，容易漏掉 prompt 变体；只靠图像，又会把 camera、phone 这类视觉相近但语义无害的对象一起误伤。论文要解决的问题是：能否在强擦除、鲁棒性和无关概念保真之间实现更稳定的平衡？

#### 二、核心方法
作者提出 **TICoE**，由两个模块组成：**CCCM** 用 prompt bank 的凸组合覆盖概念表达变体，**HVRL** 用多尺度视觉 latent token 学习目标概念的层次视觉表示。除此之外，论文还提出 **MCP** 指标，专门衡量相近无害概念的保留情况。

#### 三、技术细节
文本侧用 Dirichlet 采样的凸组合构造连续概念流形，默认 **τ = 0.7**；视觉侧使用三尺度 **{1.0, 0.75, 0.5}** 的 latent token 融合，再通过残差注入形成多尺度视觉锚点，默认 **λ = 0.5**。训练底座为 Stable Diffusion 1.5，每个概念使用 **200 张**参考图像。

#### 四、实验设置与结果
| 任务 / 指标 | 结果 |
|-------------|------|
| gun 擦除 | **ASR 0.00 / UDA 0.02 / P4D 0.04 / FID 30.86** |
| 相近无害概念保留 | camera 的 **MCP 92.06%**，phone **95.91%** |
| 效率 | 总训练时间 **52 min**，显著优于 Co-Erasing 的 **121 min** |

#### 五、局限性分析
1. prompt bank 的覆盖质量直接决定 CCCM 的效果上限。  
2. 参考图像来自模型自生成，对长尾概念可能不够稳定。  
3. 多概念联合擦除仍存在顺序敏感和轻微 spillover。  

#### 六、复现指南
准备目标概念的 **200 张**参考图像，构造 prompt bank 后按 Dirichlet 分布采样凸组合文本表示，再实现三尺度 HVRL 融合与标准擦除训练。关键核验点是：**gun 上 ASR 0.00 / UDA 0.02 / P4D 0.04，camera MCP 接近 92.06%**。

## T2T 论文（LLM 安全）

### 3. Into the Gray Zone: Domain Contexts Can Blur LLM Safety Boundaries

**《走进灰区：领域语境如何模糊 LLM 的安全边界》**

arXiv: https://arxiv.org/abs/2604.15717 | 领域: cs.CR | 作者: Ki Sen Hung et al. | 发布: 2026-04-17 | GitHub: https://github.com/JerryHung1103/JARGON

#### 一、背景与问题
模型通常被假定存在稳定的“允许 / 拒绝”边界，但专业研究语境和安全讨论语境会让这种边界软化。论文关注的问题不是某条 magic suffix 有多强，而是：**研究语境本身会不会成为系统性解锁信号**？

#### 二、核心方法
作者先观察到两类 unlocking：**Vertical Unlocking**（领域内局部放松）与 **General Unlocking**（安全研究语境导致广泛放松），再提出 **Jargon**：通过安全研究上下文构造多轮学术讨论环境，生成查询变体并结合成功轨迹记忆库进行 few-shot 复用。

#### 三、技术细节
主实验基于 **JailbreakBench 的 100 个 harmful prompts / 10 类风险**，使用 **Abstract + Method** 作为默认研究上下文，每轮生成 **8 个 query variants**，并按 **3 次 retry × 2 次 trial × 4 轮对话** 运行，成功阈值为 judge score **≥ 0.8**。

#### 四、实验设置与结果
| 指标 | 结果 |
|------|------|
| 平均 ASR | **99.0%** |
| GPT-5.2 | **93%** |
| Claude Sonnet / Opus | **100% / 100%** |
| 去掉 variants（GPT-5.2） | 从 **93%** 降到 **54%** |

#### 五、局限性分析
1. 99% ASR 依赖自动 judge，绝对数值可能偏乐观。  
2. 论文主要覆盖安全研究文本，尚未完全扩展到博客、论坛和混杂语境。  
3. 激活分析给出了灰区现象位置，但还不是强因果证明。  

#### 六、复现指南
在 JailbreakBench 上选择 100 个 harmful prompts，用一篇安全研究论文（如 X-Teaming）的 Abstract + Method 作为上下文，按 **3×2×4** 多轮循环和 **8 个变体**生成规则运行 Jargon。关键核验点是：**平均 ASR 接近 99%，GPT-5.2 有 variants 时约 93%**。

### 4. Pruning Unsafe Tickets: A Resource-Efficient Framework for Safer and More Robust LLMs

**《剪掉不安全抽签：一种资源友好的 LLM 安全增强框架》**

arXiv: https://arxiv.org/abs/2604.15780 | 领域: cs.LG / cs.CL | 作者: Wai Man Si et al. | 发布: 2026-04-17 | GitHub: 暂未公开

#### 一、背景与问题
很多安全增强方法只是把模型推向“更倾向拒答”，却没有真正移除触发危险输出的参数子结构。论文提出一个更直接的问题：如果危险行为是由少量稀疏组件承载，能否通过**部署后轻量剪枝**降低 unsafe output，同时尽量保住能力？

#### 二、核心方法
方法分四步：行为画像、响应阶段 Wanda 归因、unsafe / safe 组件对比分数计算，以及基于 **Greedy / Beam Search** 的迭代剪枝。它不依赖昂贵重训练，也不在推理时引入额外 token。

#### 三、技术细节
改造后的 Wanda 只统计 response token 的激活贡献，用 **I = S'_u / (S'_s + ε)** 衡量组件对危险输出的偏向。默认设置为：每轮组件内剪枝比例 **10%**，全局停止阈值 **3%**，Beam Search 参数 **b1 = 5, b2 = 5**。

#### 四、实验设置与结果
| 指标 | 结果 |
|------|------|
| Mistral-7B Unsafe Rate | **22.8% → 1.17%** |
| LLaVA-v1.6-Mistral-7B Unsafe | **17.1% → 1.96%** |
| AutoDAN ASR | **45.7 → 0.67** |
| 推理额外 token | **0** |

#### 五、局限性分析
1. Beam Search 的离线构建成本仍不低。  
2. 归因本质上仍是相关性剪枝，不是完整因果解释。  
3. over-refusal 仍在 25%–46% 区间，说明帮助性损失尚未完全解决。  

#### 六、复现指南
先收集 safe / unsafe 响应并用 Llama-Guard 标注，再实现 response-only masked Wanda，按 unsafe / safe attribution 比值做组件打分，并在 **p = 10%、ρ = 3%** 的条件下复现 Beam Search 剪枝。重点核对：**Mistral-7B Unsafe 22.8% → 1.17%，AutoDAN 45.7 → 0.67**。

## Agentic 论文（AI 智能体安全）

### 5. HarmfulSkillBench: How Do Harmful Skills Weaponize Your Agents?

**《HarmfulSkillBench：有害技能如何把你的 Agent 武器化？》**

arXiv: https://arxiv.org/abs/2604.15415 | 领域: cs.CR / cs.AI | 作者: Yukun Jiang et al. | 发布: 2026-04-16 | GitHub: https://github.com/TrustAIRLab/HarmfulSkillBench

#### 一、背景与问题
开放 skill 生态的风险不只在“skill 会不会被注入或投毒”，还在于**skill 本身是否是危险能力的封装体**。论文要回答的问题是：harmful skills 在真实 registry 中有多普遍？一旦暴露给 agent，会不会系统性削弱模型拒答？

#### 二、核心方法
作者先对两大 skill registry 做大规模测量，再构建 **HarmfulSkillBench**。前者在 **98,440 个 skills** 上做 taxonomy 风险打分；后者从中构造 **200 个 harmful skills / 20 类** 的基准，并在 A / B / C / D 四类条件下测 agent 的拒答、HiTL、AID 和 harmfulness。

#### 三、技术细节
识别阶段使用 **GPT-5.4-Mini**，每个 skill 重复推理 **3 次**，按 21 类 taxonomy 取最大风险分数，阈值为 **0.60**。Benchmark 中 Tier 1 为 Prohibited Use（130 条），Tier 2 为 High-Risk Use（70 条）。

#### 四、实验设置与结果
| 指标 | 结果 |
|------|------|
| harmful skills 数量 | **4,858 / 98,440 = 4.93%** |
| ClawHub harmful rate | **8.84%** |
| 平均 harm score | **A = 0.76 / B = 0.47 / D = 0.27** |
| Tier 2 safeguard | **HiTL 的减害效果约为 AID 的 2 倍** |

#### 五、局限性分析
1. 评测的是 planning / decision tendency，不是危险动作的真实执行。  
2. 只覆盖 6 个主流模型，未穷尽小模型和本地 agent。  
3. harm score 仍依赖 judge 设计，不是现实伤害的直接测量。  

#### 六、复现指南
抓取 skill registry 的元数据与 SKILL.md，用 GPT-5.4-Mini 做 3 次 taxonomy 风险打分并按 **0.60** 阈值筛出 harmful skills，再构造 200 条 benchmark，分别在 A / B / C / D 条件下评测。关键核验点是：**总体 harmful rate 4.93%，A / B / D harm score 约 0.76 / 0.47 / 0.27**。

### 6. LogJack: Indirect Prompt Injection Through Cloud Logs Against LLM Debugging Agents

**《LogJack：针对 LLM 调试代理的云日志间接提示注入》**

arXiv: https://arxiv.org/abs/2604.15368 | 领域: cs.CR | 作者: Harsh Shah | 发布: 2026-04-15 | GitHub: https://github.com/HarshShah1997/logjack

#### 一、背景与问题
日志、错误报告和遥测通常被默认当作可信运维上下文，但调试 agent 读取它们后，往往还会生成修复建议甚至直接执行命令。论文要验证的是：**日志会不会成为比网页、邮件更危险的间接注入载体**？

#### 二、核心方法
作者提出 **LogJack benchmark**，构造 **42 个 payload**、覆盖 **5 类云日志 / 遥测源**，并搭建只读工具 + 动作工具的 debugging-agent harness，在不真实执行危险命令的前提下，评测模型是否愿意把日志里的恶意载荷转化成执行动作。

#### 三、技术细节
Benchmark 包含 **26 个 attack、6 个 RCE、10 个 benign control**，条件分为 **Active / Passive / Encouraged** 三类。每个 payload / model / condition 运行 **5 次**，主指标为 **Hijack Rate、Influence Rate、Control Baseline**，同时记录 AWS / Azure / GCP 等 guardrail 的检测情况。

#### 四、实验设置与结果
| 指标 | 结果 |
|------|------|
| Llama 3.3 70B Active Hijack | **86.2%** |
| Llama 3.3 70B Influence Rate | **99.4%** |
| Active 条件可触发 RCE 的模型数 | **6 / 8** |
| Azure / GCP 输入侧检测 | 对日志 payload **近乎全部失效** |

#### 五、局限性分析
1. 每个 payload / 条件只做 5 次试验，对高温模型尾部风险估计有限。  
2. harness 更像高权限上界风险模拟，现实部署可能有更细权限限制。  
3. 论文主要给出风险证明与 benchmark，尚未提供成熟防御闭环。  

#### 六、复现指南
按五类日志源组织 42 个 payload，搭建 3 个只读工具和 2 个动作工具的拦截式 harness，对 8 个模型分别在 Active / Passive / Encouraged 条件下执行 **5 次独立试验**。关键核验点是：**Llama 3.3 70B Active Hijack 约 86.2%，6/8 模型在 Active 条件下可触发 RCE**。

## 当日趋势信号

### 1. T2I 安全从“单概念治理”进入“组合语义治理”
TwoHamsters 说明真正难防的不是显式危险词，而是**多个安全概念组合后的隐含风险**；TICoE 则说明防御也必须同步升级，不能再靠单 prompt 擦除，而要同时利用文本表达空间和视觉相似空间的信息。未来 T2I benchmark 与 concept erasure 都会越来越围绕“组合性”展开。

### 2. T2T 安全边界正在被语境软化，但也开始出现结构性后处理补丁
Into the Gray Zone 指出“研究语境”会把模型推入 gray zone；Pruning Unsafe Tickets 则给出一个很工程化的补丁思路：既然危险行为集中在少量组件，就先剪掉它们。这意味着 T2T 安全正在分化成两条线：一条研究**边界为什么会软化**，一条研究**如何在部署后做低成本修补**。

### 3. Agent 安全风险已经下沉到技能生态与运维遥测
HarmfulSkillBench 与 LogJack 共同表明，Agent 风险不再只在 prompt 或工具调用文本本身，而在**生态输入源是否可信**：skill 可以把危险能力包装成“合法可调用接口”，日志可以把恶意命令伪装成“可信修复建议”。Agent 安全的重点控制点正在转向 skill registry 治理、遥测隔离与动作执行边界。
