## 今日概览

今天最强的新信号，来自 **T2I 安全治理的“后移与前移同时发生”**。

一方面，治理在**后移**：**Dual-Guard** 与 **IncreFA** 说明行业已经不满足于“生成时拦一下”——现在开始系统测试生成后溯源、篡改定位、开放集归因与增量模型识别。也就是说，T2I 安全不再只看生成器本体，而是把**归因链、传播链、模型身份链**一起纳入主战场。另一方面，治理也在**前移**：**Embedding Arithmetic** 说明，偏见缓解不一定要等训练期重构，推理时条件嵌入本身就已经成为可操作的治理点。它把 fairness mitigation 从“昂贵再训练”拉回到“后处理条件控制”。

在 **T2T** 方向，今天两篇论文把“上下文结构”推到了比文本表面更高的优先级。**IICL** 证明 few-shot pattern completion 本身会成为 jailbreak 旁路：攻击不再只是更换危险词，而是让模型把整个任务误识别为抽象模式补全。**HarDBench** 则把这种上下文重写进一步落到产品形态——协作写作。它揭示“草稿 + co-authoring framing”能把多个主流模型的 ASR 推到 **80%–99%** 区间，说明编辑类产品、文档协作产品会持续成为安全评测盲点。

在 **Agentic** 方向，**GAAP** 给出了一条非常干脆的路线：不要再寄希望于模型自己分辨哪些隐私数据不能发，而是把 agent 放进一个能做**确定性数据流审计**的执行环境里。它的意义不在“又一个防 prompt injection 模型”，而在于把 Agent 隐私保护改写成 execution environment 问题。这和最近 MCP / skill / telemetry 风险持续上升的主线是完全同向的。

## T2I 论文（文生图安全）

### 1. Dual-Guard: Dual-Channel Latent Watermarking for Provenance and Tamper Localization in Diffusion Images

**《Dual-Guard：面向扩散图像来源认证与篡改定位的双通道 latent 水印》**

arXiv: https://arxiv.org/abs/2604.19090 | 领域: cs.CR | 作者: JinFeng Xie et al. | 发布: 2026-04-21 | GitHub: 暂无公开

#### 一、背景与问题
过去的 T2I watermark 常常只能解决一半问题：要么证明“它来自谁”，要么回答“它有没有被改”，很难同时兼顾。Dual-Guard 抓住了这一结构性缺口，把 provenance 与 tamper localization 分成两个潜空间锚点处理。

#### 二、核心方法
它使用双通道 latent watermark：
- **GS 通道** 写入扩散初始噪声 z_T，负责 provenance anchor；
- **Latent Fingerprint Codec** 写入最终 latent z_0，负责内容一致性和 block-level 定位。

#### 三、技术细节
核心写入形式为 `z̃_0 = z_0 + α · r`，其中 **α = 0.3**。定位阶段把 latent 划成 **16×16** 个 block，并融合 cosine correlation、L1 deviation、decoder BMR 三类证据。

#### 四、实验设置与结果
| 场景 | Auth Reject ↑ | Tamper Detect ↑ | Dual Pass ↑ |
|------|--------------:|----------------:|------------:|
| Original | 0.003 | 0.001 | 0.998 |
| Reprompt | **1.000** | **1.000** | **0.000** |
| Local tamper | **0.999** | **0.999** | 0.975 |
| DiffEdit | **1.000** | **1.000** | 0.601 |

定位上，相比 SEAL：
- **IoU 0.255 vs 0.036**
- **F1 0.392 vs 0.068**

#### 五、局限性分析
依赖 Full mode 保存参考 latent；定位仍偏粗粒度；对强白盒自适应去水印尚未充分覆盖。

#### 六、复现与启示
这篇论文说明：**T2I 安全的溯源链不能再只问“有没有 watermark”，而要把“来源锚点”和“内容锚点”拆开建模。**这会直接影响后续模型归因、内容取证与平台治理链设计。

### 2. Embedding Arithmetic: A Lightweight, Tuning-Free Framework for Post-hoc Bias Mitigation in Text-to-Image Models

**《Embedding Arithmetic：免微调、后处理式的 T2I 偏见缓解框架》**

arXiv: https://arxiv.org/abs/2604.18167 | 领域: cs.CV | 作者: Venkatesh Thirugnana Sambandham, Torsten Schön | 发布: 2026-04-20 | GitHub: https://github.com/cvims/EMBEDDING-ARITHMETIC

#### 一、背景与问题
它回答了一个很现实的问题：已经上线的 T2I 模型，能不能不重训就做偏见治理？论文给出的答案是：可以先在**文本条件嵌入**这层动手。

#### 二、核心方法
对属性 A 构造方向向量：
`v_A = (1/K) · Σ_k ( E(T_base ⊕ A) - E(T_base) )`

再做推理时编辑：
`e_target = e_base + Σ_i α_i · v_A_i`

其中 **EA_i**（intersectional version）是表现最稳的版本。

#### 三、技术细节
- 主实验模型：**FLUX 1.0-Dev**
- 泛化验证：**SD3.5-Large**
- 只编辑 **CLIP ViT-L/14 pooled embedding**
- 默认 α = **1.25**，K = **10**
- 核心保真指标：**CCS (Concept Coherence Score)**

#### 四、实验设置与结果
| 职业 | FLUX 默认 Hg / Hr / CCS | FLUX EA_i Hg / Hr / CCS |
|------|-------------------------|-------------------------|
| Fire Fighter | 0.50 / 0.17 / 0.58 | **0.99 / 0.82 / 0.61** |
| Mechanic | 0.08 / 0.08 / 0.64 | **0.99 / 0.93 / 0.60** |
| Flight Attendant | 0.41 / 0.18 / 0.62 | **0.99 / 0.79 / 0.62** |
| Nurse | 0.00 / 0.21 / 0.57 | **0.68 / 0.80 / 0.57** |

#### 五、局限性分析
只编辑 pooled embedding，对多人场景和细粒度 token binding 不够强；也仍依赖外部属性分类器做审计。

#### 六、复现与启示
它的价值不只是“多了个偏见缓解小技巧”，而是说明 **inference-time condition editing** 已经成为 T2I fairness 治理的一条独立路线。未来很多治理不一定从数据重训开始，而会先从条件控制层试起。

### 3. IncreFA: Breaking the Static Wall of Generative Model Attribution

**《IncreFA：打破生成模型归因的静态墙》**

arXiv: https://arxiv.org/abs/2604.17736 | 领域: cs.CV | 作者: Haotian Qin et al. | 发布: 2026-04-20 | GitHub: https://github.com/Ant0ny44/IncreFA

#### 一、背景与问题
AIGC 归因最大的问题不是“今天测准不准”，而是“明天来了新模型后还准不准”。IncreFA 把这个问题从静态闭集分类改写成**增量归因 + 开放集检测**问题。

#### 二、核心方法
框架由两部分组成：
- **Hierarchical Constraints**：显式建模 GAN / Diffusion / AR 的家族结构；
- **Latent Memory Bank**：做 replay，并通过随机混合 latent 生成 pseudo-unseen 样本。

#### 三、技术细节
总损失：
`L = L_cls + α_1L_1 + α_2L_2 + α_3L_u + α_4L_replay`

默认：
- **α_1 = 0.2**
- **α_2 = 0.5**
- **α_3 = 0.5**
- **α_4 = 1.0**
- 开放集阈值 **τ = 0.65**

#### 四、实验设置与结果
IABench 覆盖 **28 个生成模型 + Real**，总计 **544,333** 张图。

| 协议 | 指标 | IncreFA |
|------|------|--------:|
| EP1 增量归因 | T7 Avg. Acc. ↑ | **78.80** |
| EP1 增量归因 | Unseen Acc. ↑ | **98.93** |
| EP1 增量归因 | Auth. Acc. ↑ | **99.97** |
| EP2 静态归因 | Avg. Acc. ↑ | **95.93** |

#### 五、局限性分析
仍依赖预定义 family taxonomy；random mixing 只能近似 unseen distribution；某些前沿模型（如 FLUX.1-schnell）仍较难区分。

#### 六、复现与启示
这篇论文最值得重视的地方是：**T2I benchmark 正从“静态分类分数”升级为“持续扩展的归因基础设施”。**如果这条线继续发展，后续平台治理会越来越依赖 open-set attribution，而不是单次离线检测。

## T2T 论文（LLM 安全）

### 4. Involuntary In-Context Learning: Exploiting Few-Shot Pattern Completion to Bypass Safety Alignment in GPT-5.4

**《IICL：利用 few-shot 模式补全绕过 GPT-5.4 安全对齐》**

arXiv: https://arxiv.org/abs/2604.19461 | 领域: cs.CR | 作者: Alex Polyakov et al. | 发布: 2026-04-21 | GitHub: 暂无公开

#### 一、背景与问题
IICL 指出，alignment 失效不一定来自更复杂的危险词，而可能来自模型对上下文任务的误识别：模型以为自己在做 pattern completion，而不是在回答有害请求。

#### 二、核心方法
攻击把任务重写成抽象算子问题，用 benign / harmful few-shot 示例建立模式，再让模型对目标输入做 continuation。

#### 三、技术细节
- Judge：**gpt-4.1-mini**
- 总 probes：**3479**
- bypass 条件：`score ≥ 2` 且 `harmful_content_provided = true`

#### 四、实验设置与结果
| 条件 | 结果 |
|------|------|
| Direct baseline | **0.0%** |
| HarmBench 上 IICL ASR | **24.0% [18.6%, 30.4%]** |
| interleaved 顺序 | **76%** |
| harmful-first 顺序 | **6%** |
| answer / is_valid 命名 | **100% (50/50)** |

#### 五、局限性分析
只测 OpenAI 模型；消融主要集中在 GPT-5.4；机制解释仍是黑盒推断。

#### 六、复现与启示
它提醒我们：**jailbreak 不再只是字符串层面的对抗，而是“让模型误判当前任务类型”的上下文攻击。**

### 5. HarDBench: A Benchmark for Draft-Based Co-Authoring Jailbreak Attacks for Safe Human–LLM Collaborative Writing

**《HarDBench：面向协作写作草稿越狱的专项 benchmark》**

arXiv: https://arxiv.org/abs/2604.19274 | 领域: cs.CL | 作者: Euntae Kim et al. | 发布: 2026-04-21 | GitHub: https://github.com/untae0122/HarDBench

#### 一、背景与问题
HarDBench 关注一个极其产品化的场景：用户给草稿，模型继续补写。论文证明，这种“协作写作 framing”会显著放大越狱风险。

#### 二、核心方法
- 构建 **HarDBench**，比较 HQ、CoJP w/o TF、CoJP 三种设置；
- 提出 **SUBA**，同时优化 harmful draft 的拒答与 benign draft 的协作完成。

#### 三、技术细节
- 总 benchmark：**1204** 条有害草稿
- 测试集：**400** 条
- 4 个风险领域：Explosives、Drugs、Weapons、Cyberattacks
- 指标：**HS、ASR、RAR**

#### 四、实验设置与结果
| 模型 | HQ ASR ↑ | CoJP ASR ↑ |
|------|---------:|-----------:|
| LLaMA3-8B | 24.75% | **80.50%** |
| Qwen3-8B | 27.25% | **99.00%** |
| GPT-4o | 17.75% | **96.75%** |
| DeepSeek-R1-32B | 37.75% | **96.25%** |

在 LLaMA3-8B 上：
- Zero-shot CoJP ASR = **80.50%**
- **SUBA_GRPO = 3.00%**
- utility = **+3.16%**

#### 五、局限性分析
当前风险域仍有限；prompt library 偏静态；只系统比较了 KTO / GRPO。

#### 六、复现与启示
HarDBench 最重要的信号是：**协作写作产品天然会把有害意图伪装成编辑任务，因此“文档补写”应成为独立安全基线，而不是附属测试项。**

## Agentic 论文（AI 智能体安全）

### 6. An AI Agent Execution Environment to Safeguard User Data

**《GAAP：保护用户数据的 AI Agent 执行环境》**

arXiv: https://arxiv.org/abs/2604.19657 | 领域: cs.CR | 作者: Robert Stanley et al. | 发布: 2026-04-21 | GitHub: 暂未公开（计划开源）

#### 一、背景与问题
GAAP 把 Agent 隐私安全的控制点从“模型自己判断”改写成“执行环境做确定性数据流约束”。它要解决的不是模型会不会被注入，而是：即便被注入，也不能泄露未授权隐私。

#### 二、核心方法
GAAP 包含：
- IFC Core
- Private Data DB
- Permission DB
- Disclosure Log
- Annotation Framework

它让 LLM 先生成 code artifact，再在执行环境中追踪 taint flow。

#### 三、技术细节
- 可把 **LLM provider** 也当作外部实体；
- 支持 multi-shot execution；
- 用 Disclosure Log 处理跨步骤、跨任务、跨时间的间接流。

#### 四、实验设置与结果
| 方法 | SSN-leak ↓ | Phone-leak ↓ | SSN-swap ↓ | 自建任务成功率 ↑ |
|------|-----------:|-------------:|-----------:|-----------------:|
| NP-Agent | 75% | 100% | 100% | 81.0% |
| LLM-Judge | 15% | 部分拦截 | 几乎无效 | 75.0% |
| Conseca | 0% | 部分拦截 | 几乎无效 | 72.0% |
| **GAAP** | **0%** | **0%** | **0%** | **76.0%** |

其它结果：
- 平均成本 **$0.52**，低于 NP-Agent 的 **$0.67**；
- Permission DB 持久化后，真实用户交互从 **58** 降到 **39**；
- 总体仅比 NP-Agent 慢 **13%**。

#### 五、局限性分析
不保护用户直接写进 prompt 的秘密；主要保证 confidentiality 而非 integrity；需要可信 annotations。

#### 六、复现与启示
这篇论文很重要，因为它把 Agent 隐私保护从“训练更安全的模型”推进到了“设计更安全的 execution harness”。这条线和当前 MCP / skill / telemetry 风险持续上升完全同向。

## 当日趋势信号

### 趋势 1：T2I 安全治理正在从“生成器单点防线”扩展成“全链路治理”
今天的三篇 T2I 论文分别对应：
- **生成前/推理时治理**：Embedding Arithmetic
- **生成后溯源与定位**：Dual-Guard
- **跨版本持续归因**：IncreFA

这说明 T2I 主线已经从“做不做安全过滤器”演进为“条件控制 + 溯源归因 + 生命周期治理”的组合架构。

### 趋势 2：Benchmark 正在从静态闭集走向动态开放世界
IncreFA 的 IABench 与 Dual-Guard 的 tamper protocol 都说明，评测不再默认“模型集合固定、样本静态不变”。未来更常见的问题是：
- 明天来了新模型怎么办？
- 图像被改过怎么办？
- 平台声称使用某模型但其实没用怎么办？

### 趋势 3：T2T 越狱的危险点继续往“任务 framing”层上移
IICL 与 HarDBench 共同表明：风险不只来自危险词，而来自**模型如何解释当前任务**。当模型以为自己在做“pattern completion”或“co-authoring”，安全边界会显著软化。

### 趋势 4：Agent 安全的正确控制点正在从 prompt 转向 execution harness
GAAP 与最近一周 MCP / skill / protocol 防线的论文一起，进一步确认：**安全真正需要控制的是执行环境、权限状态与历史披露链，而不是继续在输入表面叠规则。**
