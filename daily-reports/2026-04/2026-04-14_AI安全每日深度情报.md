## 今日摘要

本期最值得注意的信号，不是“又多了几种 prompt 技巧”，而是**安全研究正在从输入级攻防，进一步迁移到调用边界、轨迹仓库和内部表示空间三个更深层的控制面**。Agent 方向的三篇论文形成了非常清晰的链条：ClawGuard 把防御点下沉到工具调用边界，STARS 把技能安全建模为“请求条件化的连续风险评分”，Meerkat 则把审计范围从单条 trace 扩展到跨大量轨迹的 witness 搜索。三者共同说明：Agent 安全的主战场正在从“识别恶意提示”转向“控制动作、审计轨迹、治理系统状态”。

LLM 主干安全方向则出现了两条相反但彼此咬合的路线。**Critical-CoT** 代表防御侧：攻击者已不必直接让模型输出错误，而是通过 reasoning-level backdoor 在 CoT 中插入看似自然的恶意推理，迫使防御转向“识别被污染的推理过程”。**Nullspace Steering** 代表攻击侧：如果安全对齐主要由少量注意力头与局部子空间承载，那么白盒攻击者可以直接对内部回路做几何级操纵，而不再满足于表层 prompt 搜索。二者叠加后，一个值得警惕的结论是：**未来 T2T 安全的攻防焦点，正在从文本表面移动到推理轨迹与内部表示几何。**

T2I 方向虽然今天只有一篇，但价值极高。**On the Robustness of Watermarking for Autoregressive Image Generation** 首次系统指出：自回归图像生成的 watermark 并不存在“天然稳固”的乐观结论。token 级 watermark 容易被 removal，BitMark 虽抗一般去除却对 forgery 脆弱，而一旦要求兼顾 radioactive 数据过滤，阈值调节会陷入结构性三难。这意味着 T2I 安全在扩散模型之后，正在把核心问题从“生成内容审核”推进到“来源证明体系本身是否可信”。

## T2T 论文（LLM 安全）

### 1. Critical-CoT: A Robust Defense Framework against Reasoning-Level Backdoor Attacks in Large Language Models

**《Critical-CoT：面向推理层后门攻击的鲁棒防御框架》**

arXiv: https://arxiv.org/abs/2604.10681 | 领域: cs.CR | 作者: Tuan Vu Truong, Long Bao Le | 发布: 2026-04-12 | GitHub: https://github.com/tuanvu171/Critical-CoT

#### 一、背景与问题
传统后门攻击多作用在 token 或标签层，但 reasoning model 的更大风险在于**CoT 被污染而非最终答案直接异常**。一旦恶意推理步骤本身看起来合理，依赖矛盾检测或异常 token 检测的防御就会失效。

#### 二、核心方法
Critical-CoT 通过 **CTCoT 防御数据集 + SFT + DPO** 三段式训练，让模型在看到异常 trigger 或 poisoned demonstration 时，主动指出其可疑性、显式忽略它们，并重新给出干净推理。该方法同时覆盖 ICL-based backdoor 与 FT-based backdoor。

#### 三、技术细节
SFT 学习“识别 trigger / 被污染推理后的正确响应”，DPO 再学习“后门样本偏好防御回答、干净样本偏好正常回答”的决策边界：

L_SFT(θ) = E[L_def] + λ·E[L_clean]

L_DPO(θ) = -E[log σ(β(Δ⁺_θ - Δ⁻_θ))]

关键点不在过滤输入，而在把“怀疑异常推理并拒绝跟随”的能力写回模型本体。

#### 四、实验设置与结果

| 场景 | 无防御 | **Critical-CoT** |
|------|--------|------------------|
| ICL 攻击下 ACCd (↑) | 7.8%–9.7% | **38.1%–95.9%** |
| ICL 攻击下 ASRr (↓) | 81.85%–86.7% | **0.42%–0.78%** |
| FT 攻击下 ACCd (↑) | 1.1%–5.1% | **32.4%–91.7%** |
| FT 攻击下 ASRt (↓) | 15.12%–85.23% | **0%–0.22%** |

#### 五、局限性分析
1. 需要额外训练成本与防御数据构造成本；2. 对 FT 型后门只能阻断捷径，不能完全修复已受损的推理能力；3. 当前 trigger 仍以显式字符 / 短语为主；4. 辅助数据生成依赖外部 LLM。

#### 六、复现指南
建议从 GSM8K / MATH / CSQA 采样干净样本，构造 ICL 与 FT 两类后门版本，先做 SFT 再做 DPO。论文表明只做 SFT 会出现明显过度警惕，DPO 是控制误报率的关键步骤。

### 2. Jailbreaking the Matrix: Nullspace Steering for Controlled Model Subversion

**《越狱矩阵：基于零空间操纵的可控模型颠覆》**

arXiv: https://arxiv.org/abs/2604.10326 | 领域: cs.CR | 作者: Vishal Pramanik, Maisha Maliha, Susmit Jha, Sumit Kumar Jha | 发布: 2026-04-11 | GitHub: https://github.com/VishalPramanik/Jailbreaking-the-Matrix.git

#### 一、背景与问题
主流越狱方法仍大多停留在 prompt 层搜索，查询代价高且难解释。作者提出更强问题：如果安全对齐集中在少量 attention heads 与局部子空间上，白盒攻击者是否能直接操纵内部几何结构？

#### 二、核心方法
论文提出 **HMNS (Head-Masked Nullspace Steering)**：
1. 用 KL-based counterfactual attribution 找出当前 continuation 最关键的 attention heads；
2. 在 Wᴼ 上静音这些 heads 的写路径；
3. 沿其写入子空间的正交补方向注入 steering 向量；
4. 若失败则重新 attribution，形成闭环干预。

#### 三、技术细节
关键公式包括：
- Head importance: Δ_(ℓ,h) = KL(P ∥ P̃^(ℓ,h))
- 子空间矩阵: M_ℓ = [Wᴼ_ℓ[:, h·d_h:(h+1)·d_h]]
- 零空间方向: u_ℓ = ((I - Q_ℓQ_ℓᵀ)r) / (‖(I - Q_ℓQ_ℓᵀ)r‖₂ + ε)
- 注入: δ_ℓ = α·RMS(a_ℓ)·u_ℓ

#### 四、实验设置与结果

| 模型 / 数据集 | 强基线 | **HMNS** | 查询数 |
|---------------|--------|----------|--------|
| LLaMA-2-7B, AdvBench | 次优低 90s | **98 / 93** | **2.0** |
| Phi-3-14B, JBB | 次优低 90s | **94 / 88** | **1.9** |
| LLaMA-3.1-70B, HarmBench | 中高 80s | **97 / 94** | **2.0** |

> ASR 采用 GPT-4o / GPT-5 双 grader 报告。

#### 五、局限性分析
1. 强白盒前提极高；2. 理论保证只在局部与被屏蔽 heads 范围内成立；3. attribution 与 QR 分解有明显系统开销；4. 方法具有很强双用途风险。

#### 六、复现指南
需要开放权重模型、attention hook、Wᴼ 切片访问与关闭 KV cache 的推理环境。复现核心在于：按 prompt 动态做 head attribution，而不是固定一次性 mask；没有闭环重识别，效果会显著下降。

## T2I 论文（文生图安全）

### 3. On the Robustness of Watermarking for Autoregressive Image Generation

**《自回归图像生成水印的鲁棒性再审视》**

arXiv: https://arxiv.org/abs/2604.11720 | 领域: cs.CV | 作者: Andreas Müller et al. | 发布: 2026-04-13 | GitHub: 暂无公开代码库

#### 一、背景与问题
随着 VAR、HMAR、Infinity 等自回归图像生成模型发展，watermark 成为来源识别与训练数据过滤的核心工具。但 AR watermark 的鲁棒性评估长期集中在 JPEG、噪声、再生成等常规扰动，对 token / bit 级结构攻击研究不足。

#### 二、核心方法
论文系统评估 4 类 AR watermark（IndexMark、WMAR、ClusterMark、BitMark），并提出三类结构性攻击：
- **VQ-Regen**：基于码本邻域替换的 removal
- **LatentOpt**：基于编码器潜变量优化的 removal / forgery
- **Frequency Injection**：针对 BitMark 的无模型访问频域伪造

#### 三、技术细节
核心形式包括：
- KGW 风格 watermark: p'θ(yᵢ|y_<ᵢ) = softmax(fθ(y_<ᵢ) + mᵢ·δ)
- VQ-Regen: x' = D(𝓒[t'])
- LatentOpt-Removal: argmax ‖𝓔(x+Δx) - 𝓔(x)‖²
- LatentOpt-Forgery: argmin ‖𝓔(x_c+Δx) - 𝓔(x_w)‖²

#### 四、实验设置与结果

| 场景 | 常规认知 | **本文结论** |
|------|----------|--------------|
| Token watermark 去除 | 通常被认为较稳 | **黑盒下已显著脆弱** |
| Token watermark 白盒攻击 | 预期更强 | **TPR 可降至 0.00** |
| BitMark 去除 | 对普通攻击很强 | **BitOpt-R 仍可完全去除** |
| BitMark 伪造 | 常被忽略 | **Frequency Injection 在无模型访问下 TPR 0.73–0.93** |
| Radioactive 检测 | 靠调阈值即可兼顾 | **与 anti-forgery / anti-removal 构成三难困境** |

#### 五、局限性分析
1. 未覆盖所有 AR watermark 设计；2. 某些实现并非最强发布配置；3. Frequency Injection 仍非全局最优搜索；4. 论文揭示问题多于提供最终解法。

#### 六、复现指南
需要同时准备 watermark verifier、AR 编码器 / 量化器和频域注入工具。对 token-based watermark 建议先跑 VQ-Regen / LatentOpt，再对 BitMark 单独评测 bit-level removal 与频域 forgery，最后观察 radioactive filtering 阈值的 trade-off。

## Agentic Search 论文（AI Agent 安全）

### 4. Detecting Safety Violations Across Many Agent Traces

**《跨大量 Agent 轨迹检测安全违规》**

arXiv: https://arxiv.org/abs/2604.11806 | 领域: cs.AI / cs.CL | 作者: Adam Stein et al. | 发布: 2026-04-13 | GitHub: 暂无公开代码库

#### 一、背景与问题
很多真实世界安全失败不会在单条轨迹上直接暴露，而是需要多条 trace 联合起来才能看出模式，例如 developer cheating、reward hacking 或分布式 misuse。逐条打分的 monitor 很难在海量 benign 轨迹中挖出这些稀疏失败。

#### 二、核心方法
Meerkat 把问题重新表述为 **witness set search**：
1. 先对 traces 做 embedding 与聚类；
2. 再让审计 agent 围绕自然语言定义的安全属性 φ 做簇级搜索与取证；
3. 输出 repository-level verdict、trace-level score 与候选 witness 集。

#### 三、技术细节
核心不在单条轨迹分类，而在 repository-level hyperproperty：
- R = {tᵢ}
- 目标是寻找 W ⊂ R，使其联合违反 φ

论文同时给出基于 cluster-aware score 的聚类增强策略，用于提升稀疏违规发现效率。

#### 四、实验设置与结果

| 场景 | Monitor | Naive Agent / Bayesian | **Meerkat** |
|------|---------|------------------------|-------------|
| DM-Cyber, bg=20x, Qwen（trace AP） | 0.125 | 0.632 / 0.243 | **0.837** |
| DM-Cyber, bg=100x, GPT-5.4-mini | — | 0.136 / 0.053 | **0.297** |
| CUA-SHADE-Arena（AP） | 0.84 | 0.84 | **0.86** |

#### 五、局限性分析
1. 强依赖 embedding / clustering 结构；2. 在 TRACE 等任务上并非全面碾压；3. 真实 benchmark 审计缺少完整 ground truth；4. repository-level 审计也会带来隐私与监控问题。

#### 六、复现指南
复现最重要的是先构建 trace repository 的语义聚类索引，再用自然语言属性驱动的审计 agent 按簇浏览、取证与输出 witness，而不是简单让 LLM 顺序扫描所有轨迹。

### 5. ClawGuard: A Runtime Security Framework for Tool-Augmented LLM Agents Against Indirect Prompt Injection

**《ClawGuard：面向工具增强型 LLM Agent 的运行时安全框架》**

arXiv: https://arxiv.org/abs/2604.11790 | 领域: cs.CR / cs.AI | 作者: Wei Zhao et al. | 发布: 2026-04-13 | GitHub: https://github.com/Claw-Guard/ClawGuard

#### 一、背景与问题
网页、MCP server、skill 文件和本地文档都可能把恶意指令伪装成“工具返回内容”，并被 Agent 直接写进上下文。现有对齐方法很难在开放式 Agent 生态里稳定解决这一问题。

#### 二、核心方法
ClawGuard 通过**工具调用边界防护**解决间接注入：
1. 会话开始前，根据 H₀ 自动归纳任务最小权限规则 R_task；
2. 每次调用前后执行内容脱敏；
3. 对 command / file / network 三域做黑白名单判定；
4. 对新 skill 和 ambiguous action 要求人在回路确认。

#### 三、技术细节
活动规则集： 𝓡 = 𝓡_base ∪ 𝓡_task  
单元素判定：黑名单 → ⊥，白名单 → ⊤，否则 amb  
聚合判定：只要任一属性为 ⊥，整体即拒绝；否则若有 amb，则转人工审批。

#### 四、实验设置与结果

| Benchmark | 无防御 ASR | **ClawGuard ASR** | **DSR** |
|-----------|------------|-------------------|---------|
| AgentDojo | 0.6%–3.1% | **≈0.0%** | **99.4%–100%** |
| SkillInject | 26.2%–47.6% | **4.8%–14.2%** | **71.4%–84.6%** |
| MCPSafeBench | 36.5%–44.5% | **7.1%–11.0%** | **74.9%–75.8%** |

#### 五、局限性分析
1. 当前实验只验证基础规则 R_base；2. 对“内容误导但不触发危险调用”的攻击仍有盲区；3. 任务规则归纳质量依赖 backbone LLM；4. 真正的人机审批成本尚未被系统评估。

#### 六、复现指南
需要接管 Agent 的 tool dispatcher，并在真正执行前插入 authorize 阶段。复现重点不在 prompt engineering，而在构建 command / file / network 三域规则与敏感信息净化器。

### 6. STARS: Skill-Triggered Audit for Request-Conditioned Invocation Safety in Agent Systems

**《STARS：面向请求条件化技能调用安全的审计框架》**

arXiv: https://arxiv.org/abs/2604.10286 | 领域: cs.AI | 作者: Guijia Zhang et al. | 发布: 2026-04-11 | GitHub: https://github.com/123zgj123/STARS

#### 一、背景与问题
静态 skill 审计只能回答“这个 skill 能做什么”，却回答不了“在当前请求、当前参数、当前轨迹和当前来源条件下，这次调用是否危险”。这使得 capability risk 与 activation risk 被长期混淆。

#### 二、核心方法
STARS 用三阶段连续风险建模解决这个问题：
1. **Static Prior** 评估 skill 自身的能力与来源风险；
2. **Contextual Invocation Audit** 评估 intent、arg、provenance、trajectory、taint 五类动态信号；
3. **Calibrated Fusion** 对两路分数做分位数归一化并融合，用于 triage。

#### 三、技术细节
核心形式为：

R(U, S, C) ∈ [0,1]

R_trigger = clip(text_base + λ·g(U,S,C)·context_gain + b_cross, 0, 1)

R_fused = w_static·R̃_static + w_trigger·R̃_trigger

即先做 capability risk，再做 activation risk，并用连续风险排序代替单一阈值式判定。

#### 四、实验设置与结果

| 设置 | Static Prior | Contextual | **Fusion** |
|------|--------------|------------|------------|
| Locked Test: HR-AUPRC | 0.188 | 0.195 | **0.204** |
| OOD-I PI: HR-AUPRC | 0.339 | 0.405 | **0.439** |
| OOD-I PI: Recall@10% | 0.023 | 0.146 | **0.152** |

#### 五、局限性分析
1. 在熟悉分布上增益有限；2. 连续风险标签来自 anchor + heuristic，而非人工直接打分；3. Stage B 仍偏规则化；4. OOD 主要验证 IPI，而非全部未知攻击族。

#### 六、复现指南
复现关键不在训练大模型，而在 invocation-level 数据构造：每条样本都必须包含 user request、skill、runtime context 和 canonical action，并能提取 provenance / trajectory / taint 等结构化特征。

## 本周趋势分析

### 趋势 1：Agent 安全从“提示词防御”走向“动作治理 + 仓库审计”
ClawGuard、STARS、Meerkat 三篇论文连在一起，展示了 Agent 安全的新分层：
- **动作层**：ClawGuard 把防御前移到 tool-call boundary；
- **调用层**：STARS 把具体 invocation 建模为连续风险；
- **仓库层**：Meerkat 把大量 traces 看作需要搜索 witness 的 hyperproperty 对象。
这说明只做 prompt injection classifier 已经不够，未来 Agent defense stack 必须覆盖 request → action → trace repository 三层治理。

### 趋势 2：T2T 攻防正在进入“推理链 + 子空间几何”阶段
Critical-CoT 说明攻击者已经能在 CoT 层植入后门；Nullspace Steering 则证明白盒攻击者可以直接对 attention 子空间做局部几何操控。前者要求防御学会“识别被污染的推理”；后者要求防御重新思考“安全回路是否过于低秩、是否容易被机制级定向绕过”。这是一个明确的范式升级信号。

### 趋势 3：T2I 安全的矛盾正在从内容审核转向“来源证明是否可信”
AR watermark 论文给出的最重要结论不是某项指标涨跌，而是：**anti-removal、anti-forgery、radioactive filtering 三者难以兼得**。这意味着未来 T2I 安全体系不能只看“有没有 watermark”，还必须审视 watermark 自身是否会被攻击者拿来反向污染数据治理链条。
